## Цель
Самообслуживаемая настройка распознавания по типам документов прямо в админке (без правок кода).
Владелец/админ заводит шаблон типа документа в разрезе страховой/домена → система автоматически:
LLM-разбор эталона → построение детерминированного парсера → лог расхождений по этому типу.

## Поток (как видит пользователь)
1. `/insurers` → проваливаемся в **страховую** → её **домен(ы)**.
2. Кнопка «**Добавить тип документа**» → загрузка файла-образца (PDF/Excel/RTF/…) как **шаблона типа**.
3. Система видит новый шаблон → прогон **LLM (gpt-5.5)**: «верни СТРОГИЙ JSON всех полей без перефраза».
4. По эталону строится/настраивается **парсер** под этот тип; самопроверка (parser ↔ LLM).
5. Дальше входящие письма этого типа парсятся детерминированно; пробелы (поле было, парсер не нашёл) → **лог**.
6. В карточке типа документа видно: **сколько записей в логе на разбор** по этому типу (сигнал «форма сменилась»).

## Модель (черновик)
- Сущность **DocTemplate**: insurer_id, domain, docType, sample_storage_path, gold_json (эталон LLM),
  parser_rules (jsonb: колонки/маски/метки), status (new|llm_parsed|parser_ready|drift), created/updated.
- Связь с `parse_log` по (insurer, docType) → счётчик «на разбор».
- Хранилище образцов — защищённое (ПДн), как оригиналы (`/api/original`-паттерн).

## Зависит от
- Завершённой методики «парсинг по типам» (этот же воркстрим, см. `recognition-architecture.md`, ADR D15):
  карта 27 типов (страховая × тип), эталон gpt-5.5, шаблоны, parse_log. Сначала шаблоны в пайплайне,
  потом — вынос управления ими в админку.
- Прод-пайплайн (S1): тулзы извлечения в Docker-образе.

## Вне scope (пока)
Автогенерация регэкспов из эталона без участия агента (v2); сейчас правило парсинга строится в пайплайне.
