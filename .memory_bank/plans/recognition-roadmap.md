## Цель
Большой план («карта») распознавания гарантийных писем ДМС от пути «сырое письмо → проверяемая запись в
реестре + Excel». Из него — мелкие вертикальные срезы, каждый под отдельный «деплой», последовательно.
Решение принято по **эмпирике реального корпуса** (51 письмо / 13 страховых, `domain/insurer-recognition.md`,
ADR D10), чтобы **не ошибиться** на структуре.

## Источник задачи
«Разбери, как их все парсить, реши чтоб не ошибиться. OpenAI можно по API через прокси (как в sup2,
модель та же). Сначала план.»

## Главный архитектурный принцип (ADR D10)
**Детерминированное ядро + LLM только на извлечение полей.**
- Структуру (раскрутка пересылки/base64, идентификация страховой по реестру, извлечение текста по форматам,
  дедуп, валидация) — **кодом, покрыто unit на 51 реальном письме**. Угадывать структуру нельзя.
- Вариативные поля ГП — **LLM (OpenAI прокси, `gpt-5.4-mini`, Structured Outputs `strict`)** по уже
  очищенному тексту. Логи LLM — **с маскированием ПДн** (отклонение от sup2, ADR D4/D10).

## ПРОД vs ТЕСТ (определяет конвейер)
- **Прод:** 4 ящика клиники, письма приходят **по одному напрямую** от страховых. Конвейер = забрать
  инкрементально по IMAP → определить страховую → распарсить по её формату → сохранить. **Раскрутки
  пересылки/base64 НЕТ.**
- **Тест (сейчас):** корпус пришёл одним контейнером пересланных base64-`.eml`. Раскрутка — **тест-адаптер
  ingestion** (загрузить фикстуры), не ядро прода. Детали и матрица — `domain/insurer-recognition.md`.

## Конвейер прода (целевой, слева направо)
```
IMAP(по одному, инкрем. по UID) ─▶ EmailMessage+Attachment (оригинал на сервере, всегда ссылка)
  ─▶ [ID] домен отправителя → реестр InsuranceCompany.domains[]; фолбэк тело/Received (только по реестру)
  ─▶ [Extract] по формату страховой: HTML→текст · PDF(pdftotext; скан→OCR) · .xls/.xlsx(SheetJS, построчно
       для реестров) · RTF · ZIP(шифр)+пароль · IMG→OCR
  ─▶ [LLM fields] strict JSON-схема: пациент/полис/страховая/статус/услуги/срок/лимит/№ГП + confidence
  ─▶ [Validate] полнота+уверенность → реестр (1 письмо → 1..N записей) | очередь ручной проверки
  ─▶ [Dedup] Message-ID/hash/№ГП/(пациент+страх+дата+услуги) — помечать, не удалять
```
Тяжёлые шаги (IMAP, OCR, LLM, распаковка) — в **Inngest-функциях** (ретраи, идемпотентность), не в request-пути.

## Сквозные инварианты (для всех срезов)
- **Ссылка на исходник — ВСЕГДА.** Оригинал письма (`.eml` целиком) и **каждое** вложение хранятся **на
  сервере** (защищённое хранилище вне git, путь+hash в БД). Любая запись реестра / карточка / строка Excel
  обязана ссылаться на оригинал и исходное вложение, чтобы распознанное можно было сверить с источником
  (бриф §7.4/§13/§17). Удалять оригинал нельзя — он источник истины при споре.
- **Логирование — полное разрешено** (сервер клиники РФ, ADR D10): можно логировать тело письма, текст,
  запрос/ответ LLM. В логи НЕ пишем только секреты (токены, пароли архивов).
- **Внешние вызовы** (IMAP, OpenAI-прокси, файлы) — в обёртке: Result-тип + Inngest-ретрай + user-facing ошибка.

## Скоуп — что входит (роадмап срезов)
**Phase 0 — Фундамент (инфра для всех срезов).** Drizzle-схема (EmailMessage, Attachment, GuaranteeLetter,
InsuranceCompany, ProcessingQueue, AuditLog) + миграции + **сид реестра 13 страховых с доменами**; порт
LLM-клиента из sup2 (`lib/server/llm/openai.ts`) — лог можно как есть (полное логирование разрешено, ADR D10,
секреты не пишем); подключение Inngest; обёртки внешних вызовов (Result-тип). **Модель: `EmailMessage 1—N
GuaranteeLetter`** (Excel-реестры РЕСО/Альфа = одно письмо → много записей). Проверка: миграции, сид виден,
`isLlmConfigured()` ок, unit на клиент (мок).

**S1 — Забор писем (прод-модель: IMAP по одному → БД).** Inngest cron-функция: XOAUTH2 IMAP, **инкрементально
по UID на каждый ящик** (4 в проде, 1 тестовый сейчас) → сохранить EmailMessage (заголовки, тело text/html) +
Attachment (файлы **на сервере** вне git, в БД путь/hash; **оригинал всегда со ссылкой**). UI: список писем +
статус синка по ящику. e2e: синк ящика → новые письма в БД/списке, повторный синк не дублирует (UID-курсор).

**S1-тест — Адаптер загрузки корпуса (раскрутка пересылки/base64).** ТОЛЬКО для теста: из контейнера
пересланных `.eml` достаёт **51 оригинал** (снимая `message/rfc822` и **base64-в-text/plain без CTE**,
обрабатывая срезанный `From`), кладёт как фикстуры/в БД как «пришедшие по одному». unit на 13 фикстурах = 51.
**В проде не используется** (письма приходят прямые); логика остаётся резервной (ADR D5).

**S2 — Идентификация страховой (реестр).** Домен отправителя → `InsuranceCompany` (сид S0); фолбэк по
доменам тела/Received **только против реестра** (не «любой внешний» — в теле домены клиник); промах → очередь/LLM-фолбэк.
unit: 13/13 на корпусе (вкл. calltravel.eu=Балта, luchi.ru, astrovolga.ru, energogarant.ru при срезанном From).

**S3 — Извлечение текста по форматам** (роутинг по страховой из матрицы `domain/insurer-recognition.md`):
- **HTML→текст** (Лучшее, Астро-Волга, Энергогарант — данные в теле) + общий нормализатор HTML.
- **PDF** через `pdftotext` (ВСК, Совкомбанк, Ренессанс, Балта); **скан→OCR** tesseract-rus (Балта ×1).
- **Excel через SheetJS (xlsx npm)** — читает и старый **.xls** (Ингосстрах/СОГАЗ) и **.xlsx** (РЕСО/Альфа).
  **Реестры построчно:** РЕСО (17–32 стр) и Альфа (2–11 стр) → **1 письмо → N записей** (см. модель S0).
- **RTF→текст** (РЕСО). Выход: чистый текст письма+вложений на запись. unit на реальных вложениях корпуса.

**S4 — Извлечение полей LLM (Structured Outputs).** Очищенный текст → strict JSON-схема полей ГП
(пациент ФИО/ДР/полис, страховая, № обращения/№ГП, статус согласования, услуги, срок, лимит, даты) + **confidence
по полю**. Регэксп-кросс-чек очевидного (полис/даты/№ГП из имени файла). Для Excel-реестров — извлечение на
строку. Запись GuaranteeLetter (1..N на письмо). unit: на N письмах поля совпадают с эталоном.

**S5 — Реестр + карточка + ручная правка (UI ядро админки).** Таблица реестра (фильтры: период/страховая/статус/
пациент/полис), карточка письма (поля+confidence, **ссылка на оригинал+вложения**, история), ручная правка, статусы. e2e.

**S6 — Очередь ручной проверки.** Низкая уверенность/нет обязательных полей/не понята страховая/**архив без пароля**/
дубль/конфликт → очередь; действия: править, связать, подтвердить. e2e.

**S7 — Архив + пароль (Зетта, Росгосстрах) — stateful-корреляция.** Пароль **обычно отдельным письмом** (норма):
письмо-архив → очередь «ждёт пароль» (ключ № ГП / отправитель+тема+время); письмо-пароль ищет архив; пара собрана →
распаковка (zip→PDF Зетта / zip→.xls Росгосстрах) → в конвейер S3. Росгосстрах — пароль иногда в теле (`rgs…`).
Ручной фолбэк связывания обязателен. e2e на парах Зетты/Росгосстраха корпуса.

**S8 — Excel-экспорт (exceljs).** Стабильные колонки (бриф §10), фильтры, ID записи, **ссылка на оригинал**, дата формирования. e2e.

**S9 — Дедуп + аудит-лог + роли (RBAC).** Сигналы дубля (ADR D6, помечать); AuditLog всех правок; роли
владелец/ДМС/врач/регистратура (ADR D4). e2e на права.

## Скоуп — что НЕ входит (сейчас)
МИС-интеграция, авто-ответы, ЭЦП, BI, мобилка, авто-обучение (бриф scope-out). Рабочие 4 ящика клиники — после теста.

## Порядок исполнения
Phase 0 → S1 (+ S1-тест адаптер для фикстур) → S2 → S3 → S4 → S5 → S6 → S7 → S8 → S9. Каждый срез — отдельный план `plans/<slug>.md`,
делается по «деплой», с зелёным гейтом и (где применимо) e2e. Корпус (`.mail-intake/`, gitignored) — источник фикстур.

## Файлы к изменению
Определяются per-срез в его плане. Общая канва: `lib/db/schema/*`, `lib/mail/*` (unwrap/parse), `lib/insurer/*`
(registry/ident), `lib/extract/*` (форматы), `lib/server/llm/*`, `lib/inngest/*`, `app/(admin)/*`, `app/api/*`.

## Критерии приёмки (роадмапа)
- [ ] Каждый срез: зелёный гейт (typecheck+lint+unit+integration+e2e+build) + e2e на затронутый поток.
- [ ] Ядро (раскрутка S1-тест, идентификация S2, извлечение S3) покрыто unit на реальном корпусе (51 письмо, 13 страховых).
- [ ] **Ссылка на исходник всегда есть:** оригинал .eml + каждое вложение на сервере, запись/карточка/Excel ссылаются.
- [ ] ПДн: оригиналы/вложения на сервере вне git; полное логирование разрешено (сервер клиники), секреты в логи не пишем; доступ по ролям.
- [ ] Память обновляется при отклонении от плана (ADR).
