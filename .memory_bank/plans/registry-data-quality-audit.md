## 0. Доктрина корректности (как исправляем — обязательна для КАЖДОЙ колонки)
Порядок извлечения и трактовка пустоты (правило владельца):
1. **ФАЙЛ-вложение первично** (xlsx/xls/pdf/rtf/doc/архив) — парсер обязан взять поле из файла.
2. **Тело письма вторично** — только если в файле поля нет.
3. **Только если нет ни там, ни там** → пометить **явно**: человекочитаемое «нет данных» (не пустая
   строка!), чтобы отличать «проверено — данных нет» от «техническая ошибка/не искали».

### Механизм различения «нет данных» vs «не распознано» (ADR-кандидат D18)
Сейчас поле nullable, экспорт пишет `?? ""` — пустая строка одинакова для обоих случаев. Вводим
per-field статус извлечения (аддитивно, expand-only):
- Новое поле `field_status jsonb` (по аналогии с `confidence`): `{ "policyNumber": "found" | "absent" | "unreadable" }`.
  - `found` — значение извлечено (пишем значение);
  - `absent` — парсер искал в файле И в теле, поля действительно нет → экспорт рисует **«нет данных»**;
  - `unreadable`/нет ключа — не смогли прочитать/не проверяли → экспорт рисует **«⚠ не распознано»** (или пусто),
    строка идёт в `needsReview`.
- Экспорт (`app/api/registry/export/route.ts`) и карточка переводят статус в текст. Значение-маркер
  в само поле НЕ кладём (не засоряем данные).

## 3. Очередь исправления + деплоя (по срезам; каждый = зелёный гейт + бэкап перед прод-backfill)
Принцип порядка: **impact × детерминированность**. Каждый срез: (a) правка парсера
`.mail-intake/extract_dataset.py`/`enrich.py`/`lib/*`; (b) перепарс затронутых вложений ИЗ ФАЙЛА;
(c) прод-backfill прямым SQL со **снапшотом для отката** (ingest = insert-only, не обновляет — ADR D17);
(d) verify counts; (e) синхрон парсера на `/opt/sib-intake`; (f) память/ADR.

- **S0 — Механизм «нет данных»/статусов (ADR D18) [фундамент, без мутации прод-данных].**
  Схема: `guarantee_letter.field_status jsonb` (миграция expand-only). Парсер (ED/EN) проставляет
  `found|absent|unreadable` по полю; `absent` — искали в файле И теле, нет. Экспорт (`export/route.ts`)
  и карточка: `absent`→«нет данных», `unreadable`→«⚠ не распознано». Юнит на маппинг статус→текст. Деплой миграции.
- **S1 — Файл-first в свободном пути (ED 479-511, 549/579/583) [доктрина, критично для владельца].**
  Разделить `fileText` (тексты вложений) и `bodyText` (тема+тело); регэкспы прогонять СНАЧАЛА по `fileText`,
  тело — фолбэк. Починить провенанс `source` (ED 684/712): тег по факту извлечения, не по наличию вложения.
  Юнит: поле, что есть и в файле, и в теле, берётся из файла. Задевает ВСК/Совкомбанк/Балту/Лучи.
- **S2 — Ингосстрах policy=«Полис» (364) + пустая ДР [самый явный баг, файлы есть].**
  Причина: `col()` подстрочный матч (ED 627-636) поймал заголовок как значение в конкретном xlsx-шаблоне.
  Починить маппинг колонки полиса/ДР, перепарс ИЗ ФАЙЛА (`attachment.storage_path`), backfill со снапшотом.
- **S3 — policy: формат-верификация по СК + оживить `policy_series` (schema G34).**
  Сверить с файлами: Ингосстрах «Пр…»(3397) — реальный формат или мислейбл; РЕСО len 60/330/414 — обрезать/перепарс;
  Росгосстрах/Ренессанс/Зетта кириллица — штат или утечка. Разделять серию+номер (POLIS_MASKS ED 271-283).
- **S4 — letter_date «нет данных» (Энергогарант 354, Альфа 249) + Альфа-будущая 2026-08-05 + РЕСО 36.**
  Файл→тело; где нет — `absent`→«нет данных». Будущую/битую — перепарс `letter_date()` (ED 336-361).
- **S5 — case/contract/letter_number «пусто».** По типам: списки прикрепл./откр. → `absent`→«нет данных»;
  ГП-письма — донастроить `contract_from`(ED 397-405)/`GP`(ED 251)/`case` (сейчас всегда None ED 500).
- **S6 — services + conditions/amount на табличном пути (LLM-gap EN 176-193).**
  services пусто: списки→`absent`; ГП→извлечь. `conditions` (только LLM, EN 42-44) и `amount_limit` не заполняются
  на registry-path — добрать (детерм-правила или точечный LLM-проход по табличным записям).
- **S7 — care_type: ревизия `classifyCareType` (care-type.ts 21-38) + пересчёт.** Понять переовер `combined`
  (Ренессанс 67%/Росгосстрах 62%/СОГАЗ 61%/Ингосстрах 54%): combined только при реальном совмещении, иначе по услуге.
- **S8 — Интеграл дат:** СОГАЗ from>to своп (2026-03-15→2026-01-28 ×2), valid_until<letter_date ×6; точечный перепарс.
- **S9 — Мелкие источники + наблюдаемость:** Совкомбанк(pdf)/Лучи/ВСК/Астра-Волга/Балт — доводка; Зетта — AES-архивы
  (ED 211-212 не поддержаны) — проверить распаковку; unknown-domain тихий дроп (FL 67-68) → error_report/лог.
- **S10 — Финальный аудит:** повторить `anomaly.sql`, свести остаток по колонкам, обновить `project-state` + ADR D18.

**Каждый прод-backfill:** `create table _bak_<slice>_<date> as select id, <поля> from guarantee_letter where …`
(снапшот отката) → `update` → verify counts → синхрон парсера на `/opt/sib-intake` → память. ingest insert-only
(ADR D17) — БД правим прямым SQL, файлы-оригиналы read-only.
