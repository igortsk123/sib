# План: LLM → разумный парсер (по кластерам исходников)

> Вопрос владельца (2026-07-23): «посмотри, где LLM отрабатывает, какие исходники — целесообразно это,
> или парсер можно написать разумнее». Ответ: **~76% LLM-потока — шаблонные формы, механизируемо**.
> Доктрина не меняется (D10/D15: гибрид) — LLM остаётся подстраховкой на пробелах и на сканах/новых формах.

## Текущее распределение (прод, 2026-07-23)
`deterministic` 8599 (92%) · `llm` 565 · `deterministic+llm` 147 · `llm_vision` 12 → **724 LLM-записей (7.8%)**.

## Ключевой факт из parse_log (detGap — «LLM нашла, парсер нет»)
`conditions` (95+95+61+61+56+55…) и `careType` (90+88+58+54+53…) — топ-gap у ВСЕХ страховых **не потому,
что формы дрейфуют, а потому что детерминизма для них НЕТ by design** (conditions — только LLM; careType —
LLM/TS-фолбэк). На шаблонных ГП оба извлекаются механически: «Диагноз:»/«В объеме:»/«ВАЖНО…» → conditions;
careType = classifyCareType(услуги) после мех-извлечения услуг.

## Метод валидации (сильная сторона: эталон уже есть)
Исторические LLM-значения в БД = **готовый золотой корпус**. Каждое новое правило гоняем офлайн по
СОХРАНЁННЫМ файлам кластера (`/opt/sib-storage/attachments/`) и сравниваем с текущими значениями БД:
совпадение ≥98% → правило ок (расхождения смотрим руками — часть будет исправлением ошибок LLM).
Деплой: правило в `extract_dataset.py` → регресс-diff → синк на `/opt/sib-intake`. LLM остаётся
подстраховкой (mech_ok-гибрид D15), поэтому риск деградации ~нулевой, а parse_log покажет дрейф.

## Очередь срезов (по выгоде/простоте)
- **L1** Лучи: маска полиса в POLIS_MASKS (посмотреть формат по файлам, ~7 симв. с пробелом) — 30 мин, −61 LLM-вызов.
- **L2** Росгосстрах ГП-PDF: полис+период одной строкой, «В объеме:»→услуги, «Диагноз:»→conditions, «Период действия ГП»→validUntil.
- **L3** Совкомбанк ГП-PDF: «С:/ПО:»→validUntil-период, «№ полиса» (значение на след. строке), услуги, диагноз.
- **L4** ВСК ГП-PDF: label→next-line пары формы.
- **L5** СОГАЗ направления-PDF: label-таблица формы (даты начала/окончания услуг → coverage).
- **L6** Зетта z-PDF: №ГП/срок/услуги по шаблону (ФИО+ДР уже детерминированы).
- **L7** conditions+careType механически на всех mech-путях (закрывает главный detGap).
- **L8** Ингосстрах/Альфа det+llm метки (letterNumber/validUntil/letterDate).
- **L9** Спот-чек тел Энергогарант/РЕСО → вердикт (механизировать или оставить LLM осознанно).

**Ожидаемый эффект:** LLM-поток 724 → ~150–180 (сканы, свободные тела, новые формы, редкие) ≈ −75%
вызовов на живом приёме; conditions/careType становятся детерминированными на шаблонных ГП; точность
проверяема по золотому корпусу, а не «на глаз».

## Где LLM осознанно ОСТАЁТСЯ
Сканы (vision), новые/сменившиеся формы (self-healing + алерт), свободные нешаблонные тела,
кросс-чек расхождений. Это и есть целевая роль LLM по D10.
