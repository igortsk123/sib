## Цель
Разделить записи реестра на **АМБУЛАТОРИЮ** и **СТОМАТОЛОГИЮ** — у клиники две внутренние системы с разными
возможностями переноса:
- **Стоматология**: встроенный **массовый импорт** пациентов (открепление / прикрепление / создание новых)
  → готовим файл строго под её формат массовой загрузки.
- **Амбулатория**: массового импорта **не будет** в ближайшее время → перенос ручной → даём удобный вид/экспорт
  под ручной ввод (без bulk-формата).

## Подход
**Одна таблица реестра**, деление амбулатория/стоматология — **ФИЛЬТРОМ** по полю `careType` (как фильтры
страховой/статуса). НЕ две таблицы.
**Фильтр в ДВУХ местах:** (1) на **веб-форме** реестра (дропдаун «Направление» рядом со страховой/статусом)
и (2) в **Excel-выгрузке** — экспорт наследует активный фильтр из URL (как уже делают insurer/status/date).
Т.е. выбрал «Стоматология» в вебе → и таблица, и кнопка «Выгрузить в Excel» отдают только стоматологию
(в bulk-формате); «Амбулатория» → ручной вид/выгрузка.

## Что делаем (вертикальный срез UI→БД)
1. **Поле направления** `careType` в `guarantee_letter` (enum `care_type`: `ambulatory|dentistry|other`),
   + миграция. Аддитивно.
2. **Классификация careType** (детерминизм-первой → LLM-фолбэк, по методике D15):
   - по составу `services` / тексту: стоматология — «стоматолог», «зуб», «кариес», «пломб», «эндодонт»,
     «ортопант», «удаление зуба» и т.п.; амбулатория — «поликлиник», «амбулатор», «консультация <спец>»,
     приёмы врачей. Неоднозначно → LLM; не определилось → `other` (ручной выбор).
   - правка вручную в карточке.
3. **Реестр**: фильтр/разрез по `careType` (как страховая/статус), колонка «Направление».
4. **Экспорт СТОМАТОЛОГИЯ** — отдельная выгрузка под массовую загрузку их программы:
   - разрезы **прикрепление / открепление / новые пациенты** (листы или отдельные файлы — по формату программы);
   - колонки строго под их шаблон массового импорта (НУЖЕН образец/спецификация — см. открытые вопросы).
5. **Экспорт/вид АМБУЛАТОРИЯ** — под ручной перенос: чистый список (ФИО, ДР, полис, статус, ссылка на карточку),
   группировка по статусу; bulk-формат не нужен.
6. Учитываем статусы прикрепить/открепить/аннулировать × careType (аннулирование → как открепление в их системе?).

## Открытые вопросы (владельцу — нужны для точного экспорта стоматологии)
1. **Признак деления** амбулатория/стоматология: по услугам в письме? по программе? отдельные ящики? или
   ручной выбор оператором? (гипотеза: по услугам/тексту + ручная правка).
2. **Формат массовой загрузки стоматологической программы** — образец файла/шаблон (колонки, листы,
   разрез по прикрепл/откр/новые, обязательные поля для «создания нового пациента»).
3. Для «создания новых пациентов» — какие поля обязательны их программе (ФИО, ДР, полис, телефон, адрес?).

реестр-фильтр + амбулаторный вид можно сделать сразу; стоматологический bulk — по получении шаблона.
