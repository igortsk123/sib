# План: демо-стенд — анонимизация корпуса до неотличимости

> Решение владельца (2026-07-23): продавать через демо-стенд с тестовым доступом, а не «пришлите
> 20 писем до договора» (медтайна/152-ФЗ). База — реальный корпус с ЗАМЕНОЙ (не перемешиванием!)
> ПДн и в БД, и в файлах-исходниках, чтобы «Источники для сверки» в карточке работали как в бою.

## Принципы (ABSOLUTE)
1. **ЗАМЕНА, не перемешивание**: перемешанные реальные ФИО = всё ещё список реальных пациентов.
2. **Ноль реальных ПДн на выходе** — проверяется автоматическим верификатором (P4), не глазами.
3. **Скоуп шире ФИО+ДР**: полисы, СНИЛС, адреса, телефоны, email-ы людей, № договоров клиники,
   контакты сотрудников страховых. Диагнозы можно оставить (без личности не ПДн) или свапнуть из словарика.
4. **Водяной знак «ДЕМО-ОБРАЗЕЦ»** на каждом регенерированном файле + баннер в UI — фейковые письма
   с реальным брендингом страховых без пометки = риск «поддельных документов».
5. Демо-org полностью изолирован от боевого (мультитенант уже есть); прод-пайплайн не трогаем.

## P0 — PII-инвентарь по форматам (карта «что и где менять»)
| Формат | Что внутри | Метод правки |
|---|---|---|
| xlsx (Ингосстрах/СОГАЗ/РЕСО/Альфа/Ренессанс) | колонки ФИО/ДР/полис/СНИЛС/адрес/тел | перезапись sharedStrings/inline XML — надёжно |
| xls бинарный (Ингосстрах/Ренессанс) | те же | **same-length замена** байтов (cp1251/UTF-16LE) — не ломает смещения |
| pdf текстовый (ГП РГС/Совкомбанк/ВСК/Лучи/Зетта) | ФИО, ДР, полис, диагноз, тел | координаты через `pdftotext -bbox` → растр + плашки + новый текст + watermark (робастно для кириллических CID-шрифтов) |
| pdf-скан | ПДн в пикселях | vision-координаты + растровые плашки, ЛИБО исключить из демо (проще) |
| doc бинарный (Энергогарант) | текст направления | same-length UTF-16LE замена |
| eml (тема/тело/headers) | ФИО в теме (Совкомбанк!), тело, From/To людей | пересборка письма: замены в теме/теле, анонимные вложения, новый Message-ID, пересчёт sha256 |
| zip+пароль (Росгосстрах/Зетта) | вложенные xls/pdf | распаковка → анонимизация → перепаковка с ТЕМ ЖЕ паролем |

## P1 — Генераторы (детерминированные, seed-fixed)
- **ФИО**: словарь фамилий/имён/отчеств с ПОЛОМ и склонениями (в письмах встречается «Уважаемая …а»,
  род.падеж «письмо Иванову») — генерим согласованно по полу; отображение старого→нового 1:1 на весь
  корпус (один человек везде один и тот же фейк — иначе дубли/связки сломаются!).
- **ДР**: случайная в ±3 года от исходной (сохраняет возрастную правдоподобность программ).
- **Полис**: генерация ПО МАСКЕ страховой (`POLIS_MASKS` уже есть) — фейковый номер в родном формате.
- **СНИЛС**: фейковый с ВАЛИДНОЙ контрольной суммой. **Тел/адрес**: генератор (+7 900… / ул. Демонстрационная).
- **№ договора клиники**: фейк по образцу формата. **Даты писем/сроки**: НЕ меняем (реализм таймлайна).

## P2 — Движок подмены (ядро)
1. **Карта замен строится ИЗ НАШЕЙ БД**: по каждому письму демо-выборки берём извлечённые
   patient_full_name/birth/policy (+ raw-строки из parse-контекста) → пары «реальное → фейк».
   Это ключевое преимущество: мы точно знаем, какие строки искать в каждом конкретном файле.
2. Подмена по карте во всех вариантах написания: Title/КАПС, «Фамилия И.О.», склонения (для ФИО
   генерим формы), полис с/без пробелов. Плюс广 grep-подчистка: любые НЕ заменённые ФИО-паттерны
   (FIO2-регэксп) в файле → замена на фейк из пула (ловит адресатов/подписантов/исполнителей).
3. Форматные врайтеры из P0. Выход: `demo-corpus/originals/{emails,attachments}` — зеркальная структура.

## P3 — Сборка демо-корпуса
- Выборка ~60–100 писем: все 13 страховых × их типы (ГП, прикрепление-список, открепление, аннул,
  направление, архив+пароль, «нет данных», дубль-пара, needsReview-кейс) — витрина всех фич.
- Прогон АНОНИМНОГО корпуса через НАШ ЖЕ парсер (extract→enrich→dataset) → `db:seed:corpus` в org
  «Демо-клиника» (CORPUS_ORG_NAME=Демо-клиника, отдельный DATASET_PATH; corpus.ts wipe-логику скоупнуть
  на org! — сейчас `delete(guaranteeLetter)` глобальный — ОБЯЗАТЕЛЬНАЯ правка перед сидом).
- Экспорт/карточки/оригиналы работают из коробки (storage-пути демо-файлов).

## P4 — Верификатор утечек (гейт, автомат)
- Скрипт: все реальные значения из карты замен (ФИО+вариации, ДР, полисы, СНИЛС, телефоны) →
  grep по ВСЕМ демо-артефактам: файлы (текст после экстракции всеми нашими ридерами!), dataset.json,
  БД демо-org, Excel-экспорт. **0 вхождений = зелёный**; иначе стоп.
- Плюс обратная проверка: ни один attachment демо не байт-в-байт равен боевому (sha256 сравнение).
- Ручная выборка: открыть 10 случайных файлов глазами.

## P5 — Доступ и UI
- Вход: ссылка-приглашение с токеном → demo-сессия read-only в org «Демо-клиника» (обход
  Telegram-логина ТОЛЬКО для demo-роли; TTL токена 7 дней, rate-limit).
- Баннер сверху: «ДЕМО-ДАННЫЕ: все лица и номера вымышлены». Экспорт разрешён (это фича витрины).
- «Сообщить об ошибке»/правки — выключены для demo-роли.

## P6 — Деплой и процесс
- Тот же прод-инстанс (отдельный org) — ничего нового в инфре. Гейт полный + e2e на demo-вход.
- Обновление стенда = перегенерация корпуса скриптом (идемпотентно, seed фиксирован).
- Память: ADR «демо-стенд и правила синтетики», project-state, /memory-check.
