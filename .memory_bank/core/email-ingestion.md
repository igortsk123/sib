# Забор писем — Tier 1 сводка

## Подключение
- **MVP:** один тестовый Яндекс-ящик `palmarius.ru@yandex.ru`, **IMAP `imap.yandex.ru:993` XOAUTH2**
  (доступ подтверждён — ADR D2; креды `_secrets/ACCESS.md`). Поллинг — Inngest cron-функция, не webhook.
- **Рабочий этап:** 4 ящика клиники (могут быть не-Яндекс) — параметры/доступ = `core/human-decisions.md`.
- Хранить per-ящик: тип подключения, статус, дата синка, счётчики обработано/ошибок, правила, разрешённые отправители.

## Двойная пересылка (ADR D5 — первоклассный кейс)
Тестовые письма дважды пересланы (клиника→разработчик→тестовый ящик). На каждом письме различать:
- технического отправителя последней пересылки **vs** реального исходного отправителя (страховую);
- исходные дату/тему vs данные пересылки;
- реальные вложения (вкл. nested `.eml` / `message/rfc822`).
Парсить блоки `Forwarded message` / `Пересылаемое сообщение` / `От кого`/`Кому`/`Дата`/`Тема`. Логика
остаётся резервной и на рабочем этапе.

## Дедупликация (ADR D6 — помечать, не удалять)
Сигналы дубля: Message-ID исходного письма; отправитель+тема+дата; hash вложения; № гарантийного письма;
пациент+страховая+дата+услуги; совпадение исходного текста. Помечать в админке, авто-удаление запрещено.

## Хранение и классификация
Сохранять оригинал письма, тех. заголовки, тело (text+html), вложения — до распознавания (проверяемость).
Тип письма: гарантийное / пароль к архиву / отказ / запрос инфо / служебное / нерелевантное.
