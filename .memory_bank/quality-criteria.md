# Quality Criteria — гейт и зрелость качества

> Конституция (`.claude/rules/engineering-principles.md`) — что обязательно. Здесь — детали гейта и как
> поднимать зрелость, не ломая продукт. Эталоны по областям — `reference/quality-standards/*`.

## Гейт готовности среза (обязателен перед «готово»)
`typecheck` + `lint` + `unit` + `integration` + `e2e` + `build`. **Красный = не готово.**
- unit — на чистую логику (парсинг пересылки, нормализация дат/полисов, сопоставление архив↔пароль, дедуп).
- e2e — на критический поток среза (напр. «письмо с тестового ящика → запись в реестре»).
- В CI: гейт зелёный → мёрж; деплой только на зелёном.

## Измеримость
- Критерий без зелёной проверки/порога не засчитан. «Частично» = не сделано.
- Сдвиг проверок влево: нарушение ловит машина (типы/линт/тест/CI), не ревьюер (владелец не ревьюит построчно).

## Сеть безопасности (когда появится прод — ПЕРВЫМ)
Health-эндпоинт; деплой со smoke + авто-откат; бэкап + restore-drill; pre-deploy гейты. Детали — `deployment.md`.
ПДн (ADR D4): миграции аддитивны (expand-only); бэкап перед опасными операциями над данными.

## Эталоны по областям (`reference/quality-standards/`)
- `web_frontend.md` — админка (состояния экрана, доступность, анти-гонки).
- `backend_service.md` — server actions / Inngest-функции (идемпотентность, таймауты, обработка ошибок).
- `sql.md` — Postgres/Drizzle (инварианты/CHECK, индексы, тотальный ORDER BY).

## Отложенные триггеры «если X → применить Y»
- Рост объёма писем → партиционирование / keyset-пагинация реестра.
- Публичный API для клиники → версионирование + contract-тесты.
- Отдельный клиент (mobile/внешний SPA) или публичный типизированный API → ввести **tRPC** (ADR D7).
- Несколько клиник (мультиарендность) → RLS по `org_id`.
- Много типов фоновой обработки → вынести обработчики из Next в отдельный сервис.
