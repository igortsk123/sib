# Распознавание страховых — эмпирика тестового корпуса

> Получено разбором реального тестового набора (`.mail-intake/`, gitignored, ПДн). Это **факты с
> реальных писем**, отменяющие догадки спеки (см. ADR D10). Домены страховых — корпоративные, не ПДн.

## Структура корпуса (как реально приходит на тесте)
Один **контейнер** от `igortsk@mail.ru` (тема `Fwd: Страховая Балт Ассистанс.eml (+12 писем)`) →
**13 вложений `message/rfc822`** (по одной страховой) → внутри каждого **N писем** (всего **51**) →
каждое письмо — **полное оригинальное письмо страховой, закодированное base64 внутри `text/plain`
БЕЗ заголовка `Content-Transfer-Encoding`** (стандартный парсер его не раскрывает). Иногда ещё слой
и/или у оригинала **срезан `From`** (остаётся `multipart/alternative` без заголовков отправителя).

**Вывод для парсера (критично):** рекурсивная раскрутка —
`message/rfc822` → детект-и-декод **base64-письма в text/plain даже без CTE** → повтор, до оригинала;
при срезанном `From` отправителя ищем в **теле / Received / глубоким сканом**, а не только в заголовке.
Имя файла `Страховая X.eml` в контейнере — **ручная подпись отправителя теста, НЕ продакшн-сигнал**.

## Стратегия идентификации страховой (многоуровневая — подтверждена данными)
1. **Домен отправителя оригинала** — главный признак (работает для большинства).
2. **Реестр-алиасы (конфиг, не угадывание):** домен ≠ название — `calltravel.eu`=Балта,
   `luchi.ru`=Лучшее, `astrovolga.ru`=Астро-Волга, `renhealth.com`=Ренессанс, `sovcomins.ru`=Совкомбанк.
   → таблица `InsuranceCompany.domains[]` (бриф §4) **обязательна**, заполняется этими значениями.
3. **Фолбэк по телу/Received**, когда `From` срезан пересылкой (Лучшее, Энергогарант, Астро-Волга).
   Внимание: в теле много **не-страховых** доменов (клиники Томска: gb3.ru, medsan3.ru, mozdrav.ru,
   gztomsk.ru, trecom.tomsk.ru…) → матчим **только по реестру страховых**, не «любой внешний домен».
4. **Фолбэк по ключевым словам/названию** в теме/теле — крайний случай.

## Форматы и обработка (что реально встречается)
- **Тело-only** (text и/или html) — есть у всех; часть писем без вложений (инфа в теле).
- **PDF** — Балта, ВСК, Совкомбанк, Росгосстрах. **Excel xls/xlsx** — Ингосстрах, СОГАЗ, РЕСО, Альфа,
  Ренессанс. **RTF** — РЕСО. **ZIP** — Зетта, Росгосстрах.
- **Архив+пароль** — Росгосстрах (zip + признак пароля). Сценарий брифа §5.4 реален и сконцентрирован тут.
- HTML-тело частое → нормализация HTML→текст обязательна перед извлечением полей.

## ПРОД vs ТЕСТ (важно — определяет конвейер)
- **Прод (будущее):** 4 ящика клиники, письма приходят **по одному напрямую** от страховых. Задача —
  быстро забрать по IMAP (инкрементально по UID на ящик), определить страховую, распарсить по её формату,
  сохранить. **Раскрутки пересылки/base64 в проде НЕТ.**
- **Тест (сейчас):** весь корпус пришёл одним контейнером пересланных `.eml` (base64). Раскрутка — это
  **тест-адаптер ingestion** (загрузить корпус как фикстуры), а НЕ ядро прода. Логика пересылки остаётся
  резервной (ADR D5), но не центральная.

### Сквозные правила парсинга (на чём легко ошибиться)
- **Многострочный Excel = 1 письмо → N записей** (РЕСО 17–32, Альфа 2–11). Модель данных и парсер
  обязаны поддерживать `EmailMessage 1—N GuaranteeLetter`. Не считать «одно письмо = одно ГП».
- **Excel двух видов:** старый **.xls** (OLE2, magic `D0CF11E0`) у Ингосстрах/СОГАЗ/Росгосстрах vs
  **.xlsx** (zip, `PK`) у РЕСО/Альфа → брать **SheetJS (xlsx npm)** — читает оба; exceljs только xlsx.
- **Зашифрованные архивы** (Зетта zip→PDF, Росгосстрах zip→xls). **Пароль ОБЫЧНО приходит отдельным
  письмом — это норма, а не край** (подтверждено владельцем). Исключение: Росгосстрах иногда даёт пароль
  в теле (паттерн `rgs…`).
  - Связка Зетта — по **№ ГП** (в имени файла `ГП<№>.pdf` и в теле письма-пароля «ГП<№>»).
  → В проде письма приходят по одному ⇒ **stateful-корреляция — первоклассный механизм**: письмо-архив
    кладём в очередь «ждёт пароль» (ключ: № ГП / отправитель+тема+близость по времени), письмо-пароль
    ищет свой архив (и наоборот). Расшифровка — когда пара собрана. Ручной фолбэк связывания обязателен.
- **PDF:** у большинства есть текстовый слой → `pdftotext`. **Скан** (Балта ×1) → **OCR** (tesseract rus).
- **Тело-only** (Лучшее, Астро-Волга, Энергогарант): данные в HTML/текст → нормализация HTML→текст → LLM.
- **Имена файлов несут сигнал:** `ГП<номер><дата>.pdf` (Зетта) — № и дата ГП; использовать как кросс-чек.

## Открытые точки (проверить при срезах)
- Точные адреса/доменные варианты (поддомены, smtp-релеи) каждой страховой — дополнять реестр по мере писем.
- Где именно лежит «пароль к архиву» (в том же письме / отдельном) — у Росгосстраха проверить на срезе архивов.
- Многопациентные письма / несколько ГП в одном файле — бриф §9.4, проверить на полном объёме.
