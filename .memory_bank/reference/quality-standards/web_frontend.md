# Качество web-фронтенда — эталонные критерии (project-agnostic)

> **Назначение.** Независимый от стека и проекта свод лучших практик построения
> web-фронтенда: эталон для ревью, планирования и оценки зрелости любого
> SPA/SSR-приложения. Не привязан к конкретному фреймворку — формулировки даны на
> уровне принципов, а инструменты/команды приведены как примеры.
>
> **Как пользоваться.** Каждое измерение: `Принцип` (зачем) → `Критерии` (что
> считаем качеством, проверяемо; каждый критерий имеет **стабильный ID** вида
> `SEC-1`) → `Сигналы и проверки` (чем мерить: lint/CI/grep/ручное ревью) →
> `Антипаттерны` (что считаем дефектом). При планировании берём релевантные
> критерии как acceptance-условия и ссылаемся на них по ID.
>
> **Шаблон ↔ overlay.** Этот файл — переносимый **шаблон**. Проектную привязку
> (конкретные обёртки, стек, текущий baseline и разрывы) держим в отдельном
> **overlay**-документе рядом — `frontend-quality-criteria.<project>.md` — где каждый
> ID критерия привязан к месту в коде, спринту и честной оценке baseline. ID
> бриджат шаблон и оверлей: оверлей ссылается на `SEC-1`/`PERF-2`/… не переписывая
> формулировки. Шаблон остаётся чистым от проектной специфики.
>
> **Каноны, на которые опираемся:**
> - Доступность: WCAG 2.2 (уровень AA), WAI-ARIA Authoring Practices.
> - Производительность/UX: Core Web Vitals (web.dev), RAIL-модель, бюджеты ресурсов.
> - Безопасность: OWASP Top 10, OWASP Cheat Sheets (XSS Prevention, DOM-based XSS,
>   Clickjacking Defense, CSRF Prevention, File Upload), Trusted Types, MDN Web
>   Security; приватность — GDPR/CCPA (consent, минимизация, retention).
> - Адаптивность/кросс-браузерность: Responsive Web Design (web.dev), MDN
>   cross-browser testing, прогрессивное улучшение, WCAG target-size.
> - Интернационализация: Unicode/UTF-8, ECMA-402 `Intl`, ISO 8601, IANA TZ, ISO 4217, ICU/CLDR.
> - Discoverability/SEO: HTML metadata, Open Graph/Twitter Cards, schema.org
>   (structured data), canonical/hreflang, robots/sitemaps (для индексируемых поверхностей).
> - Наблюдаемость: three pillars (logs/metrics/traces), RUM, OpenTelemetry (web).
> - Поставка: 12-Factor (config/build-release-run), SLSA (целостность сборки),
>   DORA-метрики (deploy frequency, lead time, MTTR, change-fail rate), SRI.
> - Тестирование: Testing Trophy / пирамида тестов, consumer-driven contracts,
>   visual regression.
> - Код: Clean Code, SOLID/DIP (по месту, без over-engineering), composition over inheritance.
>
> **Отличие от классического 9-мерного чек-листа фронта.** К базовому набору
> (Security, Performance, Modularity, Extensibility, Testability, Maintainability,
> a11y, Architecture, Reuse) добавлены ранее упущенные сквозные аспекты, помеченные
> ★: **Параллелизм и async-корректность**, **Сетевая устойчивость**,
> **Интернационализация**, **Наблюдаемость (клиент)**, **Поставка и целостность
> сборки**, **Документация и DX**, **Адаптивность и кросс-девайс**, **Навигация и
> URL-состояние**, **Формы и ввод**, **Discoverability / SEO**. Они применимы к
> большинству продакшен-фронтов; неприменимость фиксируется явно (N/A +
> обоснование, см. Definition of Done).

Шкала зрелости каждого измерения: **0** нет практики · **1** точечно ·
**2** есть, но без enforcement · **3** enforced (lint/CI/тест) · **4** эталон.

Базовый принцип сквозной для всех измерений: **чинить корень причины, а не
симптом**; **открыт к расширению, закрыт к модификации**; преждевременную
абстракцию не вводим (правило «трёх повторов»).

## Часть 0. Бюджеты UX / Web Vitals как корень

Best practice (по аналогии с SRE-SLO): сначала фиксируем пользовательские бюджеты
производительности и надёжности UI — остальные критерии (PERF, RESIL, OPS, A11Y)
производны от них. Конкретные цифры подтверждаются с продуктом.

| Метрика (SLI-аналог) | Цель (дефолт, уточнить) |
|----------------------|-------------------------|
| LCP (Largest Contentful Paint) | < 2.5 s (p75, field) |
| INP (Interaction to Next Paint) | < 200 ms (p75, field) |
| CLS (Cumulative Layout Shift) | < 0.1 (p75, field) |
| TBT (Total Blocking Time, lab) | < 200 ms |
| Бюджет initial JS (gzip/brotli) | ≤ согласованный (напр. ≤ 200 KB) |
| Crash-free sessions / частота JS-ошибок | ≥ 99.9% / < 0.1% сессий |
| Критичные нарушения a11y (axe) на ключевых страницах | 0 |
| Покрытие критичных пользовательских флоу тестами | ≥ согласованный порог |
| Поддерживаемая матрица браузеров/устройств | согласованный список (напр. 2 последние версии evergreen) без критичных дефектов |
| Потеря несохранённых данных формы при навигации/обновлении | 0 (guard) |

Эти бюджеты — корень: их нарушение блокирует релиз (см. Definition of Done), а
PERF/RESIL/OPS-критерии описывают, как их держать.

## Часть I. Сквозные измерения

### 1. Безопасность (SEC)

**Принцип.** Браузер исполняет всё, что мы отрендерили; любой непроверенный ввод —
потенциальный вектор. Поверхность атаки закрывается на уровне инфраструктуры, а не
точечно.

**Критерии**
- **SEC-1.** Нет небезопасного рендера HTML из данных. Вставка сырого HTML
  (`v-html`, `dangerouslySetInnerHTML`, `Element.innerHTML`) — только для
  доверенного статического контента; любой динамический проходит через текстовые
  узлы или санитайзер (DOMPurify) с явным allow-list тегов/атрибутов. Где
  поддерживается — включён **Trusted Types** (CSP `require-trusted-types-for`),
  переводящий DOM-XSS-sink'и в ошибку на уровне платформы.
- **SEC-2.** Аутентификация централизована: refresh/401/logout — в одной точке
  (api-слой/interceptor), не размазаны по компонентам. Способ хранения токена
  выбран осознанно и задокументирован (httpOnly-cookie против `localStorage`:
  cookie защищён от XSS, но требует CSRF-защиты; LS уязвим к XSS, но прост — выбор
  фиксируем в ADR).
- **SEC-3.** Внешние URL/редиректы и query-параметры собираются безопасно (билдер/
  `qs`/`URLSearchParams`), без конкатенации сырого ввода. Open-redirect исключён
  валидацией целевого URL по allow-list.
- **SEC-4.** Все `target="_blank"` имеют `rel="noopener noreferrer"` (защита от tabnabbing).
- **SEC-5.** CSP и security-заголовки заданы и минимальны (`Content-Security-Policy`,
  `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Strict-Transport-Security`).
  Список внешних доменов (CDN, карты, аналитика) задокументирован и ограничен.
- **SEC-6.** Секреты не в коде/репозитории и не в клиентском бандле: ключи — через
  env/runtime-config; в логах нет токенов/PII.
- **SEC-7.** Зависимости проверяются на известные уязвимости (`npm audit`/Dependabot/
  Snyk, SCA); lock-файл закоммичен.
- **SEC-8.** Приватность: нет PII в URL/`localStorage`/логах/телеметрии; на клиенте
  хранится только необходимое (data minimization). Аналитика и трекинг — только по
  согласию пользователя (consent-gated, GDPR/CCPA) [OPS-5].
- **SEC-9.** Сторонние скрипты/стили с CDN подключаются с SRI (Subresource
  Integrity) и pinned-доменом; целостность поставки контролируется [REL-6].
- **SEC-10.** Защита от clickjacking: встраивание в чужие фреймы запрещено
  (`Content-Security-Policy: frame-ancestors` + `X-Frame-Options`); обратное —
  список разрешённых эмбеддеров явный и минимальный [SEC-5].
- **SEC-11.** CSRF-защита осознанна и соответствует способу хранения сессии: для
  cookie-аутентификации — анти-CSRF-токен или `SameSite`-куки; CORS настроен по
  allow-list, без `*` для credentialed-запросов [SEC-2].
- **SEC-12.** Загрузка файлов валидируется на клиенте как первый барьер (тип по
  расширению **и** MIME, лимит размера, имя), с пониманием, что итоговая проверка —
  на сервере; превью пользовательских файлов не создаёт XSS-вектор (SVG/HTML).

**Сигналы и проверки**
- `grep -rIn "v-html\|innerHTML\|dangerouslySetInnerHTML" src` → каждый случай ревьюится поимённо.
- `grep -rIn 'target="_blank"' src` → у всех есть `rel="noopener"`.
- `grep -rInE "API_KEY|SECRET|Bearer |token *=" src` (вне env/runtime-config) → 0.
- `npm audit --production` без high/critical; CSP (вкл. `frame-ancestors`) присутствует в ответах (DevTools → Network → Headers).
- Ревью загрузчиков файлов: тип/размер проверяются; SVG/HTML-превью санитизированы.
- Ревью: ни один лог/событие аналитики не печатает токен/PII; согласие проверяется перед трекингом.

**Антипаттерны**
- Рендер ответа API через `v-html`/`innerHTML` без санитайзинга.
- Refresh-токен и обработка 401 продублированы в нескольких местах.
- Ключи/секреты в исходниках или в публичном бандле; `target="_blank"` без `rel`.
- PII/идентификаторы в query-строке; трекинг до получения согласия.
- Нет `frame-ancestors`/`X-Frame-Options` — приложение встраивается в чужой iframe.
- Cookie-сессия без CSRF-защиты; CORS с `*` для credentialed-запросов.
- Аплоад без проверки типа/размера; рендер загруженного SVG как разметки.

### 2. Производительность (PERF)

**Принцип.** Пользователь платит за каждый килобайт и за каждый долгий таск в main
thread. Грузим только нужное и только тогда, когда нужно.

**Критерии**
- **PERF-1.** Code-splitting: тяжёлые/редкие модули (карты, графики, редакторы, PDF/
  экспорт, drag-n-drop) — ленивые (`import()`/`React.lazy`/`defineAsyncComponent`);
  роуты разбиты по чанкам. Vendor выделен отдельно и кэшируется.
- **PERF-2.** Бюджет бандла зафиксирован и проверяется в CI (size-limit/bundlewatch);
  регрессия размера ломает сборку [REL-5]. Анализ — `build --analyze`/`rollup-plugin-visualizer`.
- **PERF-3.** Большие списки/таблицы (>~100 строк) — виртуализация или серверная
  пагинация; не рендерим весь набор. Частые апдейты (WebSocket/polling) не вызывают
  каскадных перерисовок (стабильные ключи, точечная реактивность, батчинг).
- **PERF-4.** Нет дорогих вычислений в шаблоне/рендере без мемоизации; нет глубоких
  `watch`/подписок на больших структурах без необходимости.
- **PERF-5.** Изображения оптимизированы: современные форматы (WebP/AVIF),
  `srcset`/`sizes`, `loading="lazy"`, заданные размеры (нет CLS). Иконки —
  tree-shakeable (отдельные импорты, не целый шрифт/спрайт).
- **PERF-6.** Core Web Vitals под контролем: LCP < 2.5s, INP < 200ms, CLS < 0.1 (lab
  + field, см. Часть 0). Критический CSS/шрифты не блокируют первый рендер
  (`font-display: swap`, preload).
- **PERF-7.** Гигиена памяти: подписки/слушатели/таймеры/observers очищаются на
  unmount; нет detached-DOM и «вечно живущих» замыканий. Клиентские кеши ограничены
  по размеру + TTL/eviction, не растут безгранично [STATE-5].
- **PERF-8.** Воспринимаемая производительность: на ожидание — скелетоны/прогрессивный
  контент вместо пустого экрана или прыгающего спиннера; мгновенный отклик на
  действие (оптимистичный фидбек/disabled-состояние) [RESIL-2, RESIL-5]; переходы
  плавные, без джанка, и уважают `prefers-reduced-motion` [A11Y-5]. Зарезервировано
  место под асинхронный контент (нет CLS при дозагрузке) [PERF-5].

**Сигналы и проверки**
- `build --analyze` → нет неожиданных монолитных чанков; тяжёлые либы не в main-бандле.
- `grep -rIn "() => import(\|React.lazy\|defineAsyncComponent" src` → растёт по мере выноса тяжёлого.
- `grep -rIn "addEventListener\|setInterval\|new ResizeObserver" src` → у каждого есть парная очистка на unmount.
- Lighthouse/WebPageTest на ключевых страницах → Web Vitals в зелёной зоне.
- DevTools Performance/Memory → нет long tasks >50мс и нет монотонного роста heap на навигации туда-обратно.

**Антипаттерны**
- Всё грузится синхронно; карты/charts/PDF в main-бандле.
- Бюджет бандла нигде не зафиксирован — размер растёт незаметно.
- Рендер тысяч DOM-узлов без виртуализации; изображения без размеров (скачет layout).
- `addEventListener`/`setInterval` без снятия на unmount → утечка памяти.

### 3. Модульность (Modularity / MOD)

**Принцип.** Систему меняют по частям. Чёткие границы и однонаправленные
зависимости позволяют менять модуль, не ломая соседей.

**Критерии**
- **MOD-1.** Доменная нарезка по фичам (`features/<feature>` / `modules/<feature>`):
  самодостаточны, не лезут во внутренности друг друга. Общение между фичами — через
  общий слой состояния, события или публичный API модуля, не через глубокие импорты
  чужих внутренностей.
- **MOD-2.** UI-кит/design-system не зависит от бизнес-фич (направление
  зависимостей: `features → ui-kit → framework`, не наоборот).
- **MOD-3.** Сквозная инфраструктура (http-клиент, нотификации, тема, диалоги, шина
  событий) — в общем слое (`plugins`/`composables`/`hooks`/`services`), а не
  размазана по фичам.
- **MOD-4.** У модуля явная граница: компоненты + локальные стор/хуки/типы;
  публичный вход — через index/barrel, приватное наружу не торчит. Доменная логика
  не утекает в общий UI-кит.

**Сигналы и проверки**
- `grep -rIn "from '.*features/" ui-kit` → 0 (UI-кит не знает о фичах).
- Линт границ: `eslint-plugin-import` (`no-internal-modules`, `no-restricted-imports`)
  или `dependency-cruiser` — глубокие кросс-фичевые импорты запрещены правилом.
- Ревью: новая фича добавляется как папка, без правок чужих модулей.

**Антипаттерны**
- Импорт `features/a/internal/util` из `features/b` (минуя публичный API).
- UI-кит импортирует доменные типы/сторы.
- «Общая» папка `shared/`, в которой накопилась бизнес-логика конкретных фич.

### 4. Расширяемость (Extensibility / EXT)

**Принцип.** Новые варианты добавляются данными и конфигом, а не копированием кода
и `if`-развилками.

**Критерии**
- **EXT-1.** Темизация/white-label/локализация — через данные и конфиг (токены,
  theme-объект, i18n-ресурсы), без `if (brand === 'X')`-развилок по коду [I18N-1].
- **EXT-2.** Новый тип сущности/колонки/фильтра/шага формы добавляется декларативно
  (схема/конфиг), а не клонированием компонента.
- **EXT-3.** Обёртки/компоненты расширяются пропами и **именованными слотами**
  (`#header`/`#actions`/`#empty`), а не форками. Точки расширения явные.
- **EXT-4.** Открыт к расширению, закрыт к модификации: добавление варианта не
  требует правки ядра компонента. Поведение настраивается инверсией зависимостей
  (передаём стратегию/рендер-функцию/слот), а не ветвлением внутри.

**Сигналы и проверки**
- `grep -rInE "brand ===|theme ===|=== '(gc|prod|stg)'" src` → стремимся к 0 (различия — в конфиге/токенах).
- Ревью: добавление колонки/фильтра — через проп/конфиг, без нового компонента-клона.
- Число именованных слотов/точек расширения у базовых компонентов не падает.

**Антипаттерны**
- Развилки по бренду/теме/окружению, разбросанные по компонентам.
- Копия таблицы/формы «почти как та, но с одной колонкой иначе».
- Хардкод вместо слота там, где потребителю нужно вставить свой контент.

### 5. Тестируемость (Testability / TEST)

**Принцип.** Код, который трудно тестировать, обычно плохо спроектирован. Тест —
это в первую очередь проверка дизайна, и только потом — регрессии.

**Критерии**
- **TEST-1.** Каждый пользовательский сценарий покрыт тестом подходящего уровня
  (пирамида: много unit, меньше component/integration, точечно e2e критических флоу).
- **TEST-2.** Бизнес-логика вынесена из компонентов в чистые функции/хуки/сторы →
  тестируется без монтирования DOM.
- **TEST-3.** Компоненты детерминированы: побочки (сеть/время/рандом/`crypto`)
  инжектируются и мокаются; время — через фиксируемый клок. Тесты не зависят от
  порядка/таймзоны.
- **TEST-4.** Интеграционные слои тестируются против реальной схемы/контракта
  (consumer-driven contracts/MSW), а не «угаданного» мока. Запросы к сети
  перехвачены, реальные внешние вызовы → 0.
- **TEST-5.** Селекторы для e2e стабильны (`data-testid`/роли ARIA), не привязаны к
  тексту и вёрстке. Тесты проверяют поведение, а не реализацию.
- **TEST-6.** Визуальная регрессия для design-system/ключевых экранов (snapshot/
  Chromatic/Percy/Playwright screenshots): непреднамеренный сдвиг вёрстки или темы
  ловится автоматически, а не глазами в проде. Покрываются состояния компонента
  (hover/disabled/error) и темы (light/dark/RTL) [UIKIT-5, A11Y-3].

**Сигналы и проверки**
- CI: тесты зелёные; покрытие новых сценариев не падает (порог в `coverage`).
- `find . -name "*.spec.*" -o -name "*.test.*" | wc -l` растёт вместе с фичами.
- Ревью: PR со сценарием без теста не «done».
- Запросы к сети в тестах перехвачены (MSW/nock); реальные внешние вызовы → 0.

**Антипаттерны**
- Логика живёт в компоненте → тест требует поднимать весь DOM ради проверки расчёта.
- `setTimeout`/реальное время/`Date.now()` без фиксации клока → флакающие тесты.
- e2e-селекторы по тексту кнопки/CSS-классу — ломаются от любого редизайна.

### 6. Поддерживаемость (Maintainability / MAINT)

**Принцип.** Код читают в разы чаще, чем пишут. Единый стиль и малый размер единиц
снижают стоимость каждого следующего изменения.

**Критерии**
- **MAINT-1.** Единый стиль кода, enforced: ESLint + Prettier зелёные в CI и
  pre-commit; формат не обсуждается в ревью (его проверяет машина).
- **MAINT-2.** Чиним корень причины, не симптом. Никаких inline-`style` с
  магическими числами, никакого хардкода цвета — только дизайн-токены; типографика
  и spacing — из единой шкалы (например, шаг 4px).
- **MAINT-3.** Временные обходы/back-compat помечаются грепаемым маркером
  (`TODO(<scope>)`/`FIXME`) с условием удаления и вычищаются; «шим на всякий случай»
  не оставляем.
- **MAINT-4.** Один коммит — одна логическая единица; conventional commits; PR
  небольшие и обозримые.
- **MAINT-5.** Файлы-монолиты декомпозируются; компонент/функция — одна
  ответственность; имена говорящие, комментарии объясняют WHY (неочевидный
  инвариант/обход), не WHAT; мёртвый код удаляется, а не комментируется.
- **MAINT-6.** Типобезопасность: строгий TypeScript (`strict`), `any` —
  обоснованное локализованное исключение.

**Сигналы и проверки**
- `npm run lint` / `tsc --noEmit` → 0 ошибок (в CI как gate).
- `grep -rInE 'style="[^"]*[0-9]+px' src` (магические числа) → ревью/0.
- `grep -rInE '#[0-9a-fA-F]{6}' src` (хардкод цвета вне токен-конфига) → 0.
- `grep -rIn "TODO(\|FIXME" src` → известный, ограниченный список с владельцами.

**Антипаттерны**
- Форматные правки вперемешку с логикой в одном коммите.
- Хардкод hex/px вместо токенов; «магические числа» в стилях.
- Компонент на 800 строк с пятью ответственностями; закомментированный код «на потом».

### 7. Доступность (a11y / A11Y)

**Принцип.** Интерфейс обязан работать без мыши и с ассистивными технологиями.
Дешевле всего чинить на уровне общих компонентов — один источник, массовый эффект.

**Критерии**
- **A11Y-1.** Семантика по умолчанию: используем нативные элементы (`<button>`,
  `<a>`, `<nav>`, `<label>`); кастомные интерактивы получают корректный `role` и
  `aria-*`. Иконочные кнопки — с `aria-label`.
- **A11Y-2.** Полная клавиатурная навигация: фокус виден, порядок логичен, нет
  ловушек. Оверлеи/модалки — focus-trap, возврат фокуса на триггер, `Esc` закрывает.
- **A11Y-3.** Контраст текста и контролов соответствует WCAG 2.2 AA (≥4.5:1 для
  текста) на всех темах (light/dark/high-contrast).
- **A11Y-4.** Формы: связь `<label>`↔контрол, ошибки объявляются (`aria-invalid`,
  `aria-describedby`), обязательность — `required`/`aria-required`.
- **A11Y-5.** Динамика анонсируется через `aria-live` (тосты, загрузка, валидация);
  анимации уважают `prefers-reduced-motion`.
- **A11Y-6.** Изображения имеют `alt` (или `alt=""` для декоративных); язык страницы
  задан (`lang`).

**Сигналы и проверки**
- `eslint-plugin-jsx-a11y` / `eslint-plugin-vuejs-accessibility` в CI.
- axe DevTools / Lighthouse a11y / Pa11y на ключевых страницах → без критичных нарушений (Часть 0).
- Ручная проверка: пройти основной флоу только с клавиатуры и со скринридером.

**Антипаттерны**
- `<div @click>` вместо `<button>`; иконочная кнопка без `aria-label`.
- Модалка без focus-trap/возврата фокуса; недоступная по `Esc`.
- Цвет — единственный носитель смысла (ошибка только красным, без текста/иконки).

### 8. Чистота архитектуры (Architecture / ARCH)

**Принцип.** Зависимости текут в одну сторону, абстракции изолируют фреймворк, а
значимые решения зафиксированы. Это и есть то, что позволяет жить годами.

**Критерии**
- **ARCH-1.** Слои и направление зависимостей явные: `pages/routes → features →
  ui-kit → framework`; общее состояние и инфраструктура — сквозные. Обратных и
  циклических зависимостей нет.
- **ARCH-2.** Фреймворк UI изолирован за собственным слоем абстракции
  (design-system/обёртки): потребители работают с нашим API, не с примитивами
  вендора напрямую — это упрощает мажорные апгрейды и смену вендора. Прямой вызов
  вендорных примитивов в фичах — исключение, не норма.
- **ARCH-3.** Состояние: единый источник истины (store); компоненты его не
  дублируют. Серверный кэш (data-fetching) и UI-state разделены, не путаются.
- **ARCH-4.** Значимые решения зафиксированы как ADR; никаких feature-flag/shim «на
  всякий случай». Флаг живёт с условием и датой выпиливания [REL-4].
- **ARCH-5.** Нет утечек абстракции: проброс `props`/`attrs`/слотов осознан и
  контролируем.
- **ARCH-6.** Обработка ошибок и загрузок — системная (error boundary, единый
  загрузочный/пустой state), а не ad-hoc в каждом компоненте [RESIL-2, RESIL-7].
- **ARCH-7.** Ответы API валидируются на границе слоя данных (схема/zod/io-ts);
  форме ответа не доверяем слепо, единый разбор error-envelope. Несоответствие
  контракту — явная ошибка, не «тихий» сбой рендера [STATE-3].

**Сигналы и проверки**
- `dependency-cruiser`/`madge --circular` → 0 циклов; правила направлений в CI.
- Инвентарь прямых вендорных вызовов мимо design-system → тренд вниз.
- Каждое архитектурное изменение → запись в `decisions/` (ADR).
- Ревью data-слоя: ответы проходят через валидатор схемы перед использованием.

**Антипаттерны**
- Циклические зависимости между фичами/слоями.
- Вендорные компоненты раскиданы по фичам — мажорный апгрейд невозможен без массовой правки.
- Дублирование источника истины (одно и то же состояние в store и в локальном `ref`).
- Флаги/шимы без срока жизни, «вдруг пригодится»; слепое доверие форме ответа API.

### 9. Повторное использование (Reuse / REUSE)

**Принцип.** Один способ сделать вещь. Каноничные примитивы и общие хелперы
устраняют дрейф и дублирование, но абстрагируем только по факту повтора.

**Критерии**
- **REUSE-1.** Канонические UI-примитивы единственны: одна реализация секции/
  карточки, таблицы, диалога, поля формы. Самодельные дубликаты канонического
  компонента не плодим.
- **REUSE-2.** Правило трёх: повтор ≥3 раз → выносим в utility/composable/хук/класс;
  <3 — не абстрагируем преждевременно (дешевле скопировать, чем угадать неверную абстракцию).
- **REUSE-3.** Общие хелперы — в `utils`/`composables`/`hooks`, не копипастятся
  между фичами.
- **REUSE-4.** Дизайн-токены и типографика — единый источник; локальные
  переопределения токенов запрещены.
- **REUSE-5.** Параллельные/legacy-версии одного компонента (`v2/*`, `*-new`) —
  временные, с явным планом схлопывания в один wrapper и сроком.

**Сигналы и проверки**
- `jscpd` (детектор копипаста) → дубли ниже порога; повтор ≥3 → задача на вынос.
- Инвентарь самодельных обёрток vs каноничный примитив → миграция к одному.
- Ревью: вторая копия логики помечается, третья — выносится.

**Антипаттерны**
- Три почти одинаковых «карточных» обёртки в трёх фичах.
- Преждевременная абстракция: универсальный хелпер с 6 флагами ради двух вызовов.
- Долгоживущие `v2/*` без плана схлопывания — вечный дубль.

### 10. ★ Параллелизм и async-корректность (Concurrency / CONC)

**Принцип.** Клиент асинхронен: сеть, таймеры и события приходят в неизвестном
порядке. Корректность состояния не должна зависеть от тайминга.

**Критерии**
- **CONC-1.** Out-of-order ответы обрабатываются: результат устаревшего запроса не
  перетирает свежий (latest-wins/отмена/сверка по версии запроса). Классика —
  поиск/автокомплит/фильтры/смена вкладок.
- **CONC-2.** Запросы отменяются на unmount/навигации/смене параметров
  (`AbortController`/cleanup); нет обновления состояния после размонтирования.
- **CONC-3.** Нет stale-closure багов в подписках/таймерах/обработчиках: актуальные
  значения берутся осознанно (ref/свежее замыкание), а не «замораживаются».
- **CONC-4.** Частые события (ввод, скролл, resize, drag) debounce/throttle; нет
  шторма запросов/перерисовок [PERF-3].
- **CONC-5.** Конкурентные обновления общего стора не теряют апдейты (атомарные
  мутации, нет read-modify-write гонок на одном поле) [STATE-5].
- **CONC-6.** Согласованность между вкладками: вход/выход/смена токена/критичные
  изменения данных синхронизируются между открытыми вкладками (`storage`-event/
  `BroadcastChannel`); нет «залогинен в одной, разлогинен в другой» рассинхрона [SEC-2].

**Сигналы и проверки**
- `grep -rIn "AbortController\|onUnmounted\|onBeforeUnmount" src` → есть отмена/cleanup у data-fetching.
- Ревью search/autocomplete/фильтров на гонку порядка ответов.
- Тест: ответы приходят в обратном порядке → в UI остаётся результат последнего запроса.
- Ручная проверка: logout в одной вкладке → вторая вкладка реагирует (редирект/блокировка).

**Антипаттерны**
- Результат медленного запроса перетирает уже показанный быстрый.
- `fetch` без отмены при быстром переключении вкладок/страниц.
- Live-поиск без debounce; таймер с устаревшим значением из замыкания.
- Logout в одной вкладке не виден в других — пользователь работает под «мёртвой» сессией.

### 11. ★ Сетевая устойчивость (Resilience / RESIL)

**Принцип.** Сеть ненадёжна. UI обязан предсказуемо вести себя при ошибке, задержке
и оффлайне — а не «зависать» или падать молча.

**Критерии**
- **RESIL-1.** На каждый сетевой вызов — таймаут; превышение → явное состояние
  ошибки, не бесконечный спиннер.
- **RESIL-2.** Системные состояния loading / error / empty / success для каждой
  асинхронной поверхности (единый паттерн, не ad-hoc в каждом компоненте) [ARCH-6].
- **RESIL-3.** Retry с экспоненциальной задержкой + jitter — только для идемпотентных
  чтений; мутации не ретраим вслепую (риск дублей).
- **RESIL-4.** Ошибки API не «глотаются»: пользователь видит сообщение, разработчик
  — лог/трейс [OPS-1, OPS-3].
- **RESIL-5.** Optimistic UI имеет rollback при ошибке; состояние не остаётся
  рассинхронизированным с сервером [STATE-4].
- **RESIL-6.** Деградация при оффлайне/частичном отказе: кеш последней успешной
  копии, баннер оффлайна — вместо белого экрана.
- **RESIL-7.** Глобальный error boundary ловит рендер-ошибки, не роняя всё
  приложение [ARCH-6].
- **RESIL-8.** (Если PWA/SW) Жизненный цикл обновления управляем: при выходе новой
  версии пользователь уведомляется («доступна новая версия, обновить»), нет залипания
  на устаревшем закешированном бандле; app-shell и стратегия кеширования заданы
  осознанно [REL-3]. Для не-PWA — явный N/A.

**Сигналы и проверки**
- Ревью data-fetching: у каждого есть error-ветка и таймаут.
- DevTools → Network throttling/offline → нет вечного спиннера, есть осмысленное состояние.
- Тест: мутация падает → optimistic-апдейт откатывается, состояние = серверному.

**Антипаттерны**
- `.catch(() => {})` — проглоченная ошибка.
- Спиннер навсегда при упавшем запросе; белый экран при ошибке рендера.
- Optimistic-апдейт без отката; слепой retry неидемпотентной мутации.

### 12. ★ Интернационализация и локализация (I18N)

**Принцип.** Текст, даты, числа, деньги и направление письма зависят от локали.
Локале-логику выносим из домена и не хардкодим формат.

**Критерии**
- **I18N-1.** Пользовательские строки вынесены в ресурсы/каталоги (i18n); нет
  хардкода текста в шаблонах; контент готов к переводу.
- **I18N-2.** Форматирование чисел/дат/имён — через `Intl`/i18n, не ручная склейка;
  учитывает локаль пользователя.
- **I18N-3.** Время хранится/передаётся в UTC (ISO 8601), отображается в TZ
  пользователя (IANA); корректная обработка DST и граничных зон.
- **I18N-4.** Деньги — целочисленные минорные единицы или decimal с явной валютой
  (ISO 4217); никакой float-арифметики денег на клиенте.
- **I18N-5.** Плюрализация и склонения — через ICU/i18n-механизм, не конкатенация
  `count + ' товаров'`.
- **I18N-6.** Поддержка RTL/bidi там, где требуется: логические CSS-свойства
  (`margin-inline`, `inset-inline-start`), не `left/right`. UTF-8 сквозной,
  Unicode-нормализация в поиске/сравнении.

**Сигналы и проверки**
- `grep` на хардкод-строки в шаблонах вне i18n-обёрток → ревью/0.
- `grep` на ручной формат дат (`padStart`/`split('-')`) вместо `Intl`/i18n → 0.
- Тест граничных TZ/DST, не-ASCII/эмодзи, отрицательных/больших денежных сумм.

**Антипаттерны**
- Захардкоженный текст вперемешку с разметкой.
- `new Date().toLocaleString()` без явной TZ/локали; деньги во float.
- `margin-left`/`text-align: left` там, где интерфейс должен поддерживать RTL.

### 13. ★ Наблюдаемость — клиент (Observability / OPS)

**Принцип.** Прод-инцидент на клиенте невидим без телеметрии. Ошибки, метрики и
контекст собираются и коррелируются — без утечки PII.

**Критерии**
- **OPS-1.** Клиентский error-tracking (Sentry/аналог) включён; source-maps
  загружены в релизе → читаемые стектрейсы.
- **OPS-2.** Field-метрики Web Vitals (RUM) собираются с реальных пользователей, не
  только lab (Часть 0).
- **OPS-3.** Структурированные клиентские логи/breadcrumbs с корреляцией; в
  логах/телеметрии нет токенов/PII [SEC-8].
- **OPS-4.** Provenance: версия/коммит сборки доступны в клиенте — для привязки
  ошибки к релизу [REL-7].
- **OPS-5.** Аналитика и трекинг — только по согласию (consent-gated), с уважением
  Do-Not-Track/consent [SEC-8].

**Сигналы и проверки**
- Проверка релиза: source-maps загружены, стектрейсы в трекере читаемы.
- Дашборд RUM с Web Vitals по перцентилям; алерт на рост частоты JS-ошибок.
- Ревизия payload телеметрии/аналитики на отсутствие PII; проверка consent-гейта.

**Антипаттерны**
- Ошибки видны только в консоли пользователя; минифицированные нечитаемые стектрейсы.
- PII/токен в событиях аналитики; трекинг без согласия.

### 14. ★ Поставка и целостность сборки (Delivery / REL)

**Принцип.** Сборка фронта — артефакт, который должен собираться воспроизводимо,
доставляться безопасно и откатываться быстро.

**Критерии**
- **REL-1.** Автоматизированный пайплайн build → lint → test → (a11y/security-scan) →
  deploy, без ручных шагов.
- **REL-2.** Воспроизводимая сборка из закреплённого lock-файла; один артефакт
  продвигается по средам (config из env/runtime-config, без пересборки под среду).
- **REL-3.** Ассеты версионированы/хешированы (content-hash) + корректный
  cache-busting: статика immutable-кешируется, `index.html` — нет.
- **REL-4.** Progressive delivery: feature-flags/canary для рискованных изменений;
  флаг живёт с условием и датой выпиливания [ARCH-4].
- **REL-5.** Бюджет бандла — gate в CI; регрессия размера ломает сборку [PERF-2].
- **REL-6.** SRI на сторонние скрипты/стили [SEC-9]; целостность зависимостей
  контролируется (SCA, lock) [SEC-7].
- **REL-7.** Откат быстр и проверен; любое изменение трассируемо коммит → артефакт →
  среда [OPS-4].

**Сигналы и проверки**
- Пайплайн описан в VCS; ручных шагов деплоя нет.
- Проверка content-hash на ассетах и заголовков кеширования.
- Наличие флагов со сроком; периодический rollback-drill.

**Антипаттерны**
- Ручной деплой; пересборка под каждую среду.
- Вечные feature-флаги; невозможность связать прод-ошибку с конкретным релизом.

### 15. ★ Документация и DX (Documentation / DOC)

**Принцип.** Знание о компонентах и эксплуатации не должно быть «племенным».
Дешевле всего документировать на уровне design-system — один источник, массовый эффект.

**Критерии**
- **DOC-1.** Компоненты UI-кита документированы с примерами (Storybook/MDX/живой
  каталог); пропсы/слоты/события видны потребителю [UIKIT-5].
- **DOC-2.** README/onboarding позволяет поднять проект локально с нуля по
  инструкции; команды актуальны.
- **DOC-3.** Значимые архитектурные решения зафиксированы как ADR; диаграммы
  слоёв/зависимостей актуальны [ARCH-4].
- **DOC-4.** Изменения коммуницируются: changelog; политика и сроки депрекаций
  компонентов/публичного API.
- **DOC-5.** Имена и комментарии объясняют WHY (неочевидный инвариант/обход), не WHAT [MAINT-5].

**Сигналы и проверки**
- Каталог компонентов существует и обновляется в том же PR, что и компонент.
- Smoke по README с чистого клона: проект поднимается без «племенных знаний».
- Ревизия `decisions/` — значимые решения зафиксированы.

**Антипаттерны**
- Компонент без примеров использования и описания пропсов/слотов.
- README, по которому проект не поднимается; решения «в голове у автора».

### 16. ★ Адаптивность и кросс-девайс (Responsive / RWD)

**Принцип.** Один интерфейс — разные вьюпорты, плотности, способы ввода и браузеры.
Качество — это предсказуемая работа на всей согласованной матрице, а не только на
машине разработчика.

**Критерии**
- **RWD-1.** Раскладка отзывчива: fluid-сетки и согласованные брейкпоинты, контент не
  ломается и не требует горизонтального скролла в поддерживаемом диапазоне ширин
  (mobile → desktop). Подход mobile-first/прогрессивное улучшение.
- **RWD-2.** `<meta viewport>` задан корректно; zoom не заблокирован (`user-scalable`
  не отключён — это и a11y-требование) [A11Y-3].
- **RWD-3.** Интерактивные цели достаточного размера и отступа для тача (WCAG 2.2
  target-size ≥24×24, практический комфорт ≈44px); hover-only-взаимодействия имеют
  тач/клавиатурную альтернативу [A11Y-2].
- **RWD-4.** Матрица поддерживаемых браузеров/устройств зафиксирована (Часть 0);
  автопрефиксинг и трансформы под целевой baseline; используемые web-API имеют
  фолбэк/feature-detection, а не «works on my Chrome».
- **RWD-5.** Адаптация под предпочтения среды: `prefers-color-scheme`,
  `prefers-reduced-motion` [A11Y-5], плотность/безопасные зоны (`env(safe-area-*)`),
  корректное поведение при 200% zoom и reflow без потери контента.

**Сигналы и проверки**
- Ревью/прогон ключевых страниц на брейкпоинтах (DevTools device toolbar) + на реальном мобильном.
- Кросс-браузер прогон (BrowserStack/Playwright-проекты по браузерам) на матрице из Часть 0.
- `grep` на `left/right`-фиксации и абсолютные `px`-ширины контейнеров → ревью на адаптивность.
- axe/ручная проверка zoom 200% и target-size.

**Антипаттерны**
- Фиксированные ширины/высоты в `px`, ломающие узкие вьюпорты; горизонтальный скролл на мобильном.
- `maximum-scale=1`/`user-scalable=no` — блокировка зума.
- Функциональность доступна только по hover; «тестировал в одном Chrome».
- Тач-цели в упор друг к другу, мисхиты на телефоне.

### 17. ★ Навигация и URL-состояние (Routing / URL)

**Принцип.** URL — это состояние и контракт с браузером. Значимое состояние вида
должно быть в адресе: шарабельно, восстановимо, корректно к Back/Forward.

**Критерии**
- **URL-1.** Значимое состояние представления (фильтры, поиск, пагинация,
  сортировка, активная вкладка) отражается в URL (query/path) — ссылку можно
  расшарить и восстановить тем же видом; источник истины не только в локальном
  состоянии [STATE-1].
- **URL-2.** Back/Forward семантически корректны: история не засоряется (replace vs
  push осознанно), кнопка «назад» не ломает состояние и не создаёт ловушек.
- **URL-3.** Прямой заход/перезагрузка по любому deep-link отрабатывает: состояние
  гидрируется из URL, нет «белого экрана» на не-корневом маршруте.
- **URL-4.** Несуществующие/запрещённые маршруты дают осмысленный 404/403, а не
  пустой экран или редирект-петлю; неизвестные параметры деградируют безопасно.
- **URL-5.** Восстановление позиции скролла и фокуса при навигации/возврате; якорные
  ссылки и фокус-менеджмент при смене роута корректны [A11Y-2].

**Сигналы и проверки**
- Ручная проверка: применил фильтр → скопировал URL → открыл в новой вкладке → тот же вид.
- Прогон Back/Forward по основным флоу: состояние консистентно, истории нет мусора.
- Заход по deep-link с чистой загрузкой (hard refresh) на каждый ключевой экран.
- Переход на несуществующий роут → 404-страница.

**Антипаттерны**
- Фильтры/пагинация живут только в памяти — ссылка ведёт на дефолтный вид.
- `push` вместо `replace` на каждое изменение фильтра → 50 записей в истории.
- Перезагрузка на вложенном роуте → 404 от статик-сервера/белый экран.
- «Назад» оставляет рассинхронизированное состояние или зацикливает редирект.

### 18. ★ Формы и ввод (Forms / FORM)

**Принцип.** Формы — основной канал ввода данных и самый частый источник UX-боли и
потери данных. Валидация, обратная связь и защита от потерь — системные, не ad-hoc.

**Критерии**
- **FORM-1.** Валидация многоуровневая и согласованная: клиентская — для быстрой
  обратной связи, серверная — источник истины; правила не расходятся, серверные
  ошибки маппятся на поля [ARCH-7]. Ошибки доступны (`aria-invalid`/summary) [A11Y-4].
- **FORM-2.** Защита от потери данных: уход/закрытие/refresh при «грязной» форме —
  предупреждение (`beforeunload`/route-guard) или autosave-черновик; согласованный
  паттерн, не на каждой форме свой [URL-2].
- **FORM-3.** Защита от двойной отправки: на время запроса submit заблокирован/
  индицирует прогресс; повторные клики не плодят дублей; мутации идемпотентны где
  возможно [CONC-1, STATE-4].
- **FORM-4.** Нормализация и маски ввода (телефоны, суммы, даты) предсказуемы; формат
  ввода не конфликтует с локалью [I18N-2, I18N-4]; вставка/автозаполнение
  обрабатываются.
- **FORM-5.** Согласованная обратная связь по состояниям: pending/success/error,
  фокус на первой ошибке, понятный error-summary; деструктивные действия — с
  подтверждением [A11Y-5, RESIL-2].

**Сигналы и проверки**
- Тест: сабмит с серверной ошибкой валидации → ошибка показана у нужного поля.
- Ручная проверка: правка формы → уход без сохранения → предупреждение/черновик.
- Тест: быстрый двойной клик по submit → один запрос, один объект создан.
- a11y: ошибки объявляются скринридером, фокус уходит на первую невалидную.

**Антипаттерны**
- Только клиентская валидация (обходится) или только серверная (медленный фидбек).
- Уход со страницы стирает заполненную форму без предупреждения.
- Двойной клик создаёт два заказа; submit без блокировки на время запроса.
- Серверная ошибка показывается общим тостом, поле-виновник неизвестно.

### 19. ★ Discoverability / SEO

**Принцип.** Если поверхность индексируется или шарится — метаданные и доступность
для краулеров часть качества. Для непубличных приложений измерение явно N/A.

**Критерии**
- **SEO-1.** Уникальные осмысленные `<title>` и `meta description` на значимых
  маршрутах; язык страницы (`lang`) задан [I18N-1].
- **SEO-2.** Шаринг-метаданные: Open Graph/Twitter Cards (title/description/image)
  для шарабельных страниц; canonical-URL для борьбы с дублями; `hreflang` при
  мультиязычности [I18N-6].
- **SEO-3.** Контент доступен краулерам: для SPA, где важна индексация, — SSR/SSG/
  prerender или совместимость с JS-рендерингом; ключевой контент не спрятан за
  взаимодействием. Семантический HTML и осмысленные heading'и [A11Y-1].
- **SEO-4.** `robots`/`sitemap`/`meta robots` управляют индексацией осознанно
  (служебные/приватные зоны закрыты от индексации); ссылки краулабельны (реальные
  `<a href>`, не `@click`-навигация).
- **SEO-5.** Структурированные данные (schema.org/JSON-LD) для контента, которому это
  даёт rich-результаты (товары, статьи, организация), — где применимо.

**Сигналы и проверки**
- Lighthouse SEO-аудит на публичных страницах; проверка OG через шеринг-дебаггер.
- View-source/«as Googlebot»: ключевой контент присутствует в ответе (для SSR/prerender).
- Проверка `robots.txt`/`sitemap.xml` и `meta robots` на приватных зонах.

**Антипаттерны**
- Один `<title>` на всё приложение; пустые/дублирующиеся meta.
- Публичный шаринг даёт пустую карточку (нет OG); важный контент невидим без JS у краулера.
- Навигация только через JS-обработчики без реальных ссылок; приватные зоны в индексе.

## Часть II. Критерии по слоям

Зеркало сквозных измерений в разрезе слоёв фронта. Слой-специфичные критерии
ссылаются на сквозные ID, не дублируя формулировки.

### A. Маршруты / страницы (ROUTE) — тонкий слой, ленивость, состояния

| # | Критерий |
|---|----------|
| ROUTE-1 | Страница тонкая: оркеструет загрузку и композицию, бизнес-логика в фичах/сторах [MOD-1], не в компоненте страницы |
| ROUTE-2 | Роуты code-split (ленивые) [PERF-1]; тяжёлые роуты не в main-бандле |
| ROUTE-3 | Guard'ы (auth/роли) централизованы [SEC-2], не дублируются по страницам |
| ROUTE-4 | Каждая страница имеет loading/error/empty состояния [RESIL-2]; запросы отменяются при уходе [CONC-2] |
| ROUTE-5 | Data-fetching на навигации без гонок: показан результат текущего роута [CONC-1] |
| ROUTE-6 | Состояние вида (фильтры/пагинация/таб) — в URL, deep-link/refresh восстановим [URL-1, URL-3]; несуществующий роут → 404 [URL-4] |
| ROUTE-7 | Уход со «грязной» страницы защищён guard'ом от потери данных [FORM-2]; восстановление скролла/фокуса [URL-5] |

### B. Фичи / модули (FEAT) — самодостаточность, публичный API

| # | Критерий |
|---|----------|
| FEAT-1 | Модуль самодостаточен: компоненты + локальные стор/composables/типы; публичный вход через index/barrel [MOD-4] |
| FEAT-2 | Общение между фичами через общий слой/события/публичный API, не глубокие импорты [MOD-1] |
| FEAT-3 | Доменная логика фичи тестируема без DOM (вынесена в composables/стор) [TEST-2] |
| FEAT-4 | Новая фича добавляется как папка, без правок чужих модулей [EXT-2] |
| FEAT-5 | Формы фичи следуют единым паттернам валидации/защиты от потерь/двойного сабмита [FORM-1, FORM-2, FORM-3] |

### C. UI-кит / design-system (UIKIT) — изоляция вендора, расширяемость

| # | Критерий |
|---|----------|
| UIKIT-1 | UI-кит не зависит от бизнес-фич [MOD-2]; доменная логика в него не утекает |
| UIKIT-2 | Вендорный UI-фреймворк изолирован за обёртками [ARCH-2]; потребитель видит наш API |
| UIKIT-3 | Компоненты расширяются пропами + именованными слотами [EXT-3], не форками |
| UIKIT-4 | a11y чинится на уровне UI-кита [A11Y-1] — один источник, массовый эффект |
| UIKIT-5 | Канонический примитив единствен (секция/таблица/диалог/поле) [REUSE-1]; документирован [DOC-1] |
| UIKIT-6 | Компоненты адаптивны и поддерживают темы по умолчанию [RWD-1, A11Y-3]; покрыты визуальной регрессией [TEST-6] |

### D. Слой данных / состояния (STATE) — единый источник, разделение кэша и UI

| # | Критерий |
|---|----------|
| STATE-1 | Единый источник истины (store); компоненты не дублируют состояние [ARCH-3] |
| STATE-2 | Серверный кэш (data-fetching) и UI-state разделены, не путаются [ARCH-3] |
| STATE-3 | Ответы API валидируются на границе (схема), форме не доверяем слепо [ARCH-7] |
| STATE-4 | Мутации идемпотентны где возможно; optimistic-апдейты с rollback [RESIL-5] |
| STATE-5 | Подписки/watch'и очищаются [PERF-7]; нет гонок при конкурентных апдейтах [CONC-5] |
| STATE-6 | Персистнутое состояние (localStorage/IndexedDB/persist) версионируется и мигрируется при смене схемы; quota-exceeded обрабатывается [PERF-7]; синк между вкладками осознан [CONC-6] |

## Definition of Done

- Критерий считается выполненным **только при зелёной проверке/сигнале**; «частично»
  не засчитывается. Уровень зрелости **3** означает, что нарушение ловит машина
  (lint-правило/CI-gate/тест), а не ревьюер.
- Для каждого измерения проект фиксирует **baseline** (зрелость 0–4) и конкретный
  порог по сигналам — в overlay `frontend-quality-criteria.<project>.md`, не в этом
  шаблоне.
- Применимость ★-измерений и слоёв фиксируется явно: **«применимо / N/A +
  обоснование»**. Примеры N/A: I18N — для одноязычного внутреннего инструмента;
  OPS-2 (RUM) — если прод-телеметрия отсутствует по согласованному решению; RESIL-6
  (оффлайн) и RESIL-8 (PWA-обновление) — для приложения без SW/оффлайна; **SEO** —
  для непубличного приложения за авторизацией; **RWD** — частично, если поддержка
  только desktop-киоска с фиксированным разрешением (фиксируется как baseline).
- **Релиз-блокеры** (минимум зелёного перед 100% трафика): Часть 0 (бюджеты Web
  Vitals + браузерная матрица), SEC-1, SEC-6, SEC-10, RESIL-1, RESIL-2, RESIL-4,
  A11Y критичные (0 axe-нарушений), FORM-2 (нет потери данных) и FORM-3 (нет двойной
  отправки) на критичных формах, URL-3 (deep-link/refresh не белый экран), TEST
  критичных флоу.
- Цель миграции каждого измерения — **сдвиг проверок влево**: от ручного ревью к
  CI-enforcement.

## Как применять к конкретному проекту

1. Завести **overlay** `frontend-quality-criteria.<project>.md` рядом: для каждого
   измерения честно проставить зрелость (0–4), главный долг по сигналам и привязку
   ID критериев (`SEC-1`/`PERF-2`/…) к месту в коде, компонентам и спринтам. Шаблон
   остаётся чистым; проектная конкретика — в оверлее.
2. Зафиксировать **baseline** и выбрать **рычаги**: измерения, фикс которых на уровне
   инфраструктуры/общих компонентов даёт массовый эффект (обычно — тестируемость,
   a11y, производительность, изоляция вендора, сетевая устойчивость).
3. Перевести релевантные критерии в **acceptance-условия** спринта, ссылаясь по ID.
4. Поднимать enforcement: критерий зрелости **3** = нарушение ловит машина. Цель
   миграции каждого измерения — перевести проверки из «ревью» в «CI».

> Эталон не статичен: при смене стека/требований пересматриваем критерии, но
> направление неизменно — **сдвигать проверку влево**, от ручного ревью к
> автоматическому enforcement.
