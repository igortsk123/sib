# SQL — критерии качества (project-agnostic)

> **Что это.** Переносимое ядро критериев качества SQL: эталон для код-ревью,
> Definition of Done и CI quality gates. Опора — практики PostgreSQL, размерного
> моделирования (Kimball) и колоночных/time-series расширений (Timescale-подобных).
> Документ описывает **целевое состояние**; привязок к конкретным таблицам/модулям
> здесь нет — их подставляет проект отдельным overlay-маппингом.
>
> **Как пользоваться.** Каждый критерий имеет **стабильный ID** (`O3`, `K6`, `C1`);
> при планировании/ревью ссылаемся на него по ID. Структура области: *Принцип* (зачем)
> → критерии (ID + уровень + порог) → *Антипаттерны* (что считаем дефектом).
>
> **Шаблон ↔ overlay.** Этот файл — переносимый **шаблон**. Проектную привязку (имена
> витрин, фактов, модулей доступа, текущий baseline, разрывы) держим в отдельном
> **overlay** рядом — `sql-quality-criteria.<project>.md`, где каждый ID привязан к
> месту в коде и этапу плана. Шаблон остаётся чистым.
>
> **Уровни обязательности.**
> - **MUST** — нарушение блокирует ревью.
> - **SHOULD** — отклонение требует обоснования в PR.
> - **PREFER** — стилевое предпочтение / цель миграции.
>
> Критерий без зелёной проверки/порога считается невыполненным — «частично» не засчитывается.
>
> **Каноны, на которые опираемся:**
> - Реляционная теория: нормализация + осознанная денормализация, ссылочная целостность, трёхзначная логика NULL.
> - Размерное моделирование: Kimball (star schema, грейн, SCD, аддитивность мер).
> - Производительность: EXPLAIN-driven, индексирование под план, SARGability, keyset-пагинация.
> - Конкурентность: уровни изоляции SQL, MVCC, оптимистичная блокировка, `SKIP LOCKED`.
> - Безопасность: parameterized statements (anti-injection), least privilege, RLS.
> - Данные/приватность: DAMA-DMBOK, GDPR/CCPA (retention/erasure), классификация PII.
> - Время/локаль: ISO 8601, IANA TZ, ISO 4217, Unicode/UTF-8, ICU/CLDR collation.
> - Эксплуатация: `pg_stat_statements`, `auto_explain`, statement/lock timeouts, реентерабельные миграции.
>
> **Отличие от «перф-only» ревью.** К классической оптимизации запросов добавлены
> часто упускаемые классы: **корректность данных** (NULL/детерминизм), **изоляция и
> конкурентность**, **время/таймзоны**, **текст/коллация**, **JSONB-дисциплина**,
> **доступ/PII**, **тестирование/реконсиляция/миграции**, **жизненный цикл данных**.
> Неприменимость области фиксируется явно (N/A + обоснование, см. Definition of Done).

Шкала зрелости каждой области: **0** нет практики · **1** точечно · **2** есть, но без
enforcement · **3** enforced (lint/CI/тест/EXPLAIN-gate) · **4** эталон. Цель миграции
— **сдвиг проверок влево**: от ручного ревью к автоматической проверке.

Контур: общие инварианты (корень) → OLTP → OLAP/EDW → DDL/целостность →
производительность → память → изоляция/транзакции → корректность данных → время →
текст/локаль → JSONB → эксплуатация → доступ/PII → тестирование/миграции →
конкурентность → планировщик/жизненный цикл → чек-лист ревью.

## Часть 0. Общие инварианты как корень

**Принцип.** Назначение объекта определяется нагрузкой; горячий путь читает
производное, а не живую нормализованную таблицу; каждый нетривиальный запрос
обоснован планом. Эти инварианты — корень, остальные области их детализируют.

- **G1 (MUST).** Разделяй назначение объекта по нагрузке: **OLTP** обслуживает запись
  и точечное чтение (короткие транзакции, высокая конкурентность); **OLAP/EDW**
  обслуживает агрегаты (большие сканы, низкая конкурентность). Не считай тяжёлую
  аналитику на живых OLTP-таблицах горячего пути — для этого есть материализованные
  представления и таблицы фактов.
- **G2 (MUST).** Горячий read-путь читает **производный/материализованный объект**
  (предрассчитанную витрину), а не живую нормализованную таблицу, когда запрос частый
  и тяжёлый.
- **G3 (MUST).** Технические колонки-источники CDC/версионирования (sequence/`xmin`-
  аналоги, монотонные счётчики изменений) — не дропать, не переиспользовать их
  семантику, при INSERT/UPDATE не выставлять вручную.
- **G4 (SHOULD).** Один логический владелец записи на таблицу. Запись из чужого домена
  в его таблицы запрещена; чтение чужого домена — через стабильный контракт (view /
  витрина / событие), не прямым доступом к его таблицам.
- **G5 (MUST).** Каждый нетривиальный запрос проходит `EXPLAIN (ANALYZE, BUFFERS)` до
  мёржа; план прикладывается к PR для горячих путей и для всего, что трогает большие
  фактовые/витринные объекты.

**Антипаттерны**
- Тяжёлый аналитический `GROUP BY` по живой OLTP-таблице горячего пути.
- Прямой `SELECT`/`INSERT` в таблицы чужого домена в обход контракта.
- Ручная подстановка значения в CDC/версионную колонку; мёрж запроса без EXPLAIN.

## Часть I. Критерии по областям

## 1. OLTP

**Принцип.** Короткие транзакции и точечный доступ; значения всегда параметризованы;
форма запроса — контракт, защищённый от неконтролируемого расширения и инъекций.

### Параметризация и безопасность

- **O1 (MUST).** Значения — только через параметры (`$n` / именованные биндинги).
  Через `str.format()` / f-string разрешено подставлять **только фрагменты структуры**
  запроса (списки полей, готовые предикаты, имена JOIN), собранные из доверенного кода,
  **никогда** — пользовательский ввод. Любой пользовательский литерал, попавший в
  интерполяцию структуры, — баг инъекции.
- **O2 (SHOULD).** Идентификаторы (схема/таблица/колонка), если их приходится
  интерполировать, проходят через белый список или `quote_ident`-эквивалент, не через
  прямую конкатенацию.

### Форма запроса

- **O3 (MUST).** Никаких `SELECT *` на широких/горячих витринах без `LIMIT`. Явный
  список колонок — это и контракт, и защита от неконтролируемого расширения выборки.
- **O4 (SHOULD).** Явный тип JOIN и явное условие. Декартовых произведений и неявных
  запятых-джойнов — нет. `LEFT JOIN`, который реально INNER по смыслу (предикат на его
  колонку в `WHERE`), переписывать в `INNER`.
- **O5 (SHOULD).** Предикаты **SARGable**: не оборачивай индексируемую колонку в
  функцию (`date(col) = …` → `col >= … AND col < …`). Подстрочный/полнотекстовый поиск
  на горячем пути — через специализированный индекс, а не `ILIKE '%…%'` сканом.
- **O6 (MUST).** Пагинация больших списков — keyset (`WHERE (sort_key) > $last`), а не
  `OFFSET N` на глубоких страницах.
- **O7 (SHOULD).** Убирать `-- todo: temp` и закомментированные блоки из мёржащегося
  SQL. Мёртвый JOIN, который ничего не отдаёт в список полей, удалять.

### Запись

- **O8 (MUST).** Upsert идемпотентен: `INSERT … ON CONFLICT (<natural/uniq key>)
  DO UPDATE/NOTHING`. Конфликтный таргет — реальный уникальный индекс.
- **O9 (MUST).** Многошаговая запись — в одной транзакции. Частичная запись нескольких
  таблиц без транзакции запрещена.
- **O10 (SHOULD).** Массовая запись — `executemany` / `COPY` / `unnest($1::[])`, не цикл
  из одиночных `execute`. `RETURNING` вместо повторного `SELECT`.
- **O11 (MUST).** Никаких побочных эффектов в выражениях «чистого чтения»: внутри
  `SELECT` горячего пути нет `nextval`, записи, volatile-функций с эффектами.

**Антипаттерны**
- Пользовательский ввод в `f"… WHERE name = '{x}'"`; `SELECT *` на широкой витрине без `LIMIT`.
- `OFFSET 100000` на глубокой странице; цикл из одиночных `INSERT` вместо `COPY`/`unnest`.
- `nextval`/запись внутри запроса, считающегося идемпотентным чтением.

## 2. OLAP / EDW (размерное моделирование)

**Принцип.** Схема-звезда с объявленным грейном; суррогатные ключи; идемпотентный,
реентерабельный налив; аналитика не блокирует OLTP.

- **K1 (MUST).** Схема — звезда: узкие таблицы фактов (меры + суррогатные FK-ключи) и
  таблицы измерений (атрибуты). Снежинку (dim→dim) разворачивать только при доказанной
  необходимости.
- **K2 (MUST).** Суррогатные ключи (`*_key`, `generated by default as identity`) —
  единственные связи факт→измерение. Натуральные/бизнес-ключи живут в измерении как
  атрибуты, не как FK факта. Ключ даты — целочисленный `YYYYMMDD`; время суток —
  отдельным измерением.
- **K3 (MUST).** Грейн факта объявлен и неизменен: что есть одна строка — зафиксировано.
  PK выражает грейн. Не смешивать грейны в одной таблице.
- **K4 (SHOULD).** Меры аддитивны где возможно; полу-/неаддитивные (остатки, цены,
  балансы) помечать и не суммировать вслепую. Денежные суммы и количества — `numeric`
  фиксированной точности; `float/real` — только для гео/служебных величин, не для денег.
- **K5 (SHOULD).** SCD-стратегия измерения объявлена явно (Type 1 overwrite vs Type 2
  history). Для Type 2 — `valid_from/valid_to` + `is_current` / частичный уникальный
  индекс на текущую версию.
- **K6 (MUST).** Загрузка факта **идемпотентна и реентерабельна**: `ON CONFLICT
  DO NOTHING/UPDATE` по уникальному бизнес-ключу строки — переналив не дублирует. ETL
  рестартуется без ручной чистки.
- **K7 (SHOULD).** Партиционирование/колоночное хранение: партиционный ключ — колонка
  даты; сегментирование/кластеризация — по часто-фильтруемым низкокардинальным
  колонкам; порядок — по убыванию даты. Холодные партиции → дешёвый tablespace +
  columnstore/сжатие.
- **K8 (SHOULD).** EDW не читает горячий OLTP напрямую в продовом ETL — берёт из
  стейджа/снапшота/CDC, чтобы аналитический скан не блокировал транзакции.
- **K9 (MUST).** Полное обновление материализованного представления — конкурентно
  (`REFRESH MATERIALIZED VIEW CONCURRENTLY`, требует уникального индекса на MV), чтобы
  не лочить читателей горячего пути.

**Антипаттерны**
- Натуральный бизнес-ключ как FK факта; смешанные грейны в одной фактовой таблице.
- `float` для денежных мер; `REFRESH MATERIALIZED VIEW` без `CONCURRENTLY` лочит читателей.
- ETL, требующий ручной чистки перед повторным наливом; аналитика читает горячий OLTP напрямую.

## 3. DDL, непротиворечивость структур, ссылочная целостность

**Принцип.** Целостность обеспечивается на уровне БД внутри домена; межсервисных FK
нет; любой DDL мигрируется обратимо и без долгих эксклюзивных локов.

- **D1 (MUST).** Каждая OLTP-таблица имеет PK. Естественная уникальность — `UNIQUE`-
  ограничением/индексом, а не «договорённостью».
- **D2 (MUST, внутри домена).** FK на ссылочные данные внутри **одной схемы/домена** —
  обязательны. `ON DELETE` объявлен осознанно (`RESTRICT`/`CASCADE`/`SET NULL`).
- **D3 (MUST, межсервисно).** Межсхемные/межсервисные FK — **запрещены**. Связь по
  бизнес-ключам, целостность — приложением + идемпотентностью событий. Финансовые/
  исторические данные — immutable-снапшот в своей схеме.
- **D4 (SHOULD).** Типы консистентны между связанными колонками (FK-пара —
  одинаковый тип). Деньги — `numeric`, не `float`; идентификаторы — единый размер
  (`bigint` для растущих, узкий тип для справочников-измерений).
- **D5 (SHOULD).** `NOT NULL` + `DEFAULT` по умолчанию для обязательных колонок;
  nullable — осознанное решение. Булевы флаги — `boolean NOT NULL DEFAULT`.
- **D6 (MUST).** Инварианты данных — `CHECK`-ограничениями (неотрицательные суммы,
  диапазоны статусов, валидность периода `valid_from < valid_to`), а не только в коде.
- **D7 (SHOULD).** Имена единообразны: префиксы по классу (`fact_*`/`dim_*`/`mv_*`/`v_*`),
  суффикс `_key` для суррогатов, `_id` для бизнес-идентификаторов.
- **D8 (MUST).** Любой DDL обратимо мигрируется: миграция = up + down. Без `DROP` ещё
  используемых/переходных объектов до завершения миграции потребителей.
- **D9 (SHOULD).** Изменение «горячей» колонки/индекса — `CREATE INDEX CONCURRENTLY`;
  `ADD COLUMN` с volatile default — в два шага (nullable → backfill → set default/not
  null), чтобы не брать долгий `ACCESS EXCLUSIVE`.

**Антипаттерны**
- Таблица без PK; уникальность «по договорённости» без `UNIQUE`.
- FK между схемами/сервисами; nullable-колонка по лени вместо `NOT NULL DEFAULT`.
- Инвариант проверяется только в коде; `ALTER TABLE` берёт долгий `ACCESS EXCLUSIVE` на горячей таблице.

## 4. Производительность

**Принцип.** Индекс под реальный план; тяжёлое — материализовать; OLAP-сканы опираются
на партиционный pruning. Подтверждать планом, а не догадками.

- **P1 (MUST).** Под каждый горячий предикат/JOIN/сорт — индекс, покрывающий ведущие
  колонки в порядке селективности; составной индекс упорядочен под реальный
  `WHERE … ORDER BY`. Подтверждать использование по `idx_scan`.
- **P2 (SHOULD).** Гео-проверка точки в полигоне — всегда пространственный индекс (GiST)
  + `ST_Contains` + флаг активности + дата. Без пространственного индекса — запрещено.
- **P3 (MUST).** Никаких индексов «на всякий случай»: периодически чистить
  unused/overlapped/low-usage. Каждый лишний индекс — налог на запись и память.
- **P4 (SHOULD).** Тяжёлые агрегаты не вычислять на каждый запрос — материализовать
  (MV / continuous aggregate) и читать готовое.
- **P5 (SHOULD).** Избегать N+1: один запрос с JOIN/`unnest`/`= ANY($1)` вместо цикла.
- **P6 (MUST).** Для OLAP-сканов опираться на партиционный pruning (фильтр по
  партиционному ключу ведущим предикатом). Скан большой фактовой таблицы без фильтра по
  партиционному ключу — недопустим в проде.
- **P7 (SHOULD).** Регулярный `ANALYZE`; для кластеризуемых OLAP-таблиц — `CLUSTER …
  USING <idx>` по основному порядку чтения.

**Антипаттерны**
- Индекс «на всякий случай», не подтверждённый `idx_scan`; seq scan по большой таблице на горячем пути.
- Тяжёлый агрегат на каждый запрос вместо MV; скан факт-таблицы без партиционного фильтра.
- Цикл запросов (N+1) вместо одного `= ANY($1)`/JOIN.

## 5. Потребление памяти

**Принцип.** Bounded everything: большие выборки стримятся, лишние колонки не тянутся,
сортировки/хеши не уходят в disk-spill.

- **M1 (MUST).** Большие выборки не материализуются целиком в приложении — стриминг
  курсором / серверный курсор. Любой `fetch()` без `LIMIT` на потенциально большом
  наборе — баг.
- **M2 (SHOULD).** Не тянуть лишние колонки (особенно `jsonb`/`text`/большие поля) в
  горячих списках — это раздувает `work_mem` и сеть. Явный список полей (см. O3).
- **M3 (SHOULD).** Сорт/хеш больших наборов — на индексе (избегать disk-spill); следить
  за `work_mem` vs реальным объёмом (`Sort Method: external merge` = сигнал).
- **M4 (SHOULD).** Bloat под контролем: периодический `VACUUM (FULL)` / `REINDEX
  CONCURRENTLY` по диагностике; не допускать роста dead-tuples на горячих таблицах.
- **M5 (PREFER).** Холодные данные — на дешёвом носителе + columnstore-сжатие; горячие
  — на быстром носителе.

**Антипаттерны**
- `fetchall()` по таблице без `LIMIT`/курсора; `SELECT *` тянет `jsonb`/`text` в список.
- Сортировка большого набора с `external merge` на диск; бесконтрольный рост bloat.

## 6. Изоляция, транзакции, побочные эффекты

**Принцип.** Транзакция коротка; уровень изоляции осознан под гонку read-modify-write;
никаких внешних вызовов внутри открытой транзакции.

- **I1 (MUST).** Транзакция настолько коротка, насколько возможно: без сетевых вызовов/
  внешних API внутри открытой транзакции (она держит локи).
- **I2 (MUST).** Уровень изоляции осознан. Финансовые/инвентарные инкременты —
  `SERIALIZABLE`/`REPEATABLE READ` с ретраем на serialization failure, либо явный
  `SELECT … FOR UPDATE`. Не полагаться на дефолтный `READ COMMITTED` там, где есть
  гонка read-modify-write.
- **I3 (MUST).** Порядок взятия локов единообразен между путями записи (профилактика
  deadlock). Эскалацию и взаимные локи мониторить.
- **I4 (SHOULD).** Идемпотентность записи через `event_id` + таблицу обработанных
  событий, чтобы повтор сообщения/ретрай не задваивал эффект.
- **I5 (MUST, межсервисно).** Никаких распределённых транзакций между сервисами.
  Согласованность — saga/outbox + идемпотентные consumer'ы; чужие справочники — lazy
  materialization (pull-and-cache → события).
- **I6 (MUST).** Чистота чтения (см. O11) + детерминизм: функции, помеченные
  `IMMUTABLE`/`STABLE`, реально таковы. Всё, что зависит от `now()`/`current_date`, —
  максимум `STABLE`, не `IMMUTABLE`.

**Антипаттерны**
- HTTP/внешний вызов внутри открытой транзакции — лок висит на сетевом таймауте.
- Read-modify-write на `READ COMMITTED` без `FOR UPDATE` → lost update.
- Распределённая транзакция через два сервиса; `IMMUTABLE`-функция, зависящая от времени.

## 7. Корректность данных (NULL, трёхзначная логика, детерминизм)

**Принцип.** Самый недопредставленный класс: баги, которые перф-ревью не ловит, а
данные молча портятся. NULL-логика и тотальный детерминизм — обязательны.

- **C1 (MUST).** `NOT IN (<подзапрос>)` запрещён, если подзапрос может вернуть NULL
  (вся выборка молча схлопывается в пусто). Использовать `NOT EXISTS` или `<> ALL` с
  явным `WHERE col IS NOT NULL`.
- **C2 (MUST).** Осознанный `COUNT(col)` vs `COUNT(*)` (первый игнорит NULL); агрегаты
  (`SUM/AVG`) по nullable-колонке — понимать, что NULL выпадает. Для денег —
  `coalesce(…,0)` там, где нужен 0.
- **C3 (MUST).** NULL в ключах JOIN не матчится сам с собой. Если ключ nullable и NULL
  должен «склеиваться» — это явный `IS NOT DISTINCT FROM`, а не `=`.
- **C4 (MUST).** Деление защищено: `/ nullif(denom, 0)`. Бизнес-расчёты не падают и не
  дают `Infinity`/деления на ноль.
- **C5 (MUST).** **Тотальный детерминированный `ORDER BY`** для любого `LIMIT` / keyset:
  последний ключ сортировки — уникальный (PK/уникальный бизнес-ключ). `LIMIT` без
  `ORDER BY` запрещён.
- **C6 (SHOULD).** `DISTINCT ON (…)` всегда с согласованным `ORDER BY` по тем же
  ведущим колонкам — иначе «какая строка победит» недетерминировано.

**Антипаттерны**
- `NOT IN (SELECT …)` по nullable-колонке → пустой результат «на ровном месте».
- `SUM` по nullable считают как 0; `=` по nullable ключу теряет строки.
- `LIMIT` без тотального `ORDER BY`; деление без `nullif` на ноль.

## 8. Время и таймзоны

**Принцип.** Моменты — `timestamptz` в UTC; конвертация в локаль на границе
представления; никаких «магических» сдвигов часов.

- **T1 (MUST).** Моменты времени — `timestamptz`, не `timestamp`. Хранение в UTC,
  конвертация в локаль — на границе представления. Никаких хардкод-сдвигов на
  «магическое» число часов.
- **T2 (SHOULD).** Границы суток/периодов — через `date_trunc`/интервалы в явной
  таймзоне (`AT TIME ZONE`), не сдвигом на константу часов.
- **T3 (SHOULD).** `now()`/`current_date` — понимать волатильность (см. I6): они
  `STABLE`, привязаны к старту транзакции; не использовать как «прямо сейчас» в долгой
  транзакции, где это важно.
- **T4 (MUST).** Ключ даты (`YYYYMMDD`) вычисляется из одной канонической бизнес-
  таймзоны; смешение TZ при генерации ключа = расхождение фактов EDW.

**Антипаттерны**
- `timestamp` (без tz) для момента времени; `now() + interval '3 hours'` вместо `AT TIME ZONE`.
- Ключ даты факта посчитан из разных таймзон → расхождение агрегатов.

## 9. Текст, коллация, локаль

**Принцип.** Поиск на горячем пути индексируем; сравнение регистро-/диакритико-
независимое и locale-aware, а не «на надежде».

- **L1 (MUST).** `ILIKE '%…%'` / `LIKE '%…%'` на горячем пути запрещён без
  поддерживающего индекса (`pg_trgm` GIN/GiST). Полнотекст — через предрассчитанные
  `tsvector`-витрины и `@@`.
- **L2 (SHOULD).** Регистро-/диакритико-независимое сравнение — через `citext` или
  функциональный индекс на `lower(col)` (и предикат тоже `lower`).
- **L3 (SHOULD).** Сортировка пользовательского текста — с явной locale-aware collation,
  а не байтовой `C`-сортировкой, если порядок виден пользователю.
- **L4 (PREFER).** `LIKE 'prefix%'` индексируется только с `text_pattern_ops` (или при
  collation `C`) — для префиксного поиска объявлять подходящий индекс.

**Антипаттерны**
- `ILIKE '%foo%'` по большой таблице последовательным сканом без `pg_trgm`.
- Сравнение в надежде на совпадение регистра; байтовая сортировка видимого текста.

## 10. JSONB-дисциплина

**Принцип.** JSONB — для слабоструктурированных атрибутов, не замена колонкам;
фильтруемое/сортируемое выносится в типизированные колонки.

- **J1 (MUST).** JSONB — для слабоструктурированных/редко-фильтруемых атрибутов, не как
  замена колонкам. То, что **фильтруется, сортируется или джойнится** — выносить в
  типизированные колонки (в т.ч. в колонки витрины).
- **J2 (MUST).** Запрос по jsonb на горячем пути — с GIN-индексом (`jsonb_path_ops`) или
  выражённым индексом на конкретный ключ; не последовательный скан с распаковкой.
- **J3 (SHOULD).** Инварианты внутри jsonb (обязательные ключи, типы) — валидировать
  `CHECK`/на входе, иначе schema-on-read маскирует битые данные.

**Антипаттерны**
- JSONB вместо нормальных колонок для часто-фильтруемых полей.
- Фильтр/сортировка по ключу jsonb без GIN/выражённого индекса; нет валидации структуры.

## 11. Эксплуатация: таймауты, наблюдаемость, governance

**Принцип.** Runaway-запрос ограничен по времени заранее; каждый запрос трассируем;
деградации ловятся по статистике, а не по жалобам.

- **X1 (MUST).** Горячие пулы и фоновые сессии работают с заданными `statement_timeout`,
  `lock_timeout`, `idle_in_transaction_session_timeout`. Runaway-запрос ограничен по
  времени, а не лечится вручную `pg_terminate_backend` постфактум.
- **X2 (MUST).** Каждый запрос трассируем: осмысленное имя (не `unknown`) и/или тег-
  комментарий `/* domain.query_name */`, чтобы `pg_stat_statements` и логи связывались
  с местом в коде.
- **X3 (SHOULD).** Включён `pg_stat_statements`; деградации ловятся по нему +
  `auto_explain`, а не реактивно по жалобам.
- **X4 (SHOULD).** Лимиты ресурсов на запрос (`work_mem` под конкретный тяжёлый отчёт
  через `SET LOCAL`), а не глобально раздутый `work_mem`.

**Антипаттерны**
- Запрос без `statement_timeout` висит до ручного `pg_terminate_backend`.
- Запрос с именем `unknown` — не связать с местом в коде; глобально раздутый `work_mem`.

## 12. Доступ, привилегии, PII

**Принцип.** Least privilege: рантайм-роль имеет права только на свою схему нужного
класса; PII минимизируется и не утекает в логи/нерабочие среды.

- **S1 (MUST).** Least privilege: роль приложения/сервиса имеет права только на свою
  схему и только нужного класса (нет суперюзера для рантайма). Запись/DDL и чтение —
  разделены по ролям.
- **S2 (SHOULD).** Чувствительные данные (карты, фискальные, персональные) — минимизация
  в выборках, маскирование в нерабочих средах, отдельная схема/права; при необходимости
  — RLS на мульти-арендных таблицах (`tenant_id`).
- **S3 (MUST).** Секреты/токены не хранятся открытым текстом в таблицах и не попадают в
  SQL-литералы/логи запросов.
- **S4 (SHOULD).** Опасные операции (`pg_terminate_backend`, `truncate`,
  `set tablespace`, массовый `copy`) — только под отдельной операторской ролью и с
  аудитом, не из рантайм-роли.

**Антипаттерны**
- Суперюзер/owner-роль в рантайме; PII в выборках/логах без необходимости.
- Секрет открытым текстом в таблице; `TRUNCATE`/`pg_terminate_backend` из рантайм-роли.

## 13. Тестирование, реконсиляция, миграции, документация

**Принцип.** Данные сверяются с источником; миграции реентерабельны и прогоняются в CI;
схема самодокументируется.

- **Q1 (MUST).** ETL/EDW **реконсилируется**: контрольные суммы факта сверяются с
  источником (`sum(amount) фактов == sum источника` за период), проверяются осиротевшие
  суррогатные ключи (факт без измерения). Расхождение — алерт.
- **Q2 (SHOULD).** Ключевые запросы и витрины покрыты регрессионными тестами (вход →
  ожидаемая форма/строки); миграции прогоняются в CI на копии схемы.
- **Q3 (SHOULD).** Data-quality assertions в пайплайне: row-count в коридоре, отсутствие
  дублей по бизнес-ключу, NOT NULL по обязательным — как явный шаг, а не «заметим в
  проде».
- **R1 (MUST).** Миграции и обслуживающие скрипты **реентерабельны**: `CREATE … IF NOT
  EXISTS`, `ON CONFLICT DO NOTHING`, повторный прогон без эффекта. Ad-hoc сниппеты — не
  образец для продовых миграций.
- **R2 (SHOULD).** Индексы и ограничения именуются явно (не анонимный `create index on
  t (…)`) — иначе их нельзя адресно дропнуть/пересоздать в миграции.
- **R3 (SHOULD).** Схема самодокументируется: `COMMENT ON TABLE/COLUMN` для таблиц-
  контрактов и измерений `*_key`.

**Антипаттерны**
- ETL без сверки сумм/осиротевших ключей — расхождение находят пользователи.
- Ad-hoc сниппет как продовая миграция; повторный прогон ломается (не реентерабелен).
- Анонимные индексы/констрейнты, которые не адресовать в миграции.

## 14. Конкурентность: очереди и потерянные обновления

**Принцип.** Дополняет §6 паттернами: очереди без гонок, защита денег/остатков от lost
update, координация фоновых операций.

- **W1 (MUST).** Очередь задач/воркеры — `SELECT … FOR UPDATE SKIP LOCKED LIMIT n`, а не
  выборка + флаг под гонкой.
- **W2 (SHOULD).** Координация единичных фоновых операций — advisory locks
  (`pg_try_advisory_lock`), чтобы два инстанса не делали одну работу.
- **W3 (MUST).** Read-modify-write по деньгам/остаткам защищён от lost update: либо
  `FOR UPDATE`, либо оптимистичная блокировка через version-колонку с проверкой в
  `WHERE` и реакцией на 0 затронутых строк.
- **W4 (SHOULD).** `FOR NO KEY UPDATE` вместо `FOR UPDATE`, когда ключ не меняется —
  меньше конфликтов с FK-проверками.

**Антипаттерны**
- Выборка задач + UPDATE флага под гонкой вместо `FOR UPDATE SKIP LOCKED`.
- Два инстанса делают одну фоновую работу без advisory lock; RMW по балансу без защиты от lost update.

## 15. Планировщик, партиции, жизненный цикл данных

**Принцип.** Планировщику дают корректную статистику; число партиций под контролем;
жизненный цикл данных (retention/архив/GDPR) объявлен.

- **N1 (SHOULD).** Для коррелированных колонок — `CREATE STATISTICS` (зависимости/
  n-distinct), иначе планировщик недооценивает кардинальность.
- **N2 (SHOULD).** Перекошенным колонкам — повышенный stats target / регулярный
  `ANALYZE`; проверять, что pruning партиций реально срабатывает.
- **N3 (SHOULD).** Партиционная гигиена: контролировать число партиций, иметь `default`
  партицию-ловушку и политику её разбора.
- **N4 (MUST).** Жизненный цикл данных объявлен: drop/архивация старых партиций,
  retention-политика, механизм GDPR-удаления персональных данных.
- **N5 (PREFER).** Тяжёлые CTE — осознанный `WITH … AS MATERIALIZED / NOT MATERIALIZED`
  (PG12+ сменил дефолт на инлайн): фиксировать optimization fence где нужен.

**Антипаттерны**
- Коррелированные колонки без `CREATE STATISTICS` → неверный план.
- Взрыв числа партиций без `default` и политики разбора; нет retention/GDPR-удаления PII.

## 16. Чек-лист ревью SQL (быстрый проход)

```
[ ] EXPLAIN (ANALYZE, BUFFERS) приложен для горячего пути / больших фактов/витрин [G5]
[ ] Значения только через $n; format() — лишь над доверенными фрагментами [O1]
[ ] Нет SELECT * на широких/горячих витринах без LIMIT; поля явные [O3]
[ ] JOIN-ы явные, без мёртвых/temp-блоков; LEFT→INNER где по смыслу INNER [O4, O7]
[ ] Предикаты SARGable; фильтр по партиционному ключу ведущий [O5, P6]
[ ] Пагинация keyset, не глубокий OFFSET [O6]
[ ] Запись: ON CONFLICT по реальному уникальному ключу; всё в одной транзакции [O8, O9]
[ ] Массовая запись через executemany/COPY/unnest; RETURNING вместо ре-SELECT [O10]
[ ] Большие выборки стримятся курсором, не тянутся целиком [M1]
[ ] PK/UNIQUE/NOT NULL/CHECK/FK расставлены; межсхемных FK нет [D1-D6]
[ ] EDW: грейн в PK, суррогатные ключи, идемпотентный налив, numeric для денег [K2, K3, K6]
[ ] DDL: up+down, CONCURRENTLY для индексов, без DROP используемых объектов [D8, D9]
[ ] Изоляция под гонку read-modify-write; короткая транзакция; нет API внутри tx [I1, I2]
[ ] Нет побочных эффектов в «чистом» чтении; метки IMMUTABLE/STABLE корректны [O11, I6]
[ ] Новые/лишние индексы оправданы (проверено по статистике индексов) [P1, P3]
[ ] NULL-логика: нет NOT IN с nullable; COUNT/SUM по nullable осознаны; деление через nullif [C1-C4]
[ ] ORDER BY тотальный (тай-брейк по уникальному ключу); LIMIT без ORDER BY нет [C5]
[ ] Время: timestamptz/UTC, без хардкод-сдвигов часов; ключ даты из одной TZ [T1, T4]
[ ] Текст: ILIKE '%..%' только с pg_trgm; регистр через lower-индекс/citext [L1, L2]
[ ] JSONB: фильтр/сорт по jsonb на горячем пути вынесен в колонки или GIN-индекс [J1, J2]
[ ] Таймауты заданы (statement/lock/idle); запрос трассируем (имя != unknown) [X1, X2]
[ ] Привилегии least-privilege; PII/секреты не в открытом виде; опасные операции под оператором [S1, S3, S4]
[ ] EDW реконсилируется (суммы факта vs источник, нет осиротевших ключей) [Q1]
[ ] Миграция реентерабельна (IF NOT EXISTS); индексы/констрейнты именованы; COMMENT ON [R1-R3]
[ ] Очереди — FOR UPDATE SKIP LOCKED; деньги/остатки защищены от lost update [W1, W3]
[ ] Коррелированные колонки — CREATE STATISTICS; pruning партиций подтверждён EXPLAIN [N1, N2]
[ ] Жизненный цикл данных объявлен (retention/архив/GDPR); число партиций под контролем [N3, N4]
```

## Definition of Done

- Критерий считается выполненным **только при зелёной проверке/пороге**; «частично» не
  засчитывается. Зрелость **3** означает, что нарушение ловит машина (lint/CI/EXPLAIN-
  gate/тест), а не ревьюер.
- Для каждой области проект фиксирует **baseline** (0–4) и конкретные пороги — в overlay
  `sql-quality-criteria.<project>.md`, не в этом шаблоне.
- Применимость областей фиксируется явно: **«применимо / N/A + обоснование»**. Примеры
  N/A: §2 OLAP/EDW — если в проекте нет аналитического хранилища; §9 коллация — для
  служебных ASCII-данных без пользовательского текста.
- **Ревью-блокеры** (все MUST релевантных областей), как минимум: G1/G2/G5, O1/O3/O6/O8/
  O9/O11, D1/D3/D6/D8, P1/P6, M1, I1/I2/I5, C1–C5, T1/T4, S1/S3, R1, W1/W3, N4.
- Цель миграции каждой области — **сдвиг проверок влево**: от ручного ревью к
  автоматической проверке (lint/CI/EXPLAIN-gate).

## Как применять к конкретному проекту

1. Завести **overlay** `sql-quality-criteria.<project>.md` рядом: для каждой области
   честно проставить зрелость (0–4), главный долг и привязку ID (`O3`/`K6`/`C1`/…) к
   конкретным витринам/фактам/модулям доступа и таблице разрывов `current → target`.
2. Зафиксировать **baseline** и выбрать **рычаги**: области, фикс которых даёт массовый
   эффект (обычно — корректность данных, индексация под план, изоляция, реконсиляция).
3. Перевести релевантные критерии в **acceptance-условия**, ссылаясь по ID.
4. Поднимать enforcement: зрелость **3** = нарушение ловит машина (lint на `SELECT *`/
   `NOT IN`, EXPLAIN-gate на план, тест миграций в CI).
