# Правила code-review

> Роль ревьюера: **не пиши код, находи проблемы**. Иди по приоритету сверху вниз — первые пункты
> важнее. Адаптируй список под домен проекта (добавь домен-специфичные проверки в начало).

## Приоритеты проверки
1. **Домен-критичное** (если есть) — тон/безопасность/комплаенс. _(заведи под проект)_
2. **Соответствие плану** — реализовано всё из `plans/<slug>.md`? Нет лишних изменений вне scope?
3. **Контракты/типы** — DTO совпадают с источником формы данных? Нет `any`/неявных типов?
4. **Стандарты кода** — дубли >10 строк, размер файла, запрещённые паттерны, отладочный вывод.
5. **Состояния** (UI) — покрыты `loading`/`error+retry`/`empty`/`success`?
6. **Изоляция** — не задеты файлы чужой фичи? Импорт вместо копирования?
7. **Память** — если менялась архитектура/контракты, обновлены ли memory-доки?

## Формат замечания
```
[файл:строка] Проблема: <что не так и почему это важно>
Рекомендация: <минимальное исправление>
```

## Не проверять (вне scope ревью)
- Бизнес-решения (это к продукту, не к ревью).
- Микро-оптимизации производительности без замеров.
- Вкусовщину дизайна, если не нарушены токены/гайд.

> Глубже (security/perf/SQL/resilience с измеримыми порогами) — `_optional/quality-standards/*`
> и подход к внедрению `_optional/memory/quality-criteria.md`, если подключены в проект.
