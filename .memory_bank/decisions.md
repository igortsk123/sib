# Decisions — ADR-лог

Хронологический лог архитектурных решений. Новое решение, отменяющее старое, не удаляет старую запись,
а добавляется ниже со ссылкой «отменяет [дата]». Формат: **Решение / Почему / Альтернативы / Влияет на.**

---

## [2026-06-21] D1. Стек — Next.js + TypeScript (full-stack), Drizzle/Postgres, Inngest
**Решение:** один TypeScript-моно-стек: Next.js (App Router) + TS (строго, без `any`) + Tailwind/shadcn
для UI; PostgreSQL + Drizzle ORM; **Inngest** для durable-задач (IMAP-поллинг, обработка вложений, OCR,
LLM) и ретраев внешних вызовов. LLM — OpenAI через RU-прокси (креды доступны из инфры владельца).
**Почему:** владелец задал «TypeScript Next.js полностью, чтобы ловить» (ошибки на компиляции); тот же
владелец/инфра, что и sup2 → переиспользуем паттерн Яндекс-почты XOAUTH2; Inngest закрывает принцип
«внешний вызов всегда в обёртке с ретраем» и выносит тяжёлую обработку из request-пути.
**Альтернативы:** отдельный Python-бэкенд для OCR/LLM — отвергнут (два стека усложняют самопроверку и
единый гейт; тяжёлые обработчики при необходимости вынесем позже как сервис — это **гипотеза**, проверим
на реальных объёмах). App Router vs Pages — App Router (актуальный, server actions).
**Влияет на:** всю структуру, `core/architecture.md`, `.claude/rules/code-standards.md`, `quality-criteria.md`.

## [2026-06-21] D2. Почта — входящая через Яндекс IMAP XOAUTH2; доступ подтверждён эмпирически
**Решение:** письма забираем по `imap.yandex.ru:993` (SSL, XOAUTH2) тем же OAuth-приложением и
refresh-токеном, что и sup2; ящик `palmarius.ru@yandex.ru` (тестовый). Отправка (если понадобится) —
`smtp.yandex.ru:465` тем же токеном.
**Почему:** бриф §3/§14 — продукт **забирает входящие** письма; владелец предполагал, что доступ уже есть.
**Проверка (гипотеза→факт, принцип «проверь»):** 2026-06-21 XOAUTH2-логин на IMAP = `OK`, `LIST` папок
работает — **хотя** в sup2 scope задокументирован как `mail:smtp`. Значит токен фактически покрывает IMAP,
браузерный консент НЕ требуется.
**Альтернативы:** перевыпуск токена со scope `mail:imap` (не понадобился); app-password (XOAUTH2 лучше —
не светит пароль ящика). На рабочем этапе — 4 ящика клиники (могут быть не-Яндекс) → `core/human-decisions.md`.
**Влияет на:** `core/email-ingestion.md`, `_secrets/ACCESS.md`, `.env.*`, `sync-with-external.md`.

## [2026-06-21] D3. Инженерная конституция — самопроверяемость
**Решение:** так как владелец **не ревьюит код построчно**, работа должна быть самопроверяемой:
вертикальные срезы UI→БД с unit (чистая логика) + e2e (критический поток); полный гейт
(typecheck+lint+unit+integration+e2e+build) перед словом «готово»; внешние вызовы — в обёртке с
Inngest-ретраями и заданным user-facing состоянием ошибки; спрашивать владельца только по
`core/human-decisions.md`; любое отклонение от спеки — ADR.
**Почему:** без построчного ревью качество держится на автоматических проверках и явных решениях.
**Альтернативы:** доверять ручной проверке — отвергнуто (владелец её не делает).
**Влияет на:** `.claude/rules/engineering-principles.md`, `quality-criteria.md`, весь процесс разработки.

## [2026-06-21] D4. ПДн и медицинская тайна — повышенная безопасность
**Решение:** проектируем как продукт повышенной чувствительности: хэш паролей пользователей,
шифрование токенов почтовых ящиков, RBAC по ролям, аудит-лог действий, ограничение массовой выгрузки
без прав, HTTPS, доступ к оригиналам писем/вложений по правам, бэкапы, возможность обезличить/удалить
по регламенту. Место хранения ПДн, оператор данных и требования клиники — **решение владельца**.
**Почему:** письма содержат персональные и медицинские данные пациентов (бриф §16).
**Альтернативы:** отложить безопасность на потом — отвергнуто (ПДн с первого среза).
**Влияет на:** `.claude/rules/guardrails.md`, `core/roles-and-access.md`, `core/data-model.md` (AuditLog), `deployment.md`.

## [2026-06-21] D5. Двойная пересылка — первоклассный кейс, не частный
**Решение:** для каждого письма храним и технические данные **последнего** письма, и извлечённые данные
**исходного** (реальный отправитель, исходные дата/тема, реальные вложения, вложенные `.eml`/`message/rfc822`).
Логика разбора пересылки (`Forwarded message` / `Пересылаемое сообщение` / `От кого`…) остаётся резервным
механизмом и на рабочем этапе.
**Почему:** тестовые письма дважды пересланы (клиника→разработчик→тестовый ящик), §3/§18.1 — иначе
извлечём данные пересылки вместо данных страховой.
**Влияет на:** `core/email-ingestion.md`, `core/recognition.md`, `core/data-model.md` (EmailMessage: forwarded-поля).

## [2026-06-21] D6. Дедупликация — помечать, не удалять автоматически
**Решение:** возможные дубли определяем по сигналам (Message-ID исходного письма, hash вложения, номер
гарантийного письма, пациент+страховая+дата+услуги, совпадение исходного текста) и **помечаем** в
админке; автоматически безвозвратно не удаляем.
**Почему:** письма пересылаются и повторяются; авто-удаление рискует потерять данные (бриф §12).
**Влияет на:** `core/recognition.md`, `core/admin-panel.md`, `core/data-model.md`.

## [2026-06-21] D7. Типизация — простые типы + явные DTO + Zod на границах; минимум «магии»
**Решение:** простая, читаемая типизация: понятные **DTO** (явные `type`/`interface`), **Zod-схемы** на всех
границах (входящие письма/вложения, ответы OpenAI, формы, env, payload'ы Inngest; TS-типы — через `z.infer`,
для БД — `drizzle-zod`). **Минимум «магии»**, без сверхсложных generic-типов без необходимости. Клиент↔сервер —
через **Next.js App Router (Server Actions / Route Handlers)** с явными типизированными DTO; **tRPC по умолчанию НЕ вводим**.
**Почему:** владелец задал стиль — «простые типы, понятные DTO, Zod-схемы, минимум магии, никаких сверхсложных
generic без необходимости». Для внутренней админки (немного пользователей, §19.4) Server Actions + Drizzle-инференс
+ Zod дают сквозную типобезопасность без лишнего слоя; тяжёлые generic/RPC-абстракции труднее читать и держать в
голове без построчного ревью (противоречит самопроверяемости, D3).
**Альтернативы:** **tRPC** — отложен. **Отложенный триггер** (`quality-criteria.md`): ввести при появлении
(а) отдельного клиента (mobile/RN/внешний SPA), (б) публичного типизированного API для клиники (§19.5), или
(в) боли от бойлерплейта Server Actions на реестре/очереди. Zod вводим сразу.
**Влияет на:** `.claude/rules/code-standards.md`, `core/architecture.md`, `core/data-model.md`, `sync-with-external.md`, `quality-criteria.md`.

## [2026-06-21] D8. Инфра/CI-CD — методология sup2, но полная изоляция
**Решение:** репо `igortsk123/sib` (приватный); деплой — **push в `main`** → серверный systemd-таймер
`sib-deploy.timer` (~2 мин) гонит `sib-deploy.sh`: tag prev → docker build (**гейт typecheck+test в
Dockerfile**) → `db:migrate` → swap → smoke `/api/health` → rollback на `sib:prev`. Прод на том же VPS
`193.160.208.41`, что sup2, но **всё своё**: deploy-ключ `sib_deploy`/алиас `github-sib`, сеть `sib-net`,
БД `sib-db` (:5434), контейнер `sib-frontend` (:3006), `/opt/sib`, `/opt/sib.env`, nginx `sib.docon.pro`
+ TLS. Авто-очистка диска (image+builder prune после деплоя + еженедельный cron, без `-a` на образах).
GitHub Actions НЕ заводим — таймера достаточно (проще, меньше секретов).
**Почему:** владелец просил «CI/CD как в sup2», но «максимально независимо». Методология sup2 проверена на
этом сервере; изоляция ресурсов даёт независимость без переизобретения (sup2 можно удалить — sib живёт).
**Альтернативы:** GitHub Actions instant-trigger (как у sup2) — отложен (лишние секреты на сервер);
общий с sup2 Postgres — отвергнут (изоляция ПДн, ADR D4). Cloudflare сейчас DNS-only → HTTP-01 прошёл.
**Влияет на:** `deployment.md`, `_secrets/ACCESS.md`, `project-state.md`, `deploy/` в репо.

## [2026-06-21] D9. UI-базлайн — стандартный shadcn (radix/new-york) из WFM-admin, не из sup2
**Решение:** UI-фундамент берём из **WFM-admin** (донор админки): дизайн-токены `app/globals.css` (OKLCH,
бренд-виолет, семантические токены, badge-схемы, sidebar/chart-токены) + примитивы shadcn на **radix**
(style `new-york`). Тулчейн (Next 16/React 19/Tailwind 4/Drizzle/Zod/vitest/exceljs, Dockerfile, health) —
по образцу sup2.
**Почему:** sup2 использует shadcn `base-nova` на `@base-ui` — несовместим с radix-примитивами WFM «в лоб».
Брать UI из WFM и одновременно равняться на base-ui sup2 = лишняя адаптация и привязка к sup2. Стандартный
radix-shadcn WFM переносится напрямую и независим от sup2.
**Альтернативы:** mirror base-nova sup2 + адаптация WFM под base-ui — отвергнут (дороже, привязка к sup2).
**Влияет на:** `components.json`, `app/globals.css`, `components/ui/*`, `.claude/rules/ui-rules.md`, `core/admin-panel.md`.
