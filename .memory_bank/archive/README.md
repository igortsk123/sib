# Archive — устаревшая, но ценная память

> Сюда `/memory-cleanup` переносит доки, которые **устарели/заменены, но важны для истории**.
> Архив = обратимость очистки: ничего не удаляется безвозвратно, всё можно поднять обратно.

## Структура
```
archive/
  README.md
  YYYY/
    MM/
      <оригинальное-имя>.md
```
Группировка по году/месяцу архивации. Имя файла сохраняется (при коллизии — добавить суффикс).

## Что сюда попадает
- Доки, классифицированные очисткой как **ARCHIVE** (устарели, заменены новым).
- Оригиналы перед **DELETE** (archive-before-delete — удаление без архивной копии запрещено).
- Старые версии перед крупной перезаписью canonical-дока (по желанию).

## Что НЕ попадает
- Завершённые планы → у них своя папка `completed_plans/`.
- Активный контекст/intake.

## Frontmatter архивной заметки
При переносе в архив шапку дока дополняют (или оборачивают) полями происхождения:

```yaml
---
title: <человекочитаемое имя>
memory_type: archive_note
status: archived
source_of_truth: historical
original_path: .memory_bank/<откуда>.md
archived_at: YYYY-MM-DD
archived_by: memory-cleanup
archive_reason: <superseded_by_... | obsolete | duplicate_of_...>
superseded_by: <путь к актуальному доку или ->
---
```

## Важно для аудита
Папка `archive/` (как и `changelog/`, `completed_plans/`, `_intake/`) **исключена из `memory-audit.mjs`** —
архивные доки не считаются orphan и не участвуют в decision tree. Они живут отдельно как история.

## Как поднять из архива
Перенести файл обратно в актуальное место, убрать архивные поля, обновить `updated`/`last_verified`,
вернуть в `INDEX`/реестр. Зафиксировать в `changelog/memory-log.md`.
