# Cleanup Policy — правила очистки памяти

> Правила, по которым работает скилл `/memory-cleanup`. Цель: после крупных изменений привести
> Memory Bank в порядок — пересобрать связи, убрать дубли, заархивировать устаревшее,
> систематизировать — **безопасно и обратимо**.

## Когда запускать
- После крупного изменения / merge / смены этапа проекта.
- Когда память «разъехалась»: дубли, противоречия, старые факты, мёртвые ссылки.
- Перед хендоффом / `/clear`, чтобы следующая сессия читала чистое.
- НЕ нужно после каждой мелкой правки — для быстрой гигиены есть `/memory-check`.

## `/memory-check` vs `/memory-cleanup`
| | `/memory-check` | `/memory-cleanup` |
|---|---|---|
| Что | быстрый структурный гейт | глубокая периодическая уборка |
| Делает | regen INDEX, orphan/stale/битые ссылки | то же + дубли, устаревшее по смыслу, классификация, **архивация**, лог |
| Меняет файлы | минимум (INDEX + явные ошибки) | да, но только после подтверждения и с архивацией |
| Частота | часто | после крупных изменений |

## Модель классификации
Каждую находку очистка относит к одной категории:

| Категория | Что значит | Действие |
|-----------|-----------|----------|
| **KEEP** | актуально, на месте | ничего |
| **VERIFY** | возможно устарело / нет `last_verified` / `review_after` в прошлом / противоречие | пометить, спросить/сверить — НЕ удалять |
| **COMPRESS** | раздуто (Tier 1 >3 KB, лишние детали) | сжать, детали — в Tier 2 |
| **MERGE** | дубль/перекрытие с другим доком | свести в один, второй → ARCHIVE |
| **SPLIT** | смешаны разные темы | разнести на доки |
| **ARCHIVE** | устарело/заменено, но ценно для истории | перенести в `archive/` (не удалять) |
| **DELETE** | мусор без ценности | **только после ARCHIVE-копии** |
| **PROMOTE_TO_SOURCE_OF_TRUTH** | де-факто canonical, но помечен иначе | поднять `source_of_truth: canonical` |
| **DEMOTE_FROM_SOURCE_OF_TRUTH** | помечен canonical, но устарел/выведен | понизить статус |

## Safety-модель (обязательно)
1. **Dry-run по умолчанию.** Сначала отчёт-классификация, файлы не меняются. Применение — только по явному «применить».
2. **Archive-before-delete.** Ничего не удаляется без копии в `archive/`. Hard-delete запрещён.
3. **Canonical под защитой.** Доки `source_of_truth: canonical` (`source-of-truth.md`, `decisions.md`, `project-state.md`, …) не переписываются и не удаляются без явного подтверждения пользователя.
4. **Неуверенность → VERIFY, не DELETE/ARCHIVE.** Если непонятно, актуально ли — поднять вопрос, не уничтожать.
5. **Не трогать в работе.** `plans/*` со статусом `draft`/`in_progress`, `_intake/*` — не очищать.
6. **Логировать всё.** Каждое применённое действие → запись в `changelog/memory-log.md` (дата, файл, действие, причина, approval, команда).
7. **Сверить после.** После применения — перепрогон аудита (`/memory-check` или `node tools/memory-audit.mjs`), убедиться, что «чисто».

## Что очистка НЕ делает
- Не меняет смысл решений (`decisions.md`) — устаревшее решение не стирается, а помечается отменённым (новая ADR-запись).
- Не удаляет историю — она уходит в `archive/`.
- Не делает массовых правок без отчёта и подтверждения.
