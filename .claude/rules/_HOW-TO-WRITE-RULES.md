# Как писать свои path-scoped правила

> Этот файл — инструкция, не правило (не загружается автоматически как rule). Удали или оставь
> для справки. path-scoped правила живут в `.claude/rules/*.md` и образуют Tier 0 (path-scoped).

## Зачем
Path-scoped rule грузится Claude Code автоматически, когда агент трогает файл, подходящий под
`paths:`. Так нужный контекст приходит ровно в момент работы — без раздувания постоянного Tier 0.

## Формат
```yaml
---
description: <одна строка — что это правило регулирует>
paths: ["services/**/*.ts", "app/api/**/*.ts"]   # glob-маски; опусти для always-on
alwaysApply: true                                 # вместо paths — если правило всегда нужно
---

# Заголовок

Короткие императивные пункты. Полную версию с обоснованиями держи в `guides/<тема>.md`
и сошлись на неё одной строкой.
```

## Принципы
- **Тонко:** ≤~60 строк. Длинное — в `guides/` (Tier 2), сюда только суть + ссылка.
- **Императивно:** «делай / не делай», таблицы, чек-листы. Без воды.
- **Один домен на файл:** код-стандарты, UI, API-контракты, тон, безопасность — раздельно.
- **paths точечно:** только релевантные маски, иначе правило грузится где не надо.
- **Регистрируй:** добавь новое правило в перечень в `CLAUDE.md` и (если процессное) в `INDEX.md`.

## Типовые правила, которые стоит завести по мере роста
- `api-contracts.md` (`services/`, `types/`) — DTO, форматы ошибок, polling.
- `<домен>-tone.md` — формулировки/тон, если продукт чувствителен к словам.
- `guardrails.md` — protected-paths (см. `_optional/rules/guardrails.md`).
- `agent-orchestration.md` — когда параллелить агентов (см. `_optional/rules/`).
