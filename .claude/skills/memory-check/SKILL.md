# memory-check — проверка консистентности памяти (без Node)

Ты выполняешь аудит Memory Bank вручную (то же, что делает `tools/memory-audit.mjs`, но твоими
силами — Node не нужен). Работай аккуратно, меняй только `INDEX.md` и явные ошибки.

> **check vs cleanup.** `/memory-check` — быстрый структурный гейт (regen INDEX, orphan/stale/links),
> меняет минимум, запускать часто. Для глубокой уборки после крупных изменений (дубли, устаревшее по
> смыслу, классификация, **архивация**) есть `/memory-cleanup` — он шире и пишет лог.

## Что считать «доком памяти»
Все `*.md` в `.memory_bank/`, КРОМЕ: папок `_intake/`, `completed_plans/`, `archive/`, `changelog/`;
файлов `README.md`, `_template.md`, `INDEX.md`, `METADATA_SCHEMA.md`, `CLEANUP_POLICY.md`; всего
внутри `plans/`. Эти — служебные/инфраструктурные, их не аудируем.

## Шаги

1. **Собрать frontmatter.** Прочитать шапки всех доков памяти: `tier`, `topic`, `scope`,
   `tier1`/`tier2`, `updated`, `importance`. (Шапка — между первой парой `---`.)

2. **Пересобрать decision tree в `INDEX.md`.** Между маркерами
   `<!-- GENERATED:decision-tree START -->` и `<!-- GENERATED:decision-tree END -->` заменить
   таблицу. Включать строки для доков с `tier: 1` и заполненным `topic`, КРОМЕ always-on тем
   (`source-of-truth`, `project-state`, `decisions`) — у них своя секция. Формат строки:
   `| <scope> | \`<путь к доку>\` | <tier2 или —> |`. Сначала `importance: high`.

3. **Orphan.** Любой док памяти без `topic` в шапке → сообщить: «ORPHAN <путь> — нет topic,
   невидим в навигации». (Чинить: добавить осмысленный `topic`/`scope` или удалить док.)

4. **Stale.** Для дока Tier 1 с `tier2:` — если у целевого Tier 2 дока `updated` новее, чем у
   сводки → «STALE <путь> — сводка старше своего Tier 2, сверь содержимое».

5. **Broken.** Указатели `tier1:`/`tier2:` и ссылки `[[имя]]` в тексте (вне код-блоков), которые
   не ведут на существующий файл/topic → «BROKEN <путь> — <что не резолвится>».

6. **Отчёт.** Кратко: сколько доков, что в decision tree обновилось, список проблем (или «чисто»).
   Очевидные мелочи (битая ссылка из-за опечатки) — предложить или сразу поправить с пометкой.

## Принципы
- Не выдумывай — если данных в шапке нет, помечай как проблему, а не угадывай.
- Меняй минимум: `INDEX.md` (decision tree) и явные ошибки. Содержимое сводок не переписывай без нужды.
- Это быстрая гигиена памяти, а не редактирование смысла.
