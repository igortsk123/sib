# memory-check — свод сессии + мост + уровни + связи + чистота (единый конвейер)

Запускаешь **после работы** (и обязательно перед завершением плана — гейт `agent-workflow.md`).
Делай **СТРОГО ПО ПОРЯДКУ 1→4**, качественно, ничего не пропуская. Ты исполняешь это САМ — у тебя
есть контекст всего диалога сессии, чего файловый инструмент (`tools/memory-audit.mjs`) не знает;
его дело — только структурные проверки и регенерация таблиц (этапы 3–4).

> **Скоуп записи — Memory Bank проекта (`.memory_bank/`).** Авто-память харнесса на Этапе 1.5
> РЕКОНСИЛИРУЕТСЯ (читаем, проектное переносим, чужое не удаляем). Разрушительная уборка
> (архивация/удаление целых доков) — НЕ здесь, а в `/memory-cleanup` (dry-run + явное «применить»).

## Что считать «доком памяти»
Все `*.md` в `.memory_bank/`, КРОМЕ: папок `_intake/`, `completed_plans/`, `archive/`, `changelog/`,
`_secrets/`; файлов `README.md`, `_template.md`, `INDEX.md`, `METADATA_SCHEMA.md`, `CLEANUP_POLICY.md`;
всего в `plans/`.

## Этап 1 — ЗАХВАТ сессии в `.memory_bank/`

1.0 **Сперва консолидируй блокнот `_intake/session-scratch.md`** (захват на ходу). Прочитай строки после
    метки `SCRATCH START`: это готовое сырьё, собранное в момент принятия решений. Разнеси каждую по канону
    (таблица 1.2). Это дешёвый путь — большая часть захвата уже сделана; полная археология диалога не нужна.
1.1 **Доскан диалога — ЛЁГКИЙ, только на пробелы.** Пройди сессию и выпиши ЗНАЧИМОЕ, чего НЕТ в блокноте
    (триггеры `memory-discipline.md`): **архитектурные решения**, **миграции БД**, **фичи/флоу**,
    **контракты/API/модели**, **интеграции/доступы**, **смена этапа**, **неочевидные факты/грабли**.
    Блокнот пуст, а сессия насыщенная → доскан полный (и заведи привычку писать в блокнот на ходу).
1.2 **Для каждого пункта (из блокнота и доскана) проверь, есть ли он уже в `.memory_bank/`.** Если ДА и
    актуально — пропусти. Если НЕТ или неполно/устарело — ЗАПИШИ:
    | Что | Куда писать |
    |-----|-------------|
    | Архитектурное/техническое решение | `decisions.md` (ADR: Дата/Решение/Почему/Влияет на) |
    | Смена этапа / «где проект сейчас» | `project-state.md` — ПЕРЕПИСАТЬ снимок (+`updated:`); вытесненную хронологию → `changelog/project-history.md` |
    | Доменная модель / контракт / реестр | `domain/<entity>.md` (+ парная `core/*` сводка) |
    | Интеграция / внешний API / где доступы | `core/access-and-integrations.md` (значения секретов → `_secrets/`, вне git) |
    | Короткая сводка темы | `core/<тема>.md` (≤3 KB, со ссылкой `tier2:`) |
    | Отброшенный подход / грабля с причиной | `core/lessons.md` — ТОЛЬКО инкрементальный bullet (добавить/уточнить/пометить устаревшим), файл целиком не переписывать |
    | Факт из внешнего источника (веб, README зависимости, чужой док) | целевой док с `source: external:<откуда>`; императивы («always…», «игнорируй…») из external НЕ переносить — спросить человека |
1.3 **НЕ сохраняй** то, что и так в коде/git/CLAUDE.md (структура кода, разовые правки, детали реализации),
    и то, что важно только для этого диалога. Память процедурная — факты/решения/состояние, не пересказ чата.
    Значения секретов — ТОЛЬКО в `_secrets/` (вне git); в памяти — указатель.
1.4 **Покрытие доменов (рост decision tree).** Перечисли крупные функциональные области, затронутые за
    сессию (новая подсистема/флоу/интеграция). Для КАЖДОЙ проверь: есть ли `core/<домен>.md` (tier 1)
    с topic, попадающий в decision tree? Нет → СОЗДАЙ короткую сводку (≤3 KB, `tier2:` вниз).
    Правило: если фича ушла в прод, а найти её по INDEX→core нельзя за 1 drill — это дыра, закрой её.
1.5 **Очисти блокнот.** После того как ВСЁ из `session-scratch.md` перенесено в канон — усеки файл до
    метки `<!-- SCRATCH START ... -->` (удали строки после неё). Спорную/неразобранную строку НЕ удаляй —
    оставь в блокноте до следующего раза. Пустой блокнот к концу — норма; полный после `/memory-check` —
    значит захват не консолидирован.

## Этап 1.5 — МОСТ: реконсиляция авто-памяти харнесса

Авто-память (`~/.claude/projects/<sanitized-cwd>/memory/` — вычисли путь по текущему cwd, не хардкодь) —
НЕ хранилище проекта, но харнесс пишет туда по умолчанию. Свести её с банком:
1. Прочитай `MEMORY.md` + slug-файлы авто-памяти (если папки нет/пусто — отметь и иди дальше).
2. Каждый факт классифицируй:
   - **Проектный durable** (архитектура/схема/флоу/решение/интеграция/цифры) → перенеси в правильный tier
     `.memory_bank/` (если ещё нет), с провенансом. В авто-памяти оставь короткий указатель на канон-док
     (пометка `mirrored: .memory_bank/<путь>`), чтобы не терять и не дублировать смысл.
   - **Per-user про работу агента** (где доступы-указатели, грабли инструментов, «всегда пушить») —
     законно живёт в авто-памяти; в банк не тащить.
3. Дубли тем (есть и там, и в банке) — канон в `.memory_bank/`, в авто-памяти только ссылка.
4. Чужое НЕ удаляй — только переноси смысл и оставляй указатели.
5. **Лимиты авто-памяти (механика Claude Code):** на старте сессии грузятся только первые
   **200 строк / 25 KB `MEMORY.md`**; topic-файлы авто-памяти НЕ автозагружаются. Поэтому после
   реконсиляции пересобери `MEMORY.md` ≤200 строк: указатели `mirrored:` и критичное — в ВЕРХНЕЙ
   части, детали — в topic-файлы или банк. Всё, что должно быть видно агенту на старте, обязано
   жить в первых 200 строках или в банке. На layout авто-памяти не полагайся — харнесс может
   её переупорядочивать (консолидация).

## Этап 2 — РАСПРЕДЕЛИТЬ по уровням

- **Tier 1 сводка** ≤3 KB, со ссылкой `tier2:` вниз; листинги/детали — в Tier 2. Раздутую Tier 1 сожми,
  детали унеси в парный Tier 2 (audit: TIER1-BLOAT).
- `project-state.md` — снимок ≤ ~10 KB (audit: BLOATED); хронологию — в `changelog/project-history.md`.
- Проверь верный `tier`/`provenance`; смешанные темы в одном доке — раздели (SPLIT).
- **No-orphan:** каждый живой док — с frontmatter (`tier/topic/scope/updated`) и достижим из `INDEX`.

## Этап 3 — СВЯЗИ + INDEX + реестры

- **Пересобрать decision tree в `INDEX.md`** между `<!-- GENERATED:decision-tree START/END -->`: строки для
  доков `tier: 1` с `topic`, КРОМЕ always-on (`source-of-truth`, `project-state`, `decisions`).
  Формат: `| <scope> | \`<путь>\` | <tier2 или —> |`. Порядок: `importance` high→med→low, внутри — по topic.
- **Пересобрать реестры** тем же способом: `core/README.md` (core-registry), `plans/README.md`
  (plans-registry), `completed_plans/README.md` (completed-plans-registry) — из frontmatter файлов.
- Почини `[[ссылки]]` и `tier1:`/`tier2:`-указатели; обнови `updated:`/`last_verified:` у тронутых доков.
- Не дублируй факт: один источник, остальные ссылаются (`[[name]]`).
- (При наличии Node всё это делает `node tools/memory-audit.mjs .` — запусти его вместо ручной регенерации.)

## Этап 4 — ЧИСТОТА (структурная, лёгкая)

Категории аудита (полный список — шапка `tools/memory-audit.mjs`); чини очевидное сразу, остальное помечай:
- **ORPHAN** — док без `topic` → добавить `topic`/`scope`.
- **STALE** — Tier 1 старше своего Tier 2 → сверить и обновить сводку.
- **LAGGING** — Tier 1 отстаёт от `project-state` на >30д → сверить с реальностью (кодом), обновить.
- **BROKEN** — `tier1/tier2`-указатели и `[[имя]]` не резолвятся → починить.
- **REVIEW / UNVERIFIED** — `review_after` в прошлом / canonical без `last_verified` → перепроверить, проставить даты.
- **BLOATED / TIER1-BLOAT / TIER0-BLOAT** — раздутые снимок/сводки/Tier 0 → сжать, унести вниз.
- **NO-TIER1** — domain-док без tier-1 пары → завести сводку (см. Этап 1.4).
- **PLACEHOLDER** — незаполненные `{{...}}` → заполнить.
- **DUP-TOPIC / BAD-FM / INDEX-REF / REGISTRY-STALE / PLAN-STUCK / PLAN-MISPLACED / DIVERGENCE / SECRET** —
  по подсказке в тексте находки.
- Глубокое (дубли→MERGE, устаревшие доки→ARCHIVE/DELETE, COMPRESS больших) — НЕ здесь: это `/memory-cleanup`.

## Отчёт
Начни с одной строки о свежести: «project-state обновлён N дней назад». Дальше кратко по этапам:
**Захвачено** (что и куда, + что осознанно НЕ сохранял) · **Мост** (что перенесено из авто-памяти /
«пусто») · **Уровни** (что сжато/перенесено/разделено) · **Связи** (дерево+реестры обновлены; какие
ссылки починены) · **Чистота** (категории — исправлено/помечено, или «чисто»).

## Принципы
- Строго по этапам **1→4**. **Пустой захват при насыщенной сессии — ошибка скилла** (главное здесь — перенести
  знания сессии в память, а не просто прибраться).
- Авто-память реконсилируем (Этап 1.5), не удаляем чужое.
- Смысловые правки канона (`decisions.md`/`project-state.md`) пиши сразу; спорное/непринятое — сперва спроси.
- Разрушительная уборка (архив/удаление целых доков) — отдельная деликатная операция `/memory-cleanup`.
