# Android-приложение — критерии качества (project-agnostic)

> **Что это.** Переносимая целевая планка качества для **любого** Android-приложения:
> эталон для код-ревью, Definition of Done спринта и CI quality gates. Документ
> намеренно очищен от привязки к конкретному продукту/модулям — конкретику (имена
> экранов, фич, флагов, порогов) проект добавляет отдельным overlay-маппингом.
>
> **Как пользоваться.** Каждое измерение имеет **стабильный ID** вида `ARCH-1`; при
> планировании берём релевантные критерии как acceptance-условия и ссылаемся на них
> по ID. Структура измерения: *Принцип* (зачем) → таблица критериев (ID + уровень +
> порог/проверка) → *Антипаттерны* (что считаем дефектом).
>
> **Шаблон ↔ overlay.** Этот файл — переносимый **шаблон**. Проектную привязку
> (конкретные обёртки, стек, текущий baseline и разрывы) держим в отдельном
> **overlay** рядом — `android-quality-criteria.<project>.md`, где каждый ID привязан
> к месту в коде, спринту и честной оценке baseline. Шаблон остаётся чистым.
>
> **Формат и уровни.** Каждый критерий **проверяем**: измеримый порог или бинарное
> «есть/нет» с местом проверки. Критерий без зелёной проверки считается невыполненным
> — «частично» не засчитывается.
> Легенда обязательности: **[B]** blocker — без этого PR не мёржится · **[M]** major —
> фиксим в спринте · **[N]** nice — техдолг.
>
> **Каноны, на которые опираемся:**
> - Архитектура: Clean Architecture, Guide to App Architecture (Android), MVVM/MVI, UDF (unidirectional data flow).
> - Модульность: Now in Android (modularization guidance), Gradle convention plugins, version catalogs.
> - Производительность: Android Vitals, Macrobenchmark/Baseline Profiles, Compose performance (stability/recomposition), Perfetto.
> - Память/энергия: App Standby Buckets/Doze, WorkManager guidance, LeakCanary.
> - Безопасность: OWASP MASVS / Mobile Top 10, Android security best practices, Keystore.
> - Доступность: WCAG 2.2 (AA), Android Accessibility (TalkBack, target size).
> - Интернационализация: Unicode/UTF-8, ICU/CLDR, ISO 8601, IANA TZ, ISO 4217.
> - Наблюдаемость/поставка: three pillars (logs/metrics/traces), Play Vitals, DORA-метрики, staged rollout.
> - Тестирование: пирамида/Testing Trophy, Compose UI Test, screenshot-тесты.
> - Код: Clean Code, SOLID (DIP/LSP), composition over inheritance — по месту, без over-engineering.
>
> **Отличие от классического чек-листа.** К инженерному ядру (Архитектура, Модульность,
> Навигация, Производительность, Память, Энергия, Безопасность, Повторное
> использование) добавлены ранее упущенные сквозные аспекты, помеченные ★:
> Доступность, Интернационализация, Адаптивный UI, Состояния экрана, Сетевая
> устойчивость, Данные/миграции, Разрешения/приватность, Совместимость платформы,
> Наблюдаемость, Релиз-инжиниринг. Неприменимость измерения фиксируется явно (N/A +
> обоснование, см. Definition of Done).

Шкала зрелости каждого измерения: **0** нет практики · **1** точечно · **2** есть,
но без enforcement · **3** enforced (lint/CI/тест) · **4** эталон. Цель миграции —
**сдвиг проверок влево**: от ручного ревью к CI-enforcement.

Базовый принцип сквозной: **чинить корень причины, а не симптом**; преждевременную
абстракцию не вводим (правило «трёх повторов»). Метрики измеряем на **release-сборке**
+ минимум одном «слабом» устройстве (≈ 2–3 ГБ RAM, API 26).

## Часть 0. Бюджеты UX и стабильности как корень

Best practice (по аналогии с SRE-SLO): сначала фиксируем пользовательские бюджеты —
остальные измерения (PERF, MEM, RESIL, OPS, A11Y) производны. Конкретные цифры
подтверждаются с продуктом и владельцем надёжности.

| Метрика | Цель (дефолт, уточнить) |
|---------|-------------------------|
| Cold start TTID (p99, baseline-устройство) | ≤ 500 мс |
| Warm start | ≤ 300 мс |
| Jank: кадры без пропуска | ≥ 99%; нет frozen frames (>700 мс) |
| Crash-free users | ≥ 99.5% (для критичных — выше) |
| ANR-rate (Play Vitals) | в зоне «good» (< 0.47% затронутых сессий) |
| Размер APK/AAB | ≤ согласованного; рост > X% ломает сборку |
| Критичные нарушения a11y (Accessibility Scanner) на ключевых экранах | 0 |
| Покрытие критичных пользовательских флоу тестами | ≥ согласованный порог |
| Утечки памяти (LeakCanary) на ключевых сценариях | 0 |
| Потеря пользовательского ввода при process death | 0 (через `SavedStateHandle`) |

Нарушение бюджета блокирует релиз (см. Definition of Done); измерения ниже описывают,
как эти бюджеты держать.

## Часть I. Сквозные измерения

### 1. Архитектура (ARCH)

**Принцип.** Зависимости текут **внутрь** (Presentation → Domain ← Data); домен не
знает про Android-фреймворк — это и позволяет менять UI и хранилище независимо.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| ARCH-1 | Зависимости направлены внутрь; Domain не знает про Android SDK, Compose, Retrofit, Room | **[B]** | grep по `domain/`: нет импортов `android.*`, `androidx.*`, `retrofit2.*`, `androidx.room.*` |
| ARCH-2 | ViewModel не содержит ссылок на `Context`, `View`, `Activity`, Android-ресурсы | **[B]** | ревью + lint `StaticFieldLeak` |
| ARCH-3 | Каждый экран публикует единственный `StateFlow<UiState>`; `UiState` — sealed (Loading/Success/Error) | **[B]** | ревью VM |
| ARCH-4 | UI stateless: Composable получает state + лямбды событий; ViewModel не дёргается из вложенных Composable (только корневой screen) | **[M]** | ревью |
| ARCH-5 | Бизнес-логика — в UseCase/репозитории, не в ViewModel и не в Composable; ViewModel только оркестрирует | **[M]** | ревью |
| ARCH-6 | Маппинг DTO ↔ domain ↔ entity явный, в Data-слое; DTO от Retrofit не утекают в Presentation | **[M]** | grep: `*Dto` не импортируется из `presentation/`/`domain/` |
| ARCH-7 | Side-effects (навигация, snackbar, toast) — через одноразовый канал (`Channel`/`SharedFlow`), не через `StateFlow` | **[M]** | ревью |
| ARCH-8 | Ошибки доменного уровня типизированы (sealed `DomainError`/`Result`), не голые `Exception` в UI | **[M]** | ревью |

**Антипаттерны**
- ViewModel держит `Context`/`View`; бизнес-логика живёт в Composable.
- DTO Retrofit прокинут прямо в UI; маппинг размазан по слоям.
- Side-effect отдан через `StateFlow` → повторное срабатывание (тост/навигация) при rotation.
- Голый `Exception`/`Throwable` доходит до UI без типизации.

### 2. Модульность (MOD)

**Принцип.** Систему меняют по частям. Чёткие границы и однонаправленные зависимости
позволяют менять фичу, не ломая соседей.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| MOD-1 | `:feature:*` не зависят друг от друга; межфичевое взаимодействие — только через `:core` или навигацию (route + args) | **[B]** | `./gradlew :feature:X:dependencies` |
| MOD-2 | `:core:*` не зависят от `:feature:*` и от `:app` | **[B]** | граф зависимостей |
| MOD-3 | `:app` — только сборка графа DI + NavHost; нет бизнес-логики и экранов фич | **[M]** | ревью |
| MOD-4 | Публичный API модуля минимален: внутренние классы `internal` | **[M]** | grep `internal` в data/impl-классах |
| MOD-5 | Зависимости версионируются через version catalog (`libs.versions.toml`), без хардкода версий в `build.gradle.kts` | **[M]** | grep версий-строк в `*.gradle.kts` |
| MOD-6 | Один feature = один вертикальный срез (data+domain+presentation); нет «God-модуля» | **[M]** | структура |
| MOD-7 | Фича не тянет лишние `:core`-модули «на всякий случай»; время чистой сборки не деградирует | **[N]** | `dependencies` + build scan |
| MOD-8 | Конвенции сборки вынесены в `build-logic`/convention plugins (нет копипасты gradle-блоков) | **[N]** | наличие convention plugins |

**Антипаттерны**
- `:feature:a` импортирует внутренности `:feature:b` напрямую.
- `:core` знает о `:feature`/`:app`; бизнес-логика просочилась в `:app`.
- Версии библиотек захардкожены в каждом `build.gradle.kts`.
- «God-модуль», в который складывают всё подряд.

### 3. Навигация и организация Activity (NAV)

**Принцип.** Single Activity + type-safe граф; навигация — это состояние: инициируется
событиями, переживает конфиг-изменения и process death.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| NAV-1 | Одна `Activity` (`MainActivity`); новые Activity не заводятся без ADR; экраны = Composable destinations | **[B]** | grep `: ComponentActivity` / манифест |
| NAV-2 | Маршруты type-safe (`@Serializable` в Navigation 2.8+), без строковой склейки путей и ручного парсинга аргументов | **[B]** | ревью NavHost |
| NAV-3 | Навигация инициируется из ViewModel-события (ARCH-7) → корневой NavHost; `NavController` не прокидывается глубоко | **[M]** | ревью |
| NAV-4 | Граф разбит по фичам (nested graphs); `:app` агрегирует; фича объявляет destinations через extension на `NavGraphBuilder` | **[M]** | структура NavHost |
| NAV-5 | `BackHandler`/back-stack корректны: нет «двойного» экрана, нет утечки back-stack при возврате на корневой таб | **[M]** | ручной прогон + ревью |
| NAV-6 | Аргументы навигации — идентификаторы/примитивы, не тяжёлые объекты; данные грузятся на целевом экране | **[M]** | ревью routes |
| NAV-7 | Deep links / push-навигация ведут через тот же type-safe граф, с восстановлением back-stack | **[M]** | тест deep link |
| NAV-8 | Конфиг-изменения (rotation, dark mode) не теряют состояние и не рестартят навигацию | **[M]** | прогон с «Don't keep activities» |
| NAV-9 | Process death: `SavedStateHandle` хранит ключевой ввод (черновики форм, фильтры) | **[M]** | тест с принудительным kill процесса |

**Антипаттерны**
- Новая `Activity` на каждый экран; маршруты собираются конкатенацией строк.
- `NavController` прокинут в фичевые Composable глубоко в дереве.
- Тяжёлые Parcelable-модели в аргументах навигации.
- Поворот экрана теряет состояние / рестартит граф.

### 4. Производительность (PERF)

**Принцип.** Пользователь платит за каждый кадр и миллисекунду старта; не делаем лишней
работы на main-потоке и в рекомпозиции.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| PERF-1 | Cold start (TTID) ≤ 500 мс, warm ≤ 300 мс на baseline-устройстве (Часть 0) | **[M]** | Macrobenchmark `StartupTimingMetric` |
| PERF-2 | Jank: ≥ 99% кадров без пропуска; нет frozen frames на основных экранах | **[M]** | Macrobenchmark `FrameTimingMetric` / Perfetto |
| PERF-3 | Нет лишних рекомпозиций: стабильные параметры, `remember` для дорогих вычислений, ключи в `LazyColumn` | **[B]** | Layout Inspector + Compose compiler metrics |
| PERF-4 | Списки — `LazyColumn`/`LazyVerticalGrid` с `key = {…}`; без рендера всего датасета; пагинация (Paging) для длинных списков | **[B]** | ревью |
| PERF-5 | Нет блокирующих вызовов в main: IO/CPU на `Dispatchers.IO`/`Default`; `runBlocking` запрещён в проде | **[B]** | grep `runBlocking`; StrictMode |
| PERF-6 | `collectAsStateWithLifecycle()` вместо `collectAsState()` | **[M]** | grep `collectAsState(` |
| PERF-7 | Изображения через image loader (Coil/Glide) с явным `size`/downsampling; нет full-res в превью/списки | **[M]** | ревью вызовов загрузки |
| PERF-8 | R8/ResourceShrinking включены для release; Baseline Profile сгенерирован для горячих путей | **[M]** | `build.gradle` + baseline-prof.txt |
| PERF-9 | Запросы к БД/сети не в цикле UI; Room возвращает `Flow`, тяжёлые — пагинированы | **[M]** | ревью DAO |
| PERF-10 | Размер APK/AAB под контролем: ресурсы оптимизированы (WebP/vector), нет неиспользуемых зависимостей | **[N]** | APK Analyzer |

**Антипаттерны**
- `runBlocking`/синхронный IO на main-потоке.
- Рендер всего датасета без `LazyColumn` и без `key`; список «дёргается».
- `collectAsState()` — flow работает в фоне ушедшего экрана.
- Загрузка full-res изображений в списки/превью; нет Baseline Profile.

### 5. Потребление памяти (MEM)

**Принцип.** Ресурсы и подписки живут не дольше своего владельца; утечка — баг, а не «потом почистим».

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| MEM-1 | Нет утечек памяти на ключевых флоу | **[B]** | LeakCanary в debug — 0 leaks за прогон |
| MEM-2 | Корутины привязаны к `viewModelScope`/`lifecycleScope`; нет «висящих» job после ухода с экрана | **[B]** | ревью + LeakCanary |
| MEM-3 | Камера (CameraX)/сенсоры/распознавание освобождаются (`unbind`, закрытие analyzer) на `onStop`/уходе | **[B]** | ревью lifecycle |
| MEM-4 | Bitmap/изображения не держатся дольше нужного; memory cache image loader настроен (не «бесконечный») | **[M]** | Memory Profiler heap dump |
| MEM-5 | Нет статических ссылок на `Context`/`Activity`/`View` (companion, object, синглтоны) | **[B]** | lint `StaticFieldLeak` |
| MEM-6 | Слушатели/коллбеки (sensors, location, broadcast, Flow) снимаются симметрично регистрации | **[M]** | ревью |
| MEM-7 | Нет неограниченного роста in-memory кэшей (списки/история ограничены или пагинированы) | **[M]** | ревью + heap dump под нагрузкой |
| MEM-8 | Под давлением памяти нет OOM; реакция на `onTrimMemory`/release кэшей | **[N]** | стресс-тест |

**Антипаттерны**
- Статическая ссылка на `Context`/`Activity` в синглтоне/companion.
- Камера/analyzer не освобождены на `onStop` → удержание превью в памяти.
- Job переживает экран; подписка зарегистрирована, но не снята.
- In-memory кэш растёт без границ/eviction.

### 6. Энергопотребление (PWR)

**Принцип.** Фон дорог по батарее; вся фоновая работа управляема, батчится и переживает Doze.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| PWR-1 | Фоновая работа — только через WorkManager; нет своих фоновых сервисов/потоков | **[B]** | grep `Service`/`Thread`; ревью |
| PWR-2 | Воркеры используют `Constraints` (network/charging где уместно) и backoff; нет агрессивного поллинга | **[B]** | ревью Worker constraints |
| PWR-3 | Нет постоянных `WakeLock`/foreground-сервисов без необходимости и ADR | **[B]** | grep `WakeLock`, манифест |
| PWR-4 | Сетевые запросы батчатся; нет частых одиночных запросов в фоне; refresh-токен лениво, по 401 | **[M]** | ревью + Network Profiler |
| PWR-5 | Камера/сенсоры/GPS активны только на нужном экране и снимаются при уходе/`onStop` | **[B]** | ревью (пересекается с MEM-3/MEM-6) |
| PWR-6 | FCM обрабатывается без тяжёлой работы на main; долгие задачи делегируются WorkManager | **[M]** | ревью FCM service |
| PWR-7 | Анимации/Canvas-графики не крутятся вхолостую вне экрана | **[N]** | ревью |
| PWR-8 | App Standby Buckets / Doze учтены: воркеры переживают Doze, не нарушая лимиты | **[N]** | тест в Doze (`adb shell dumpsys deviceidle`) |

**Антипаттерны**
- Свои `Thread`/`Service` вместо WorkManager; поллинг каждые N секунд.
- Постоянный `WakeLock`/foreground-сервис без обоснования.
- GPS/камера/сенсор активны вне нужного экрана.

### 7. Безопасность (SEC)

**Принцип.** Токены, секреты и трафик защищены; в логах нет PII; поверхность атаки минимальна.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| SEC-1 | `access`/`refresh` токены — в DataStore + EncryptedSharedPreferences/Keystore, не в открытом виде и не в логах | **[B]** | ревью + grep токенов в `Log.*` |
| SEC-2 | Только HTTPS; cleartext запрещён (`usesCleartextTraffic=false`); рассмотреть certificate pinning для прод-API | **[B]** | манифест + network-security-config |
| SEC-3 | В логах нет PII/токенов/тел с чувствительными данными; `HttpLoggingInterceptor` — `NONE`/`BASIC` в release | **[B]** | grep уровня логирования по buildType |
| SEC-4 | Никаких секретов/ключей в репозитории и коде; API-ключи — через `local.properties`/CI secrets | **[B]** | grep + проверка `.gitignore` |
| SEC-5 | Release собирается с R8 + обфускацией (minify/shrink on) | **[M]** | `build.gradle` release block |
| SEC-6 | Приватные данные пользователя не кэшируются и не отображаются вне прав доступа | **[M]** | ревью репозиториев |
| SEC-7 | Чувствительные экраны — `FLAG_SECURE`/скрытие в recents по требованию | **[N]** | ревью |
| SEC-8 | Экспортируемые компоненты (`exported=true`) минимизированы; deep-link intents валидируют ввод | **[M]** | манифест + ревью deep link |
| SEC-9 | Зависимости без известных CVE; периодический аудит | **[N]** | `dependencyCheckAnalyze` / Dependabot |
| SEC-10 | Backup контролируется (`allowBackup`, `dataExtractionRules`) — токены не утекают в облачный бэкап | **[M]** | манифест |

**Антипаттерны**
- Токен в обычных `SharedPreferences`/в `Log.d`; cleartext HTTP-трафик.
- Секреты/ключи закоммичены в VCS или зашиты в бандл.
- Release без обфускации; `exported=true` без валидации входных intents.

### 8. Повторное использование (REUSE)

**Принцип.** Один способ сделать UI-вещь. Канонические примитивы и общие хелперы
устраняют дрейф и дублирование; абстрагируем только по факту повтора (правило трёх).

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| REUSE-1 | UI-примитивы (кнопки, поля, карточки, баннеры) — в `:core:ui` (design system) и переиспользуются; фичи не дублируют базовые Composable | **[B]** | grep дублей; ревью |
| REUSE-2 | Цвета/типографика/отступы — только токены темы; нет хардкода hex-цветов и `*.dp` magic-чисел в фичах | **[M]** | grep `Color(0x` / литералов в фичах |
| REUSE-3 | Общие форматтеры (дата, числа), валидаторы, константы — в `:core:common`, не копируются по фичам | **[M]** | grep дублей форматтеров |
| REUSE-4 | Сетевой клиент, Authenticator, сериализация — единые в `:core:network`; фичи описывают только свои API-интерфейсы | **[B]** | структура network |
| REUSE-5 | Room-сущности/DAO общего пользования — в `:core:database`; нет дублей сущностей | **[M]** | структура |
| REUSE-6 | Переиспользуемые приёмы зафиксированы в общем каталоге паттернов; при 3+ повторениях — выносим | **[N]** | каталог паттернов |
| REUSE-7 | Composable в `:core:ui` имеют `@Preview` и не зависят от ViewModel/DI (чистые, конфигурируемые параметрами) | **[M]** | ревью |
| REUSE-8 | Нет копипасты UiState/маппинга между фичами — общий каркас (base VM/UiState helper) при повторении | **[N]** | ревью |

**Антипаттерны**
- Своя копия базовой кнопки/карточки в фиче вместо `:core:ui`.
- Хардкод `Color(0xFF…)`/`16.dp` magic-числа вместо токенов.
- Копипаст форматтера даты/денег по трём фичам; дубль Room-сущности.

### 9. ★ Доступность (A11Y)

**Принцип.** Интерфейс обязан работать с TalkBack и без точного тача. Дешевле всего
чинить на уровне `:core:ui` — один источник, массовый эффект.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| A11Y-1 | Значимые иконки/изображения имеют `contentDescription`; декоративные — явно `null` | **[B]** | Accessibility Scanner + TalkBack |
| A11Y-2 | Тач-таргеты ≥ 48dp | **[M]** | Accessibility Scanner |
| A11Y-3 | Контраст текста/иконок ≥ 4.5:1 (≥ 3:1 для крупного) | **[M]** | Accessibility Scanner / токены |
| A11Y-4 | Масштаб шрифта до 200%: размеры в `sp`, верстка не обрезается/не перекрывается | **[M]** | прогон с увеличенным шрифтом |
| A11Y-5 | Корректная семантика Compose: `semantics`, `mergeDescendants`, роли, состояния (selected/checked) | **[M]** | ревью + TalkBack |
| A11Y-6 | Логичный порядок фокуса, навигация с клавиатуры/D-pad; нет «ловушек» фокуса | **[M]** | физ. клавиатура / Switch Access |
| A11Y-7 | Ключевые флоу проходимы целиком под TalkBack | **[B]** | ручной прогон |
| A11Y-8 | Уважается «уменьшить анимацию» (reduce motion / animator scale) | **[N]** | ревью |

**Антипаттерны**
- Иконочная кнопка без `contentDescription`; декоративная картинка озвучивается.
- Тач-таргет < 48dp; элементы в упор друг к другу.
- Цвет — единственный носитель смысла (ошибка только красным).
- Верстка ломается при крупном системном шрифте.

### 10. ★ Локализация и интернационализация (I18N)

**Принцип.** Текст, даты, числа и направление письма зависят от локали; формат не хардкодим.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| I18N-1 | Ноль хардкод-строк в UI — всё через ресурсы (`strings.xml`) | **[B]** | lint `HardcodedText` |
| I18N-2 | Нет конкатенации строк — только плейсхолдеры (`%1$s`) и `plurals` | **[M]** | grep / ревью |
| I18N-3 | Поддержка RTL: `start/end` вместо `left/right`, `supportsRtl=true` | **[M]** | dev-опция Force RTL |
| I18N-4 | Даты/числа/валюты — по локали (`NumberFormat`/`DateTimeFormatter` с locale), без ручной склейки | **[M]** | ревью форматтеров |
| I18N-5 | Строки не предполагают грамматику одного языка; учтены длинные переводы | **[N]** | ревью |
| I18N-6 | Псевдолокализация прогоняется (обрезанные/непереведённые строки) | **[N]** | pseudolocale `en-XA`/`ar-XB` |

**Антипаттерны**
- Захардкоженный текст в Composable/layout; `"Найдено " + count + " товаров"`.
- `margin-left`/`padding(start=…)` через `left/right`; ручной формат даты вместо locale-aware.

### 11. ★ Адаптивный UI и форм-факторы (ADAPT)

**Принцип.** Один UI — разные размеры экрана, плотности, темы и системные вставки.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| ADAPT-1 | Layout строится на Window Size Classes; нет жёсткой привязки к телефону | **[M]** | прогон на планшете/складном/resizable |
| ADAPT-2 | Edge-to-edge включён; контент учитывает insets (status/navigation bar, IME, cutout) | **[B]** | прогон + ревью `WindowInsets` |
| ADAPT-3 | Поддержаны тёмная тема и (где применимо) dynamic color (Material You) | **[M]** | прогон light/dark |
| ADAPT-4 | Портрет/ландшафт, multi-window и складывание — без потери состояния | **[M]** | ручной прогон |
| ADAPT-5 | Разные плотности: vector/9-patch, нет растягивания растров | **[N]** | ревью ресурсов |
| ADAPT-6 | Учтены вырезы/закруглённые углы/taskbar; критичные элементы не перекрываются | **[N]** | прогон на устройствах с cutout |

**Антипаттерны**
- Жёсткие размеры под телефон; контент уезжает под status/nav bar.
- Состояние теряется при resize/повороте; растягивание растрового изображения.

### 12. ★ Состояния экрана (STATE)

**Принцип.** Каждый экран явно обрабатывает loading / empty / error / success — единым
паттерном, не ad-hoc в каждом Composable.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| STATE-1 | Каждый экран обрабатывает loading/empty/error/success (не «пустой белый экран») | **[B]** | ревью UiState + прогон |
| STATE-2 | Ошибки имеют понятное сообщение и retry; нет «сырых» Exception/HTTP-кодов в UI | **[B]** | ревью обработки ошибок |
| STATE-3 | Долгие операции показывают прогресс/skeleton; нет «зависшего» UI без обратной связи | **[M]** | прогон с медленной сетью |
| STATE-4 | Деструктивные действия подтверждаются; есть undo где уместно | **[M]** | ревью |
| STATE-5 | Формы: инлайн-валидация, сохранение ввода при ошибке, понятные сообщения у полей | **[M]** | ревью |

**Антипаттерны**
- Пустой белый экран на ошибке/загрузке; вечный спиннер.
- Сырой стектрейс/HTTP-код в UI; деструктив без подтверждения.

### 13. ★ Сетевая устойчивость и offline-first (RESIL)

**Принцип.** Сеть ненадёжна; UI предсказуем при ошибке/задержке/оффлайне; БД — source of truth.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| RESIL-1 | Приложение не падает и информирует при отсутствии сети; критичные данные доступны из кэша | **[B]** | прогон в airplane mode |
| RESIL-2 | Стратегия кэширования определена (source of truth — БД; сеть обновляет кэш) | **[M]** | ревью репозиториев |
| RESIL-3 | Запросы имеют таймауты и retry с backoff; нет бесконечных ретраев; мутации не ретраятся вслепую | **[M]** | ревью OkHttp/Retrofit |
| RESIL-4 | Синхронизация устойчива к конфликтам/частичным сбоям; повторные запуски идемпотентны | **[M]** | ревью sync + тест |
| RESIL-5 | Пользователю виден статус синхронизации/время последнего обновления (для offline-first) | **[N]** | ревью UI |

**Антипаттерны**
- Краш/белый экран в airplane mode; бесконечный спиннер при упавшем запросе.
- Слепой retry неидемпотентной мутации → дубли; синхронизация теряет данные при сбое.

### 14. ★ Слой данных: персистентность и миграции (DATA)

**Принцип.** Схема версионируется; миграции явные и протестированы — это частый источник прод-крэшей при обновлении.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| DATA-1 | Миграции Room явные и протестированы; destructive-миграция запрещена в проде | **[B]** | `MigrationTestHelper` + ревью |
| DATA-2 | Схема БД версионируется; экспорт схемы (`exportSchema=true`) хранится в VCS | **[M]** | наличие schema-json |
| DATA-3 | Запись в БД атомарна где нужно (`@Transaction`); нет частично записанных состояний | **[M]** | ревью DAO |
| DATA-4 | Миграции/дефолты DataStore (в т.ч. Proto) безопасны при обновлении приложения | **[M]** | ревью + тест обновления |
| DATA-5 | Целостность данных: внешние ключи/индексы где нужно; нет дублей | **[N]** | ревью схемы |

**Антипаттерны**
- `fallbackToDestructiveMigration()` в проде; миграции без теста.
- Многошаговая запись без `@Transaction`; небезопасная миграция DataStore-схемы.

### 15. ★ Runtime-разрешения, приватность и комплаенс (PRIV)

**Принцип.** Минимум разрешений, запрос в момент использования, прозрачность собираемых данных.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| PRIV-1 | Разрешения запрашиваются в момент использования, с rationale; отказ обрабатывается без краша и с деградацией | **[B]** | прогон allow/deny |
| PRIV-2 | Запрашивается минимум разрешений; Photo Picker/scoped storage вместо широкого доступа | **[M]** | ревью манифеста |
| PRIV-3 | `POST_NOTIFICATIONS` (API 33+) запрашивается корректно | **[M]** | прогон на API 33+ |
| PRIV-4 | Play Data Safety form актуальна и соответствует реально собираемым данным | **[B]** | сверка с кодом сети/аналитики |
| PRIV-5 | Есть механизм удаления данных/аккаунта (требование Play); учтены GDPR/согласия | **[M]** | ревью + флоу удаления |
| PRIV-6 | Сбор аналитики/идентификаторов прозрачен; advertising ID/трекинг — только с согласия | **[M]** | ревью аналитики |

**Антипаттерны**
- Запрос всех разрешений на старте; краш/тупик при отказе.
- Широкий доступ к storage вместо Photo Picker; Data Safety form расходится с кодом.

### 16. ★ Совместимость и политика платформы (COMPAT)

**Принцип.** Приложение корректно на всём диапазоне поддерживаемых API; behavior changes учтены.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| COMPAT-1 | `targetSdk` соответствует требованиям Play; `minSdk` обоснован | **[B]** | build.gradle + Play policy |
| COMPAT-2 | API-level guarding: вызовы новых API защищены проверкой версии; нет крашей на min-устройствах | **[B]** | lint `NewApi` + прогон на minSdk |
| COMPAT-3 | Учтены behavior changes: predictive back, foreground service types (API 34), edge-to-edge enforcement (API 35), 16 KB page size (API 35+) | **[M]** | ревью + прогон на новых API |
| COMPAT-4 | Deprecated API не вносятся в новый код; для существующих есть план миграции | **[M]** | lint `Deprecation` |
| COMPAT-5 | App Links верифицированы (`autoVerify`) там, где есть веб-домен | **[M]** | `adb shell pm verify-app-links` |

**Антипаттерны**
- Вызов нового API без `Build.VERSION` guard → краш на minSdk.
- Устаревший `targetSdk`; deprecated API в новом коде; неверифицированные App Links.

### 17. ★ Наблюдаемость в проде (OPS)

**Принцип.** Прод-инцидент невидим без телеметрии; ошибки, метрики и контекст собираются и коррелируются — без PII.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| OPS-1 | Подключён crash reporting (Crashlytics/Sentry); mapping R8 загружается для деобфускации | **[B]** | дашборд + CI upload |
| OPS-2 | Crash-free users ≥ 99.5% (порог по проекту) | **[B]** | Play Vitals / дашборд |
| OPS-3 | ANR-rate в зоне Play Vitals «good»; ANR мониторятся | **[B]** | Play Vitals |
| OPS-4 | Логирование структурировано и уровнями; в проде нет шумных/PII-логов (SEC-3) | **[M]** | ревью логгера |
| OPS-5 | Ключевые продуктовые события покрыты аналитикой; есть funnel для критичных флоу | **[M]** | ревью трекинга |
| OPS-6 | Production performance monitoring (startup, фризы, сеть), а не только локальный бенч | **[N]** | Firebase Performance / трейсинг |

**Антипаттерны**
- Краши видны только в logcat пользователя; mapping не загружен → нечитаемые стектрейсы.
- PII/токен в событиях аналитики; нет прод-мониторинга производительности.

### 18. ★ Релизная инженерия и дистрибуция (REL)

**Принцип.** Сборка воспроизводима, подписана и быстро откатываема; релиз трассируется до коммита.

| ID | Критерий | Уровень | Порог / проверка |
|----|----------|---------|------------------|
| REL-1 | Конфиг подписи безопасен (keystore не в VCS, в CI secrets); release подписан | **[B]** | ревью signingConfig + `.gitignore` |
| REL-2 | Сборка — App Bundle (AAB); `versionCode`/`versionName` управляются консистентно | **[M]** | build.gradle / CI |
| REL-3 | Раскатка поэтапная (staged rollout); есть kill-switch/feature flags/remote config для отката | **[M]** | конфиг релиза |
| REL-4 | Релиз только из защищённой ветки/по тегу; артефакт воспроизводим из CI | **[M]** | CI pipeline |
| REL-5 | Play Integrity / App Integrity подключён там, где требуется защита | **[N]** | ревью |

**Антипаттерны**
- Keystore закоммичен в VCS; ручной деплой.
- Вечные feature-флаги без срока; прод-краш невозможно связать с релизом.

## 19. Тестирование (сквозное, Definition of Done) (TEST)

**Принцип.** Тест — в первую очередь проверка дизайна. Пирамида: много unit, меньше
integration, точечно UI/e2e критических флоу. Стек: JUnit, MockK, Turbine, Compose UI
Test, Hilt Testing, Macrobenchmark, Paparazzi/Roborazzi.

| ID | Критерий | Уровень |
|----|----------|---------|
| TEST-1 | Каждый новый use case покрыт тестом (unit для domain/VM, UI/integration для экрана) | **[B]** |
| TEST-2 | ViewModel-тесты проверяют переходы `UiState` через Turbine (Loading→Success/Error) | **[B]** |
| TEST-3 | Domain/UseCase — чистые unit-тесты без Android-зависимостей | **[M]** |
| TEST-4 | Критичные UI-флоу (авторизация, ключевые формы, работа с камерой) — Compose UI Test | **[M]** |
| TEST-5 | Интеграция с реальной БД/бэкендом не мокается там, где есть реальная схема | **[M]** |
| TEST-6 | Перф-сценарии (startup, скролл основных списков) — Macrobenchmark в CI (хотя бы nightly) | **[N]** |
| TEST-7 | LeakCanary гоняется в debug на ключевых сценариях перед релизом | **[M]** |
| TEST-8 | Screenshot/snapshot-тесты ключевых Composable (Paparazzi/Roborazzi); UI-регрессии ловятся в CI | **[N]** |
| TEST-9 | Флаки-тесты отслеживаются и чинятся/карантинятся; CI не «зелёный» за счёт слепых ретраев | **[M]** |
| TEST-10 | Соблюдается порог покрытия на diff; критичные edge-cases (ошибки, пустые, границы) покрыты | **[M]** |
| TEST-11 | Контракт-тесты / проверка соответствия API-схеме там, где возможно (а не только моки) | **[N]** |

**Антипаттерны**
- Логика в Composable → тест требует поднимать весь UI ради проверки расчёта.
- Реальное время/рандом без фиксации → флакающие тесты.
- «Зелёный» CI за счёт автоматических ретраев флаки-тестов.

## Definition of Done

- Критерий считается выполненным **только при зелёной проверке**; «частично» не
  засчитывается. Зрелость **3** означает, что нарушение ловит машина (lint/CI/тест), а
  не ревьюер.
- Для каждого измерения проект фиксирует **baseline** (0–4) и конкретные пороги — в
  overlay `android-quality-criteria.<project>.md`, не в этом шаблоне.
- Применимость ★-измерений и слоёв фиксируется явно: **«применимо / N/A +
  обоснование»**. Примеры N/A: I18N — для одноязычного внутреннего приложения; ADAPT —
  частично, если поддержка только телефонов; RESIL offline — если приложение требует
  постоянной сети по согласованному решению.
- **Релиз-блокеры** (минимум зелёного перед раскаткой на 100%): Часть 0 (бюджеты),
  отсутствие открытых **[B]**, ARCH-1..3, SEC-1..4, MEM-1/MEM-5, PERF-3..5, A11Y
  критичные (0 нарушений Accessibility Scanner), STATE-1/STATE-2, DATA-1, OPS-1..3,
  REL-1, TEST критичных флоу.
- Цель миграции каждого измерения — **сдвиг проверок влево**: от ручного ревью к
  CI-enforcement.

## Как применять к конкретному проекту

1. Завести **overlay** `android-quality-criteria.<project>.md` рядом: для каждого
   измерения честно проставить зрелость (0–4), главный долг и привязку ID
   (`ARCH-1`/`PERF-3`/…) к месту в коде, модулям и спринтам. Шаблон остаётся чистым.
2. Зафиксировать **baseline** и выбрать **рычаги**: измерения, фикс которых на уровне
   `:core`/общих компонентов даёт массовый эффект (обычно — a11y, производительность,
   тестируемость, изоляция модулей, сетевая устойчивость).
3. Перевести релевантные критерии в **acceptance-условия** спринта, ссылаясь по ID.
4. Поднимать enforcement: зрелость **3** = нарушение ловит машина. Направление
   неизменно — **сдвигать проверку влево**, от ревью к CI.
