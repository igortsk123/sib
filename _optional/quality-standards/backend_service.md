# Критерии качества веб-сервиса — универсальный шаблон

> **Что это.** Переиспользуемая целевая планка качества для **любого** бэкенд/веб-сервиса:
> эталон, к которому идёт команда, а **не** описание текущего состояния кода. Документ
> намеренно очищен от привязки к конкретному проекту, стеку и языку — конкретику (имена
> файлов, схем, эндпоинтов, спринтов, флагов) проект добавляет отдельным оверлеем-маппингом.
>
> **Как использовать.** Скопируйте файл в проект, для каждого критерия задайте конкретный
> порог/проверку под свой контекст (часть порогов даны как разумные дефолты, их надо
> подтвердить с продуктом/SRE), и заведите оверлей `quality_criteria.<service>.md`, где каждый
> ID критерия привязан к месту в коде и к этапу плана.
>
> **Формат.** Каждый критерий — **цель** + **измеримый порог/проверка**. Без порога критерий
> нельзя ни принять, ни отклонить, поэтому пороги обязательны. Критерий без зелёной проверки
> считается невыполненным — «частично» не засчитывается.
>
> **Каноны, на которые опираемся:**
> - Безопасность: OWASP ASVS, OWASP API Security Top 10 (2023), least privilege, defense-in-depth.
> - Конфигурация/процессы: 12-Factor App.
> - Надёжность: Google SRE (SLI/SLO, error budget), паттерны Release It! (timeout, circuit breaker, bulkhead).
> - Микросервисы: S. Newman «Building Microservices», C. Richardson «Microservices Patterns»
>   (database-per-service, API gateway/adapter, saga, transactional outbox, CDC).
> - Протоколы: семантика HTTP (RFC 9110), идемпотентность, явное версионирование, consumer-driven contracts.
> - Наблюдаемость: three pillars (logs/metrics/traces), OpenTelemetry, RED/USE-методики.
> - Данные: нормализация + осознанная денормализация, индексирование под план, prepared statements, EXPLAIN-driven; управление данными — DAMA-DMBOK.
> - Приватность/комплаенс: GDPR/CCPA, классификация данных, минимизация, retention/erasure; контроли SOC 2 / ISO 27001 / PCI-DSS по применимости.
> - Поставка: DORA-метрики (deploy frequency, lead time, MTTR, change-fail rate), trunk-based/GitOps, целостность сборки (SLSA).
> - Код: Clean Code, цикломатическая/когнитивная сложность (McCabe), SOLID (DIP/LSP), program-to-interface, composition over inheritance, GoF — по месту, без over-engineering.
> - Интернационализация: Unicode/UTF-8, ISO 8601, IANA TZ, ICU/CLDR.
> - Стоимость: FinOps Foundation (видимость, оптимизация, аккаунтабилити).
> - Доступность интерфейсов: WCAG 2.2 (для сервисов с UI-поверхностью).
> - Кросс-сервисные данные: **Event-Carried State Transfer** (M. Fowler) — внешние данные
>   захватываются в момент записи бизнес-события; хранится минимально необходимый срез.

> **Отличие от исходного отраслевого чек-листа.** К классическому набору (Security, Performance,
> Protocol, Isolation, Scalability, Testability, Observability, Fault-tolerance, Resources, Design
> patterns) добавлены ранее упущенные сквозные аспекты, помеченные ★: Сопровождаемость/здоровье
> кода, Параллелизм и согласованность, Управление данными/приватность/комплаенс, Резервирование и
> аварийное восстановление, Релиз-инжиниринг и поставка, Интернационализация, Стоимость и
> эффективность, Удобство API для потребителя (DX), Документация и эксплуатационная поддержка,
> Доступность интерфейсов, Переносимость и паритет сред.

## Часть 0. SLO как корень критериев

Best practice (SRE): сначала фиксируем пользовательские SLO, остальные критерии — производные.
Конкретные цифры подтверждаются с продуктом и владельцем надёжности.

| SLI | Целевой SLO (дефолт, уточнить) |
|-----|-------------------------------|
| Доступность ключевых операций (успех / все валидные запросы) | ≥ 99.9% (для критичных — ≥ 99.95%) |
| Латентность чтений p99 | ≤ согласованный бюджет (напр. ≤ 200 мс) |
| Латентность мутаций p99 | ≤ согласованный бюджет |
| Корректность контракта (consumer-driven / golden-diff) | 100% на ключевых эндпоинтах |
| Свежесть производных/реплицируемых данных (лаг p99) | ≤ согласованный SLO (напр. ≤ 30 с) |
| Доля деградаций при здоровом сервисе | < 0.1% запросов |
| Долговечность данных (durability) | ≥ согласованная (напр. 99.999999999%) |

**Error budget** из этих SLO управляет темпом раскатки изменений (canary/флаги) и приоритетом
надёжности против фич. Исчерпание бюджета замораживает рискованные релизы.

## Часть I. Сквозные аспекты

### 1. Безопасность (SEC) — OWASP ASVS / API Security Top 10, least privilege

| # | Критерий | Обоснование | Порог / проверка |
|---|----------|-------------|------------------|
| SEC-1 | Минимальная поверхность атаки: открыты только необходимые порты/интерфейсы, внутренние — не наружу | reduce attack surface | скан портов; нет лишних listen |
| SEC-2 | Аутентификация каждого запроса; нет «доверия по сети» (zero-trust): проверяемый секрет/JWT/mTLS | API2:2023 Broken Authentication | 401 без валидного credential; цель — mTLS/JWT |
| SEC-3 | Авторизация на уровне объекта и функции (ACL/RBAC применяются в домене, не только на транспорте) | API1/API5:2023 BOLA/BFLA | приватные данные не утекают; тест авторизации |
| SEC-4 | Секреты только из env/secret-store, ротация; нет секретов в коде/образе/логах (12-factor III) | — | scan репозитория и логов = 0 секретов |
| SEC-5 | Защита от инъекций: параметризованные запросы/prepared statements; весь ввод валидируется и нормализуется | injection / API8 | статический аудит: 0 конкатенаций ввода в запросы |
| SEC-6 | Least privilege на уровне хранилища: сервисная роль имеет гранты только на свои объекты, без admin/DDL в рантайме | privilege separation | ревизия грантов |
| SEC-7 | Минимизация данных: хранится и отдаётся только необходимое (data minimization) | API3 excessive data exposure | ревизия схем и ответов |
| SEC-8 | Rate limiting / квоты на публичных и дорогих путях | API4:2023 unrestricted resource consumption | контракт-тест лимита |
| SEC-9 | Валидация и ограничение размера входа (body size, длины коллекций, диапазоны, типы) | API6 / mass assignment | fuzz/negative-тесты |
| SEC-10 | Аудит чувствительных действий с корреляцией (`trace_id`), без PII/токенов в логах | logging & monitoring | ревизия аудит-лога |
| SEC-11 | Актуальные зависимости, SCA/SBOM, без known-CVE в проде; контроль цепочки поставки | supply chain | CI-скан (SCA/контейнерный скан) |
| SEC-12 | Шифрование в транзите (TLS) и at-rest для чувствительных данных; управление ключами | confidentiality | проверка TLS-конфига и шифрования хранилища |
| SEC-13 | Безопасные заголовки/ответы (CORS осознан, нет лишней разглашающей информации в ошибках/заголовках) | hardening | скан заголовков |

### 2. Производительность (PERF) — RED-метрики, EXPLAIN-driven, кеширование

| # | Критерий | Порог / метрика |
|---|----------|-----------------|
| PERF-1 | Накладные расходы транспорта (хоп до сервиса/БД) измерены и в бюджете | бенч p50/p99 |
| PERF-2 | p99 ключевых эндпоинтов в рамках SLO (Часть 0); регресс ловится | dark-launch / нагрузочное сравнение |
| PERF-3 | Переиспользование ресурсов: пул соединений/клиентских сессий на процесс, без создания на запрос | code review + метрика коннектов |
| PERF-4 | Кеширование редко меняющихся данных (cache-aside + TTL + инвалидация по событию) | hit rate > целевого |
| PERF-5 | Отсутствие N+1; батч-загрузка / dataloader-паттерн где уместно | счётчик запросов на эндпоинт |
| PERF-6 | Бюджеты таймаутов и дедлайны на всю цепочку (deadline propagation); без слепых ретраев на неидемпотентных | code review |
| PERF-7 | Нагрузочный тест на целевой RPS: деградация плавная, нет cliff | load test перед 100% трафика |
| PERF-8 | Пагинация обязательна для коллекций (`per_page` ≤ лимит); keyset для глубоких списков | проверка планов запросов |

### 3. Непротиворечивость протоколов (PROTO) — HTTP-семантика, contract-first, версионирование

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| PROTO-1 | Consumer-driven contract tests: ответ соответствует согласованному контракту побайтово/по схеме | golden-diff = 0 |
| PROTO-2 | Единый стабильный envelope/формат ответа, документированный (OpenAPI/JSON Schema) | схема в репозитории |
| PROTO-3 | Корректная HTTP-семантика (RFC 9110): статусы, методы, идемпотентность GET/PUT/DELETE | контракт-тест статусов |
| PROTO-4 | Идемпотентность мутаций: idempotency-key, безопасный повтор без дублей | тест дедупликации |
| PROTO-5 | Явное версионирование (`/v1/`); backward-compatible эволюция; breaking — через депрекацию (410 Gone + срок) | политика версионирования |
| PROTO-6 | Симметричная сериализация (round-trip без потерь), стабильные типы (даты/деньги/null) | property-тест round-trip |
| PROTO-7 | Деградационные/дефолтные ответы по форме идентичны реальным | тест каждого fallback-объекта |
| PROTO-8 | Единый машиночитаемый формат ошибок (тип, код, trace_id), напр. RFC 9457 problem+json | контракт ошибок |

### 4. Изоляция (ISO) — database-per-service, bulkhead, loose coupling

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| ISO-1 | Database-per-service: нет межсервисных FK на чужие данные; локально хранятся бизнес-ключи, не чужие PK | ревизия DDL |
| ISO-2 | Сервис — единственный писатель своих данных; чужие данные только read-only (реплика/события) | ревизия грантов |
| ISO-3 | Bulkhead: отказ сервиса изолирован — потребитель не получает каскад 5xx (fallback + breaker) | chaos-тест |
| ISO-4 | Изоляция фоновых задач: падение воркера/репликатора не влияет на обслуживание чтений (отдельный пул/процесс) | chaos-тест |
| ISO-5 | Деплой/рестарт без каскада: breaker `open` → мгновенный fallback; rolling deploy | проверка деплоя |
| ISO-6 | Слабая связанность по времени: обратимый откат флагом без миграций (strangler fig) | проверка отката |
| ISO-7 | Изоляция конфигов/секретов сервиса от соседей (отдельные env/роли) | ревизия |

### 5. Масштабируемость (SCALE) — stateless, 12-factor, шардирование данных

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| SCALE-1 | Stateless-процессы (12-factor VI): состояние в БД/кеше, горизонтальное масштабирование репликами | нет межзапросного мутабельного state |
| SCALE-2 | Ограниченный настраиваемый пул ресурсов; деградация под нагрузкой контролируема (backpressure, лимит очереди) | конфиг + метрика насыщения |
| SCALE-3 | Партиционирование/шардирование больших таблиц: рост и миграция без глобальных локов | план партиционирования |
| SCALE-4 | Инкрементальная обработка/репликация по курсору; полный resync — по расписанию, не на тик | проверка курсора |
| SCALE-5 | Предсказуемость при N процессах (per-process кеш/breaker документированы); нет глобального узкого места | нагрузочный тест |
| SCALE-6 | Read-масштабирование: реплики чтения БД без изменения кода | архитектурная проверка |
| SCALE-7 | Автоскейл по нагрузке с верхними лимитами; без cold-start cliff на старте инстанса | проверка autoscaling-политики |

### 6. Тестируемость (TEST) — test pyramid, hexagonal, детерминизм

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| TEST-1 | Пирамида тестов: много unit, контрактные на границах, мало e2e; golden-тесты ключевых эндпоинтов в CI-gate | покрытие критичных путей |
| TEST-2 | Гексагональная архитектура: домен тестируется без HTTP/БД через порты/моки | unit домена без сети |
| TEST-3 | Chaos автоматизирован (сервис лёг/лаг/500/деградация зависимостей) | chaos-suite в CI |
| TEST-4 | Тесты на реальной схеме данных (партиции, индексы, проекции) | интеграционные SQL-тесты |
| TEST-5 | Каждый деградационный/дефолтный объект покрыт тестом формы | тест fallback-фабрик |
| TEST-6 | Детерминизм: фиксируемые clock/seed, изоляция тестов, ephemeral-окружение (testcontainers-стиль) | CI без флака |
| TEST-7 | Граничная логика (rehydrate контекста, парсинг) тестируется изолированно с фикстурами | unit граничного слоя |
| TEST-8 | Измеримое покрытие критичных путей с порогом в gate; мутационное тестирование на ядре (опционально) | coverage-gate |

### 7. Наблюдаемость (OPS) — three pillars, OTel, 12-factor

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| OPS-1 | Distributed tracing сквозной (потребитель→сервис→хранилище), W3C `traceparent`/OTel | трейсы видны в бэкенде |
| OPS-2 | RED-метрики (Rate/Errors/Duration) на эндпоинт + доменные (деградации, breaker state, лаг) | дашборд |
| OPS-3 | SLO-алерты на симптомы, не на причины (рост ошибок/латентности, open breaker, лаг); alert fatigue минимизирован | алерт-правила |
| OPS-4 | Конфиг строго из env (12-factor III); один артефакт на все среды | ревизия |
| OPS-5 | Health + readiness + liveness (включая статус зависимостей и лаг) для оркестратора | проба |
| OPS-6 | Структурированные логи (JSON), уровни, корреляция по `trace_id`, без PII | ревизия |
| OPS-7 | Наблюдаемость очередей/outbox/репликации: глубина, возраст старейшего, ошибки повтора | дашборд |
| OPS-8 | Версия/коммит/конфиг в `/health` и логах старта (provenance) | проверка |

### 8. Устойчивость к ошибкам (FT) — Release It!: timeout, circuit breaker, bulkhead, retry-with-backoff

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| FT-1 | Таймаут на каждый внешний вызов; превышение → fallback, не зависание | проверка таймаутов |
| FT-2 | Circuit breaker: N сбоев → `open` на T, мгновенный fallback, `half-open` проба | latency не растёт при «убитой» зависимости |
| FT-3 | Graceful degradation: мутации при недоступности → outbox + идемпотентный повтор (at-least-once + дедуп) | тест деградации |
| FT-4 | Fallback-кеш отдаёт последнюю успешную копию, не пустоту | тест fallback-кеша |
| FT-5 | Контроль дрейфа реплик/производных данных: курсор + плановый resync + алерт на лаг | проверка курсора |
| FT-6 | Согласованность: критичное в моменте (цена/лимит/остаток) — sync-вызов к owner, не реплика; saga/compensation для распределённых мутаций | политика консистентности |
| FT-7 | Retry только для идемпотентных операций, экспоненциальная задержка + jitter; нет retry-штормов | code review |
| FT-8 | Ни один сбой зависимости (кеш/реплика/воркер/частичный отказ БД) не даёт 5xx наружу | chaos-матрица |
| FT-9 | Корректный backpressure/перегрузка: load shedding вместо коллапса | load test |

### 9. Потребление ресурсов (RES) — bounded resources, streaming, no leaks

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| RES-1 | Bounded everything: пагинация, без загрузки всей коллекции в память | проверка горячих путей |
| RES-2 | Кеши ограничены по размеру + TTL + eviction (LRU); не растут безгранично | конфиг maxmemory/eviction |
| RES-3 | Обработка/репликация батчами по курсору, без вычитки всей таблицы-источника | размер батча конфигурируем |
| RES-4 | Нет утечек: ресурсы управляются жизненным циклом приложения; нет per-request кеша, переживающего запрос | профиль памяти стабилен |
| RES-5 | RSS/heap под устойчивой нагрузкой стабилен (нет монотонного роста за ≥1 ч прогона) | leak-тест |
| RES-6 | Большие ответы стримятся; курсорная вычитка из БД для крупных выборок | проверка горячих путей |
| RES-7 | CPU/файловые дескрипторы/сокеты ограничены и не текут (нет роста FD за прогон) | мониторинг FD |

### 10. Паттерны проектирования (PAT) — уместность, а не «ради паттерна»

> Паттерн оправдан, только если **снижает** связанность/дублирование или открывает точку
> расширения. Над-инженерия — антипаттерн и отклоняется на ревью наравне с отсутствием паттерна.
> Базис: program-to-interface, composition over inheritance, SOLID (особенно DIP и LSP).

| # | Критерий | Обоснование | Порог / проверка |
|---|----------|-------------|------------------|
| PAT-1 | **Dependency Injection**: зависимости приходят в конструктор; домен их не достаёт сам (нет service-locator/глобальных синглтонов) | DIP, тестируемость | сборка в DI-слое; в домене нет прямого доступа к БД/конфигу |
| PAT-2 | Корректные scope провайдеров: stateless-ресурсы — singleton; per-request — factory; конфиг — с env-override | управление жизненным циклом | ревизия DI-конфигурации |
| PAT-3 | Граница модулей/контейнеров повторяет bounded-context; внешние зависимости — отдельный модуль | модульность, изоляция [ISO] | контейнер на контекст |
| PAT-4 | **Adapter**: клиент адаптирует транспорт под доменный интерфейс; репозитории адаптируют хранилище под порты | сокрытие транспорта/хранилища | смена транспорта не трогает домен |
| PAT-5 | **Strategy** для семейств взаимозаменяемых алгоритмов: общий интерфейс, выбор по коду/конфигу, без `if/elif`-каскадов по типу | open/closed | новая стратегия = новый класс |
| PAT-6 | **Полиморфизм вместо ветвления по типу**: null-object (LSP), отсутствие `isinstance`/`type()`-диспетчеризации в домене | LSP | grep на type-диспетчеризацию в домене обоснован/отсутствует |
| PAT-7 | **Fluent Builder** для составных запросов: пошаговая сборка, единый терминальный `build()`; параметры связываемые, не строковая склейка | управляемая сложность, anti-injection | билдер покрыт тестами на комбинации |
| PAT-8 | Паттерн уместен и не избыточен: нет фабрик/обёрток на единственной реализации; абстракция при ≥2 реализациях или явной точке расширения | YAGNI / KISS | ревью отклоняет паттерн без второго пользователя |
| PAT-9 | Программирование на интерфейс: домен зависит от портов, реализации внедряются — заменяемы на фейки в тестах | DIP [TEST-2] | unit-тесты домена с фейковыми репозиториями |

### 11. ★ Сопровождаемость / здоровье кода (MAINT) — Clean Code, метрики сложности

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| MAINT-1 | Ограничена цикломатическая/когнитивная сложность функций (порог в линтере) | напр. cyclomatic ≤ 10 на функцию; CI-gate |
| MAINT-2 | Дублирование кода под порогом; копипаст вынесен в общий модуль | напр. duplication < 3–5%; отчёт анализатора |
| MAINT-3 | Нет мёртвого/недостижимого кода и закомментированных блоков; неиспользуемое удаляется | dead-code линтер = 0 |
| MAINT-4 | Высокая связность / низкое сцепление модулей; зависимости направлены к стабильным абстракциям | ревизия графа зависимостей |
| MAINT-5 | Форматирование и стиль автоматизированы (formatter + linter) и проходят в gate | CI-gate, 0 ошибок линтера |
| MAINT-6 | Техдолг отслеживается: `TODO`/`FIXME` связаны с тикетом; есть реестр долга с приоритетом | ревизия; нет «висячих» TODO |
| MAINT-7 | Имена выражают намерение; комментарии объясняют WHY (неочевидный инвариант/обход), не WHAT | code review |
| MAINT-8 | Размер единиц управляем (функции/файлы/классы не разрастаются сверх согласованного порога) | метрика размера |
| MAINT-9 | Обязательный code review с чек-листом; ни один мёрж в основную ветку без ревью | branch protection |
| MAINT-10 | Бюджет maintainability в CI (например, индекс сопровождаемости/quality gate инструмента) не деградирует | quality-gate тренд |

### 12. ★ Параллелизм и согласованность (CONC) — race-free, корректные гарантии доставки

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| CONC-1 | Нет гонок данных на разделяемом состоянии; критические секции защищены или состояние неизменяемо | анализ + стресс-тест конкурентности |
| CONC-2 | Конкурентные обновления используют оптимистичную блокировку (version/ETag) или явные локи; lost-update исключён | тест конкурентной записи |
| CONC-3 | Нет дедлоков: согласованный порядок захвата ресурсов/локов; таймауты на блокировки | нагрузочный/стресс-тест |
| CONC-4 | Явно выбраны и задокументированы гарантии доставки (at-least-once + дедуп / exactly-once-эффект) | ADR + тест дедупликации [PROTO-4] |
| CONC-5 | Уровень изоляции транзакций осознан; нет неожиданных аномалий (phantom/write-skew) на критичных путях | ревизия транзакций |
| CONC-6 | Идемпотентные обработчики событий/сообщений; повтор сообщения не ломает инвариант | тест повторной доставки |
| CONC-7 | Корректная отмена/таймаут асинхронных операций без утечки ресурсов и «осиротевших» задач | тест отмены |

### 13. ★ Управление данными, приватность, комплаенс (DATA) — DAMA, GDPR/CCPA

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| DATA-1 | Инвентарь и классификация данных: PII/чувствительное помечено, известно где хранится и зачем | data map актуален |
| DATA-2 | Политики хранения и удаления (retention/purge) заданы и автоматизированы для каждого класса данных | джоба очистки + проверка |
| DATA-3 | Право на удаление/экспорт (GDPR/CCPA): механизм erasure/portability реализуем и протестирован | тест erasure-флоу |
| DATA-4 | Минимизация и псевдонимизация/анонимизация PII там, где полные данные не нужны | ревизия схем |
| DATA-5 | Целостность данных: валидация на входе + инвариантные проверки (constraints/checks), сверка консистентности | constraints в схеме; reconciliation-джоба |
| DATA-6 | Аудиторский след изменений чувствительных данных (кто/что/когда), tamper-evident | ревизия аудит-таблиц |
| DATA-7 | Резидентность/локация данных и трансграничная передача соответствуют требованиям | ревизия размещения |
| DATA-8 | Лицензии зависимостей совместимы с продуктом (OSS-license compliance), запрещённые лицензии заблокированы | license-скан в CI |
| DATA-9 | Применимые контроли (SOC 2 / ISO 27001 / PCI-DSS и т. п.) замаплены на критерии и проверяются | матрица контролей |
| DATA-10 | Схема данных версионируется; обратная совместимость данных и эволюция документированы (schema registry для событий) | реестр схем |

### 14. ★ Резервирование и аварийное восстановление (DR) — RTO/RPO, BCP

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| DR-1 | Заданы целевые RTO и RPO для сервиса и согласованы с продуктом | документ + согласование |
| DR-2 | Регулярные резервные копии по стратегии 3-2-1; копии зашифрованы | проверка расписания и шифрования |
| DR-3 | Восстановление **проверяется реальным restore-drill**, а не только наличием бэкапа | периодический drill, лог результата |
| DR-4 | Point-in-time recovery доступен для критичных хранилищ в пределах RPO | тест PITR |
| DR-5 | Отказоустойчивость инфраструктуры: multi-AZ (и при необходимости multi-region) failover | тест failover |
| DR-6 | Runbook аварийного восстановления актуален и исполним без «племенных знаний» | ревизия + учебная тревога |
| DR-7 | Деградация без потери данных: незавершённые мутации не теряются (durable outbox/журнал) при сбое | chaos + проверка журнала |
| DR-8 | Бэкапы и DR покрывают не только БД, но и конфиг/секреты/инфраструктуру (IaC в VCS) | ревизия охвата |

### 15. ★ Релиз-инжиниринг и поставка (REL) — CI/CD, DORA, целостность сборки

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| REL-1 | Полностью автоматизированный пайплайн: build → test → security-scan → deploy, без ручных шагов | пайплайн в VCS |
| REL-2 | Воспроизводимая сборка из закреплённых зависимостей (lock-файлы); один артефакт продвигается по средам | reproducible build |
| REL-3 | Целостность артефактов: подпись и provenance (SLSA-уровень), проверка перед деплоем | подпись + verify в CD |
| REL-4 | Progressive delivery: canary/blue-green/feature-flags с автоматическим rollback по SLO | проверка стратегии выката |
| REL-5 | Миграции БД безопасны и обратимы (expand/contract, zero-downtime), прогоняются в гейте | гейт миграций |
| REL-6 | DORA-метрики измеряются и в целевой полосе (deploy frequency, lead time, MTTR, change-fail rate) | дашборд DORA |
| REL-7 | Любое изменение трассируемо: коммит → артефакт → среда (provenance в `/health` [OPS-8]) | трассировка релиза |
| REL-8 | Откат проверен и быстр (rollback drill); «вперёд-фикс» не единственный путь | rollback-тест |

## Часть II. Критерии по слоям

### A. Транспорт / API-слой — best practice: thin transport, fail-safe

| # | Критерий |
|---|----------|
| API-1 | Тонкий транспорт: роут только (де)сериализует/валидирует/делегирует; нулевая доменная логика в хендлерах |
| API-2 | Явный middleware-конвейер: auth → rehydrate контекста → trace → handler → единый error-handler |
| API-3 | Fail-safe ошибки: необработанное исключение → 500 + `trace_id`, без утечки стектрейса/внутренностей (RFC 9457) |
| API-4 | Graceful shutdown (SIGTERM): drain активных запросов, закрыть session/pool, зафиксировать курсоры (12-factor IX) |
| API-5 | Защита входа: max body size, таймаут чтения тела, лимиты на bulk, валидация схемы запроса |
| API-6 | Неблокирующий обработчик: I/O асинхронен/не блокирует воркеры; CPU-тяжёлое — в отдельный пул/executor |
| API-7 | Версионированная декларативная таблица роутов — зеркало методов сервиса |
| API-8 | Каждый запрос логируется (trace_id, method, latency, status); access-log пригоден для RED-метрик |
| API-9 | Корректные таймауты сервера (keepalive, request) и обратное давление при насыщении пула |

### B. Доменный слой — best practice: rich domain, ports & adapters

| # | Критерий |
|---|----------|
| DOM-1 | Независимость от транспорта: домен принимает свой контекст-объект, не транспортный `Request` (dependency inversion) |
| DOM-2 | Чистота границ: мёртвый/легаси-код не переносится — удаляется до выноса |
| DOM-3 | Конфиг вместо данных там, где уместно: метаданные/справочники — статический конфиг, не запрос на лету |
| DOM-4 | Стабильный публичный контракт: публичные методы = API; изменение сигнатуры = новая версия |
| DOM-5 | Инкапсуляция инвариантов: бизнес-правила внутри домена; джойны локальны, не межсервисны |
| DOM-6 | Самодостаточность: доменные стратегии живут в коде сервиса, не импортируются из чужих модулей |
| DOM-7 | Детерминированная сериализация; формы ответов — единственный источник для деградационных объектов |
| DOM-8 | ACL в домене: фильтрация прав на уровне модели, не только транспорта [SEC-3] |
| DOM-9 | Явная обработка отсутствия/ошибок (Optional/Result), без «магических» дефолтов |

### C. Слой доступа к данным / хранилище — best practice: EXPLAIN-driven, prepared, least privilege

| # | Критерий |
|---|----------|
| STO-1 | Только параметризованные/prepared statements; пользовательский ввод не конкатенируется (anti-injection) [SEC-5] |
| STO-2 | Доступ только к своим объектам/схемам; чужие данные — через реплику/события (database-per-service) [ISO-1] |
| STO-3 | EXPLAIN-driven: горячие пути имеют индексы под план; нет seq scan по большим таблицам |
| STO-4 | Partition pruning работает: запросы к партиционированным таблицам несут ключ партиции |
| STO-5 | Нет N+1: списки/лента — один запрос или bulk; без построчной рекурсии в приложении |
| STO-6 | Самодостаточность функций хранилища: процедуры/функции в своей схеме, без зависимости от чужих |
| STO-7 | Управляемые производные структуры (materialized views/проекции): стратегия обновления, не блокируют чтения |
| STO-8 | Все выборки ограничены (`LIMIT`/keyset); нет unbounded `SELECT *` по большой таблице |
| STO-9 | Инкрементальное чтение по курсору: индекс по курсорному полю, keyset-пагинация |
| STO-10 | `statement_timeout` роли согласован с клиентским дедлайном — медленный запрос не висит дольше клиента |
| STO-11 | Транзакции коротки; уровень изоляции осознан; outbox пишется в одной транзакции с доменным изменением |
| STO-12 | Миграции версионируются, обратимы, zero-downtime (expand/contract + alias-views) [REL-5] |

### D. Исходный код и зависимости — best practice: явные зависимости, гигиена импортов

| # | Критерий |
|---|----------|
| SRC-1 | Зависимости объявлены на уровне модуля/верхушки файла, сгруппированы (stdlib/third-party/local); скрытый ленивый импорт в теле функции запрещён |
| SRC-2 | Исключение из SRC-1 — локальный импорт допустим только для (а) bootstrap/разрыва циклов на старте, (б) динамически/опционально подгружаемых модулей; каждый сопровождается комментарием-причиной |
| SRC-3 | Нет циклических зависимостей; цикл — сигнал неверной границы модуля, лечится переносом абстракции/портом (DIP) [PAT-9] |
| SRC-4 | Нет wildcard-импортов; явные имена; нет неиспользуемых импортов/символов (линтер) |
| SRC-5 | Тяжёлые/опциональные зависимости, если нужна ленивость, оформляются как провайдер в DI, а не как функция-локальный импорт в домене [PAT-1] |
| SRC-6 | Версии зависимостей закреплены (lock); обновления контролируемы, автоскан на CVE [SEC-11, REL-2] |

> Контроль SRC-1/SRC-2: линт-правило с allowlist для инициализаторов и явным обоснованием
> в разрешённых случаях. CI отклоняет функция-локальный импорт без обоснования.

### E. Кросс-сервисные данные (ECST) — Event-Carried State Transfer

> **Принцип**: вместо ETL/репликации на уровне БД данные из внешних сервисов/агрегатов
> захватываются **в момент записи бизнес-события** — из тела запроса или доменного события.
> Сервис хранит ровно столько, сколько нужно для своей логики, и не более.
>
> **Reference by Natural Key** — внешняя сущность не нужна локально; храним только натуральный
> ключ. Обращаться к источнику — только когда критично (см. FT-6).
>
> **Inline Snapshot** — нужны конкретные поля для отображения; захватываем их при создании и
> моделируем как Value Object (неизменяемый, без FK на внешнюю таблицу).

| # | Критерий | Порог / проверка |
|---|----------|-----------------|
| ECST-1 | Новые write-операции с зависимостью на внешний агрегат принимают нужные поля в теле запроса, а не вызывают смежный сервис синхронно для доогрузки | code review: нет sync-запросов к другим сервисам в write-path (кроме FT-6) |
| ECST-2 | Reference by Natural Key: если данные внешней сущности не нужны для домена — хранится только натуральный ключ; нет реплики «про запас» | DDL ревизия |
| ECST-3 | Inline Snapshot: встроенные поля — Value Object (`NOT NULL`, без `REFERENCES` на внешнюю таблицу), принадлежит агрегату-владельцу | DDL: нет FK на чужие схемы; тест формы |
| ECST-4 | Snapshot не устаревает критично: для отображения — допустимо; для финансового решения — sync к владельцу (FT-6) | политика свежести задокументирована |
| ECST-5 | Суррогатные ключи для внутренних связей; натуральные ключи — только точка интеграции, не FK внутри домена | DDL: FK по суррогатам; natural key — `UNIQUE INDEX` |

## Definition of Done (как применять шаблон)

- Критерий без зелёной проверки/порога считается невыполненным — «частично» не засчитывается.
- Для каждого критерия проект задаёт **конкретный порог** и **место проверки** (тест/линт/дашборд/ревизия) в оверлее.
- Релиз на 100% трафика не идёт без зелёных: **SLO (Часть 0), PROTO-1, FT-1..4, FT-8, OPS-1..3, DR-1..3, REL-4..5**.
- Очистка/удаление легаси не стартует, пока не выполнены **ISO-1..6** и не снят дубль-код домена.
- Применимость секций фиксируется явно: для каждой ★-секции и для A11Y отметьте «применимо/N/A + обоснование».
- Привязку к коду и этапам ведите в отдельном файле `quality_criteria.<service>.md` (оверлей), не загрязняя этот шаблон.
