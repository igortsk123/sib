# Agent Orchestration — параллелить только когда выгода >> риск конфликта

Суб-агенты дают параллелизм и offload контекста, но платят риском конфликтов в shared-файлах.
Использовать, только когда выгода существенно превышает риск.

## Можно запускать (в т.ч. параллельно)
1. Агент пишет ТОЛЬКО новые файлы — без пересечений.
2. Несколько агентов, каждый в своей изолированной области.
3. Один большой логически цельный воркстрим — делегировать целиком.
4. Open-ended research без правки файлов.

## НЕЛЬЗЯ параллельно (серийно или самому)
1. Два агента редактируют один shared-файл (registry/индекс/общий конфиг).
2. Агент B зависит от output агента A (data dependency).
3. Правка маленькая (<50 строк, 1–2 файла) и контекст уже есть — самому быстрее.

## Матрица файлов перед запуском нескольких агентов
```
Agent A → создаёт: [...]   редактирует: [X]
Agent B → создаёт: [...]   редактирует: [X]   ← КОНФЛИКТ
```
Пересечение в столбце «редактирует» → запускать **серийно**, не параллельно.

## Cost/benefit
| Размер | Подход |
|--------|--------|
| <50 строк, 1–2 файла | сам |
| Новый файл 50–300 строк | сам или агент |
| Новый воркстрим (200+ строк) | агент |
| Несколько независимых воркстримов, чистая матрица | агенты параллельно |
| Зависимая цепочка | серийно или одним агентом |

## Антипаттерн
Запустить N агентов «для скорости», когда все патчат один файл → race condition и потеря правок.
