# Quality Criteria — как применять эталоны качества

> Дополняет `_optional/quality-standards/*` (сами эталоны — цель+порог, MUST/SHOULD/PREFER).
> Здесь — **подход к внедрению**: как из большого свода критериев получить выполнимый план и
> поднимать зрелость, **не ломая живой продукт**. Дистиллировано из реального внедрения.

## Базовые принципы
- **Цель, а не статус.** Эталон описывает целевое состояние. Текущее — в overlay проекта
  (`quality-criteria.<project>.md`): зрелость 0–4 по областям + разрывы `current → target`.
- **Измеримость.** Критерий без зелёной проверки/порога не засчитывается. «Частично» = не сделано.
- **Сдвиг проверок влево.** Цель — нарушение ловит **машина** (typecheck/lint/тест/CI-gate/EXPLAIN/health),
  а не ревьюер. Зрелость 3 = enforcement.
- **Без over-engineering.** Эталоны сами требуют YAGNI/KISS. Берём то, что даёт ценность сейчас;
  остальное — N/A с обоснованием и **отложенным триггером** (см. ниже).

## Roadmap-подход (как из свода → выполнимый план)
1. **Прочитать релевантные эталоны**, отметить применимость (N/A фиксировать явно).
2. **Верхнеуровневый roadmap** = набор EPIC'ов под РЕАЛЬНОСТЬ проекта (монолит ≠ микросервисы;
   маленькие данные → OLAP/шардинг N/A). Каждый EPIC: цель · ID критериев · что входит · что N/A · откат.
3. **EPIC за EPIC: план → делаю → следующий.** Один EPIC = один план в `plans/`, со списком файлов,
   гейтами и откатом. Не браться за всё сразу.

### Типовая последовательность EPIC'ов (порядок важен)
0. **Сеть безопасности (ПЕРВЫМ).** Health-эндпоинт · деплой со **smoke-проверкой + авто-откатом** ·
   бэкап + **restore-drill** (бэкап ≠ восстановление!) · pre-deploy гейты (typecheck/test/build).
   Без этого остальное рискованно — она делает изменения обратимыми и проверяемыми.
1. **Корректность данных** (дёшево, высокоценно): CHECK-инварианты в БД, NULL-логика, тотальный
   ORDER BY с уникальным тай-брейком, statement/lock/idle-timeout на роль БД.
2. **Безопасность**: security-заголовки, SCA-гейт (CVE прод-дерева; CLI-тулзы → devDeps),
   least-privilege роль БД, ротация секретов.
3. **Устойчивость UX**: error-boundary вместо белого экрана, таймауты/состояния loading-error-empty,
   анти-гонки (AbortController/latest-wins), синк выхода между вкладками.
4. **Наблюдаемость**: структурные JSON-логи с маскированием PII, захват клиентских ошибок на сервер.
5. **DX/сопровождаемость**: lint-гейт, чистка мёртвого кода, ADR, README-онбординг.
6. **Тесты**: unit на чистое ядро (валидации/расчёты — макс. отдача), в гейт сборки.
7. **Lifecycle/PII**: retention транзиентных данных (истёкшие сессии/попытки), инвентарь PII, GDPR-удаление.

## «Не сломать продукт» — обязательные привычки
- **Гейт сборки**: deploy идёт только при зелёных typecheck + test + build. Падение гейта — старый контейнер не трогаем.
- **Миграции аддитивны** (expand-only): `ADD COLUMN`/`CREATE TABLE`/`ADD CONSTRAINT NOT VALID → VALIDATE`.
  Тогда откат образа безопасен (старое приложение работает с новой схемой). Breaking — в два шага (expand → деплой → contract).
- **Деплой со smoke + rollback**: после свапа — проверка health; не-200 → авто-откат на прошлый образ.
  ⚠️ Откат образа **не** спасает от смены env/конфига (напр. DATABASE_URL роли) — такие шаги выверять отдельно, с проверкой ДО переключения.
- **Перед опасной операцией над данными** — свежий бэкап + проверка на пустой результат (CHECK добавлять после `count(нарушений)=0`).

## Отложенные триггеры «если начнём X → применить Y»
Что сейчас N/A, но станет нужным при росте — записывать как условие, чтобы вспомнить:
- инсайты/аналитика → OLAP/Kimball (star schema, MV, аналитика не на живом OLTP);
- большие таблицы → партиционирование, keyset, `CREATE STATISTICS`;
- второй сервис → database-per-service, ECST, saga/outbox;
- 2-й язык → полноценный i18n (`Intl`/ICU, RTL-готовность); PWA → SW/offline; публичные страницы → SEO;
- мультиарендность → RLS по `tenant_id`; внешние потребители API → версионирование + contract-тесты.
Держать в overlay-файле проекта (напр. `quality-deferred-triggers.md`), проверять при соответствующих фичах.
